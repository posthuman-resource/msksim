## 1. Goal

Ship **F13 (parameter sweep)** from `docs/spec.md` §4.3: a UI and runner that lets a researcher pick one to three parameters from an `ExperimentConfig`, attach a grid of values to each parameter (numeric start/stop/step or an explicit list, or a checklist of categorical values), pick a replicate count per cell, see a live total-run count (with a warning when it exceeds 500), click **Start sweep**, watch every cell's replicates execute on top of the **step 27 batch queue** (so concurrency, partial failure isolation, cancellation, and per-replicate persistence are all inherited), accumulate per-cell aggregate statistics (mean / std dev / dominant classification) in memory from the persisted replicate outcomes, and render a **heatmap visualization** of the chosen outcome metric across the swept grid (2-parameter sweep → grid heatmap, 1-parameter sweep → bar chart, 3-parameter sweep → small-multiples of heatmaps keyed on the third axis), with a dropdown to re-color the heatmap by a different outcome metric without re-running the sweep. Step 28 is the primary mechanism for **RQ1** (identifying assimilation/segregation thresholds over population ratios and interaction probabilities) and the cleanest answer to **RQ2** (topology as a categorical sweep parameter with `lattice` / `well-mixed` / `network` producing visibly different heatmap bands). Concretely, this step delivers: (a) a **sweep configuration screen** reached from the experiments page — either a dedicated route at `app/(auth)/experiments/sweep/new/page.tsx` or a modal inside the existing experiments page; the route option is chosen and justified in section 7 because the sweep form is substantially larger than the batch modal from step 27 and deserves its own URL for bookmarking and for the MCP harness to `navigate_page` to directly; (b) a **sweepable-parameter catalog** at `lib/sim/sweep/parameters.ts` — a **hand-maintained** list of known sweepable field paths with their type metadata (numeric range or categorical enum), derived by inspection of the Zod schema from step 01 but not programmatically introspected; the hand-maintained choice is picked and justified in section 4's path-not-taken against a runtime Zod introspection utility; (c) a **sweep runner module** at `app/(auth)/experiments/sweep/sweep-runner.ts` that composes the **step 27 `createWorkerPool` factory** once per sweep (not once per cell — the pool handles all cells sequentially via a flat replicate list, because the pool already serializes work across concurrent slots and there is no benefit to spawning a pool per cell) and tracks the `(cellIndex, replicateIndex) → runId` mapping in a plain `Map` so that cell aggregation can later read the correct replicates out of the `runs` table via step 26's read helpers; (d) a **cartesian-product helper** at `lib/sim/sweep/cartesian-product.ts` plus its Vitest unit test at `lib/sim/sweep/cartesian-product.test.ts` — a pure `cartesianProduct<T>(axes: T[][]): T[][]` implementation using the `reduce`-over-`flatMap` pattern documented in research item 4 below, with tests for zero, one, two, three, and many axes; (e) a **sweep form** at `app/(auth)/experiments/sweep/sweep-form.tsx` — the Client Component that hosts (1) a base-config dropdown populated by `listConfigs()` from step 25's DAL helper, (2) a **parameter picker** that lets the user add 1-3 parameters from the sweepable-parameter catalog, (3) per-parameter grid-value editor (numeric inputs for start/stop/step or an explicit-values textarea; categorical checkbox list), (4) a replicates-per-cell number input (default 3, min 1, max 30), (5) a live "total runs = cells × replicates" display with a **red warning** if the total exceeds 500, (6) a **Start sweep** button that hands off to the sweep runner; (f) a **heatmap visualization component** at `app/(auth)/experiments/sweep/heatmap.tsx` that renders (1) a 2-parameter sweep as an SVG grid heatmap with x and y axes labeled by the swept parameter values and each cell filled with a color interpolated from a sequential viridis-like scale according to the chosen outcome metric's mean, (2) a 1-parameter sweep as a Recharts `BarChart` with one bar per value and the bar height equal to the metric mean, (3) a 3-parameter sweep as a row of small-multiples heatmaps (one per value of the third axis) sharing a color scale; the choice between Recharts and plain SVG is resolved in section 4 and section 7 — the heatmap itself is **plain SVG** (because Recharts has no first-class heatmap component and its Scatter+Cell workaround is heavier than a 15-line SVG grid), while the 1-parameter bar chart reuses Recharts from step 22 to keep the visual vocabulary consistent with the metrics dashboard; (g) a **metric selector dropdown** above the heatmap with options for `final assimilationIndex`, `final segregationIndex`, `final communicationSuccessRate`, `meanTokenWeightFinal`, `numDistinctTokensFinal`, and `timeToConsensus` (exact option list depends on which summary metric fields step 17 ships — the implementing claude greps step 17's plan file and picks whichever summary fields are actually populated on every run); changing the dropdown re-runs the per-cell aggregation in memory and re-renders the heatmap without re-running any simulations; (h) **per-cell aggregation helpers** at `lib/sim/sweep/aggregate.ts` with known-answer unit tests at `lib/sim/sweep/aggregate.test.ts` — three pure functions `mean(values: number[]): number`, `stdDev(values: number[]): number`, and `dominantLabel(values: string[]): string | null`, plus a composite `aggregateCell(runs: RunSummary[], metric: string): { mean: number; stdDev: number; classification: string | null; n: number }`; (i) an **MCP verification script** in section 10 that exercises the F13 example sweep from the step-specific context: log in, create a 50-tick base config, navigate to the sweep form, select the base config, pick `world1.monoBiRatio ∈ [0.3, 0.6]` and `experiment.preferentialAttachment.enabled ∈ [true, false]` as the two sweep parameters, set replicates per cell = 3, verify the total-run count displays `12`, click Start sweep, wait for all twelve runs to complete (~60 seconds at 2-way concurrency on the seed user's machine), assert the 2×2 heatmap renders with four distinct cell colors, change the outcome-metric dropdown to a different metric, assert the heatmap re-colors without network activity, take `docs/screenshots/step-28-sweep.png`, and assert console + network are clean. Scope boundary is strict: step 28 does **not** persist sweeps to a new `sweeps` database table (per-cell aggregates live in memory for v1 only — this is explicitly documented in section 4's path-not-taken and in section 7's implementation approach), does **not** ship CSV/JSON export of sweep results (step 30), does **not** ship run comparison (step 29), does **not** modify the step-27 worker pool contract (it composes the existing `createWorkerPool` factory without patching it), does **not** modify `workers/simulation.worker.ts` or the step-20 `SimulationWorkerApi`, does **not** add any dependencies beyond what steps 01-27 already installed, does **not** ship the topology-as-sweep-parameter variant beyond making it possible via the categorical path (the researcher can manually select `world1.topology.type` and check `lattice` / `well-mixed` — the catalog lists it, but the v1 MCP script exercises the numeric+boolean example above and leaves the categorical topology sweep as a researcher action not a gated acceptance criterion).

## 3. Spec references

- `docs/spec.md` **§4.3 Batch runner mode → F13. Parameter sweep.** The literal feature step 28 implements. Spec text: _"Select one or more parameters (e.g. mono:bi ratio, interaction probability, Δ⁺) and a grid of values; the system generates the cartesian product of configs and runs them through the batch queue. Results are aggregated into per-cell summaries with means and confidence intervals."_ Acceptance clause: _"A 5×5 sweep with 10 replicates per cell (250 runs) can be defined in the UI in under a minute; the aggregated view shows a heatmap of the chosen outcome metric across the grid; raw per-run metrics remain accessible."_ Supports clause: _"RQ1 — this is the primary mechanism for identifying critical thresholds. RQ2 — a sweep over 'topology' as a categorical parameter cleanly answers the lattice question."_ Every sentence maps to a concrete deliverable in step 28: (1) "select one or more parameters" → the parameter picker in `sweep-form.tsx` that lets the user add up to 3 parameters from the sweepable-parameter catalog (the catalog includes `world1.monolingualBilingualRatio`, `world1.topology.type`, `experiment.deltaPositive`, `experiment.preferentialAttachment.temperature`, `experiment.preferentialAttachment.enabled`, and others); (2) "grid of values" → the per-parameter grid editor that accepts numeric start/stop/step, an explicit list, or a categorical checkbox list; (3) "cartesian product of configs" → the `cartesianProduct<T>` helper in `lib/sim/sweep/cartesian-product.ts` which is the pure kernel of this feature; (4) "runs them through the batch queue" → the sweep runner composes step 27's `createWorkerPool` and submits cells as sequential batches; (5) "per-cell summaries with means and confidence intervals" → the aggregation helpers in `lib/sim/sweep/aggregate.ts` compute mean and std dev per cell per metric (confidence intervals are a linear function of std dev, so v1 exposes std dev directly in the cell tooltip; CI computation can be layered on in v2 without any schema change); (6) "heatmap of the chosen outcome metric" → the SVG heatmap component with the outcome-metric dropdown; (7) "under a minute to define a 5×5 sweep with 10 replicates per cell" → the form's UX is optimized for this: parameter picker is a searchable dropdown, grid editor accepts either range-syntax (`0.1-0.5 step 0.1`) or an explicit list, and the replicate count is a single numeric input. - `docs/spec.md` **§1.2 RQ1 — Assimilation vs. segregation thresholds.** The primary research question step 28's sweep mechanism enables answering: _"Under what population ratios (monolingual:bilingual) and interaction probabilities do bilingual immigrants in World 2 assimilate into the host linguistic community versus segregate into an insular one?"_ The MCP example sweep in section 10 is a minimal 2×2 instance of exactly this research question: `world1.monolingualBilingualRatio` at two values crossed with `experiment.preferentialAttachment.enabled` at two values. Step 28's UI is the tool that makes that workflow one form fill and one Start click.

- `docs/spec.md` **§1.2 RQ2 — Role of spatial topology.** _"Does a 2D lattice constraint produce qualitatively different macroscopic outcomes than well-mixed (random-interaction) populations with identical composition?"_ The spec's F13 supports clause calls out the categorical sweep over topology as the _"cleanest answer"_ to RQ2. If step 27's batch queue has a bug, every sweep inherits it; conversely, every sweep benefits from step 27's partial-failure isolation and cancellation, so a single worker throwing in cell (2, 3) does not poison the rest of the sweep.

### External references (WebFetched during research)

5. **MDN `Array.prototype.flatMap` — cartesian product pattern.** `https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Array/flatMap` (WebFetched during research). Load-bearing facts:
   - The canonical JavaScript cartesian product of two arrays is `arr1.flatMap(x => arr2.map(y => [x, y]))`. Generalizing to N arrays, the idiomatic pattern is `arrays.reduce((acc, curr) => acc.flatMap(a => curr.map(c => [...a, c])), [[]] as T[][])`, which is what step 28's `cartesianProduct<T>(axes: T[][]): T[][]` implementation uses.
   - This pattern is `O(∏|axes|)` in both time and space, which is exactly what the cartesian product requires — there is no asymptotic optimization over the naive approach. For N ≤ 500 cells (step 28's warning threshold), the entire product computation is well under 1ms on any modern JavaScript engine.
   - The empty-axes case (`cartesianProduct([])`) returns `[[]]` — a single empty tuple — matching the mathematical convention that the product of zero sets is the one-element set containing the empty tuple. This is tested explicitly in the unit test in section 9.
   - The single-axis case (`cartesianProduct([[1, 2, 3]])`) returns `[[1], [2], [3]]`, not `[1, 2, 3]` — every output element is a tuple regardless of input arity. This is also tested.

6. **Recharts library overview — heatmap pattern.** `https://github.com/recharts/recharts` (WebFetched during research). Load-bearing facts:
   - Recharts has a `Cell` component that accepts a `fill` prop per data point, used inside `Pie`, `Scatter`, `Bar`, and `Treemap` charts. A grid heatmap can be assembled with `ScatterChart` + `Scatter shape="square"` + one `<Cell fill={colorForValue(datum.value)} />` per datum, positioned on a numerical x/y grid. This works but produces ~7 Recharts nesting levels plus a custom `Shape` render function for square cells.
   - Recharts has **no first-class heatmap component**. Every public example that looks like a heatmap is either (a) a Treemap (hierarchical rectangles, wrong for a regular grid), (b) a Calendar heatmap built on top of react-calendar-heatmap (a separate library, not Recharts), or (c) a ScatterChart workaround as described above.
   - For a 2×2 or 3×3 heatmap, plain inline SVG is simpler: 15-20 lines of `<svg>` with `<rect>` per cell and `<text>` for labels. The color scale is a one-line interpolation: `d3-interpolate`-style `t => \`hsl(${240 - t \* 240}, 80%, 50%)\``where`t ∈ [0, 1]` is the normalized metric value. Step 28 picks **plain SVG for the heatmap** (section 7 justifies this in detail) and **Recharts for the 1-parameter bar chart** (to match step 22's existing visual vocabulary). The 3-parameter small-multiples case is plain SVG repeated N times.
   - The research confirms Recharts **is already a dependency** via step 22; importing Recharts in the 1-parameter case adds zero bundle cost beyond what the metrics dashboard already pays.

7. **Programmatic Zod schema introspection to generate the sweepable-parameter catalog.** Considered and rejected in favor of a hand-maintained list. Zod 4's `z.ZodObject.shape` + recursive traversal can produce a list of dot-paths and their types (numeric ranges, enum values, booleans) automatically, and a utility like `zod-to-json-schema` can flatten a schema into a descriptor suitable for form generation. The rejection is on three grounds: (a) **not every field in `ExperimentConfig` is a sensible sweep axis** — `vocabularySeed` is a deeply nested record, `seed` is conceptually a replicate axis not a sweep axis, `schedulerMode` is a rarely-tuned enum, and blindly introspecting the schema produces a 50+ field list most of which would be noise in the parameter picker; (b) **the hand-maintained list is small** — the step-specific context names 5 fields (`world1.monoBiRatio`, `world1.topology.type`, `experiment.deltaPositive`, `experiment.preferentialAttachment.temperature`, `experiment.preferentialAttachment.enabled`) and the realistic v1 catalog is maybe 10-15 entries; maintaining them by hand is ~30 lines of data declaration and zero runtime parsing overhead; (c) **the runtime assertion walk** described in section 2 catches schema drift without needing the full introspection machinery — it is strictly less powerful but also strictly simpler, and for a project with two researchers the simpler path is correct. If v2 needs to dynamically add sweep axes (e.g., a user-facing "add custom parameter" flow), Zod introspection can be revisited. This is the required path-not-taken per the step-specific context (_"extracted from the Zod schema via introspection — or a hand-maintained list; document your choice"_).

8. **Route vs modal for the sweep form.** Considered and resolved in favor of a **dedicated route** at `app/(auth)/experiments/sweep/new/page.tsx`. The step-27 batch queue chose a modal inside the experiments page because the batch form has 5 fields and fits comfortably in a dialog. Step 28's sweep form has: base config dropdown, 1-3 parameter pickers each with a grid editor, replicate count, metric dropdown (post-completion), plus the heatmap visualization that needs real estate to render 2×2 or larger grids with axis labels. A modal would be cramped for all of this, and the form's bookmarkability is a meaningful UX win for researchers who want to tweak a sweep they defined yesterday. The dedicated route also cleanly separates step 28's client bundle from the experiments page's bundle, which matters because the heatmap + metric re-aggregation logic is ~400 lines of client code. The experiments page gains a **link** to the sweep route, not a modal trigger, next to the step-27 batch button. The step-specific context says _"A 'New sweep' screen (can be under `/experiments/sweep/new` or inside the experiments page). Pick one and document."_ — this plan picks the route explicitly.

9. **A `sweeps` database table tracking the sweep as a first-class persisted entity.** Rejected for v1 and documented explicitly here and in section 7. The step-specific context says _"Store in-memory for now; persistence to a `sweeps` table is a v2 concern — document this explicitly."_ Rationale: (a) each replicate is already a self-contained `runs` row with its config, seed, and summary — the researcher can always re-fetch them after a page reload by querying the runs browser for rows with matching `configId` and a seed range; (b) the sweep itself is a **client-side orchestration concept**, exactly as step 27's batch was — it exists in the researcher's head and in the current tab's memory, and losing it on page reload is acceptable because the underlying runs are persisted; (c) a `sweeps` table would require a schema migration, a new DAL module, a new Server Action path, and a new runs-browser join, which is substantial complexity for zero v1 user value. v2 can add a `sweeps` table if the researchers find themselves wanting to name and bookmark sweeps — the decision can be deferred without any v1 schema debt because step 28 does not write any new columns to `runs` either. If the researcher navigates away from `/experiments/sweep/new` mid-sweep, the in-flight batches continue running on the worker pool (because step 27's pool lives in the modal component and will be cancelled on unmount per step 27 §5's cleanup rule) — **correction**: the pool in step 28 lives in the sweep form component, which means navigating away _does_ cancel it. This is the correct v1 behavior and matches step 27. A researcher who wants to run a large sweep and walk away should stay on the page.

10. **Confidence intervals computed via bootstrap resampling.** Considered and rejected for v1. The spec's F13 text says "per-cell summaries with means and confidence intervals." For a normally-distributed outcome metric, a 95% CI is `mean ± 1.96 * (stdDev / sqrt(n))`, which is a closed-form computation from the values step 28 already computes. For a non-normal metric (bimodal assimilation/segregation outcomes at a threshold), a bootstrap resampling would give a more honest CI but requires the per-replicate raw values to be retained and resampled ~1000 times per cell. v1 computes mean and stdDev only, and the heatmap cell tooltip shows both; the researcher can infer a rough CI from stdDev mentally. v2 can add a bootstrap helper if the published figures demand it. This is a conscious scope reduction relative to the spec text — documented here so a future reviewer can see why the simpler path was chosen. (The spec's acceptance clause does not literally mention CIs, only "means and confidence intervals" in the description; no MCP assertion depends on CIs being present.)

11. **Dropping metric aggregation in-memory and instead computing it on the fly inside the heatmap render.** Considered and rejected. A naive implementation would have the heatmap component iterate `cellReplicates: Map<cellKey, RunSummary[]>` on every render and compute the mean/stdDev for each cell inline; changing the metric dropdown triggers a render, which triggers re-computation, which is O(cells × replicates × metrics) per render. For a 5×5 sweep with 10 replicates and one metric at a time, that is 250 number additions per render, which is trivial. But the cleaner architecture is to precompute the aggregate for each (metric, cell) pair on completion of the full sweep, cache the results in a `Map<metricKey, Map<cellKey, CellAggregate>>`, and read cached values on metric dropdown change. The precomputed path makes the metric change instant (no computation, just a lookup) and makes the code more testable (the aggregation function is a pure one-shot pass, not tangled with render logic). Section 7 uses the precomputed path.

- `lib/sim/sweep/cartesian-product.ts` — **pure cartesian product helper**, client-safe, no imports from React or any server-only module. Exports a single function:

  ```ts
  export function cartesianProduct<T>(axes: readonly (readonly T[])[]): T[][] {
    if (axes.length === 0) return [[]];
    return axes.reduce<T[][]>(
      (acc, currentAxis) => acc.flatMap((prefix) => currentAxis.map((value) => [...prefix, value])),
      [[]],
    );
  }
  ```

  That is the entire implementation — ~10 lines plus a JSDoc comment explaining the empty-axes convention (`[[]]` not `[]`). The function is generic so callers can pass `cartesianProduct<number>([ ... ])` for numeric grids, `cartesianProduct<string>(...)` for categorical grids, or `cartesianProduct<number | string | boolean>(...)` for mixed grids — which is the realistic step 28 use case since the three parameters can be mixed types (e.g., a numeric ratio crossed with a boolean enabled flag crossed with a string topology type).

    1. Expand the cartesian product: `const grid = cartesianProduct(spec.parameterPicks.map(p => p.values))`. Each entry is a tuple of parameter values in the order the picks were declared.
    2.     3. Construct the worker pool **once** via `this.pool = createWorkerPool({ persist: options.persist, createWorker: options.createWorker })`. The pool is a field on the runner instance (via closure); it is reused across all cells.
    4. For each cell in FIFO order:
       a. Set the cell's status to `'running'` in the sweep state, emit `onUpdate`.
       b. Compute the base seed for this cell: `const cellBaseSeed = (spec.baseSeed ?? 1) + cellIndex * spec.replicatesPerCell`. This is the "unique seed per run" formula from the step-specific context: `baseSeed + cellIndex * replicateCount + replicateIndex`, implemented by the pool's `baseSeed + id` default where `id ∈ [0, replicateCount)`.
       c. Subscribe to the pool's `onUpdate` events for this cell's batch. The subscription closure captures the `cellKey` and writes the replicate states into the sweep state's per-cell tracking structure.
       d. Call `await this.pool.startBatch({ config: cell.config, configId: spec.configId, replicateCount: spec.replicatesPerCell, baseSeed: cellBaseSeed, totalTicks: spec.totalTicks, concurrency: spec.concurrency })`. The `startBatch` method in step 27 is synchronous (returns `void` immediately), so the runner awaits completion by polling the pool state via its `onUpdate` subscription — waiting until all `replicateCount` replicates are terminal (`completed` / `failed` / `cancelled`) before advancing to the next cell. Implementation: `await new Promise<void>((resolve) => { const unsub = pool.onUpdate((state) => { if (state.replicates.every(r => isTerminal(r.status))) { unsub(); resolve(); } }); });`.
       e. After the pool drains for this cell, read every replicate's `runId` from `pool.getState().replicates` (skipping any that are `null` — those are the failed or cancelled replicates where persistence might not have returned a runId). For each non-null runId, call `await options.loadRunSummary(runId)` and collect the summaries into `cellReplicates.set(cell.cellKey, summaries)`.
       f. Emit `onUpdate` for the cell's completion.
    5. Once all cells have drained, set `state.phase = 'completed'`, set `finishedAt`, emit a final `onUpdate`. The entire `startSweep` method then resolves.
  - `SweepRunner.cancel(): Promise<void>` — (1) set an internal cancellation flag; (2) if a pool batch is in flight, call `this.pool.cancelBatch()` which terminates the in-flight workers and marks their replicates `cancelled`; (3) transition all remaining cells to `'cancelled'` without running them; (4) set `state.phase = 'cancelled'`, emit `onUpdate`.   The **`TwoDimensionalGrid` subcomponent** is plain SVG:
  - Compute the x-axis values and y-axis values from `picks[0].values` and `picks[1].values`.
  - Compute the min and max of every cell's `aggregate.mean` for the current metric to build the color scale.
  - Render an `<svg>` with a `width` and `height` chosen such that each cell is at least 60×60 px (legible for a 2×2 or 5×5 grid).
  - For each cell, render a `<rect x={xOffset + xIndex * cellWidth} y={yOffset + yIndex * cellHeight} width={cellWidth} height={cellHeight} fill={colorScale(cell.mean)} />`.
  - For each cell, render a `<text>` centered in the cell showing the cell mean formatted to 3 significant figures.
  - Render axis labels (`<text>` elements) on the left and bottom edges.
  - Render a legend/color bar on the right side: a vertical gradient with tick labels showing the min/mid/max values.
  - Wrap each cell in a `<title>{cell tooltip text}</title>` so hovering shows the full `{mean, stdDev, n, classification}` record.
  - Add `data-testid={`heatmap-cell-${xIndex}-${yIndex}`}` for MCP verification.
    The color scale is a simple viridis-like interpolation: `const t = (value - min) / (max - min); const hue = 240 - t * 240; return \`hsl(${hue}, 75%, 50%)\`;` — 240° (blue) at the min, 0° (red) at the max. This is not the true viridis colormap (which is perceptually uniform and requires d3-scale-chromatic to get right) but it is visually acceptable for a v1 research tool and adds zero dependencies. If the researchers push back on the color scheme, d3-scale-chromatic can be added in v2 as a targeted improvement.
The **`OneDimensionalBarChart` subcomponent\*\* uses Recharts (already a dependency from step 22):

  ```tsx
  <ResponsiveContainer width="100%" height={300}>
    <BarChart data={dataArray}>
      <XAxis dataKey="parameterValue" label={picks[0].label} />
      <YAxis label={metricLabel} />
      <Tooltip
        formatter={(value, name, { payload }) => [
          `${value.toFixed(4)} ± ${payload.stdDev.toFixed(4)}`,
          metricLabel,
        ]}
      />
      <Bar dataKey="mean" fill="#4a90e2" />
    </BarChart>
  </ResponsiveContainer>
  ```

  `dataArray` is built from the aggregates map: one entry per parameter value with `{ parameterValue, mean, stdDev, n, classification }`.
  The **`ThreeDimensionalSmallMultiples` subcomponent** iterates the third parameter's values and renders one `<TwoDimensionalGrid>` per value, all sharing the same color scale (computed from the full 3D aggregate range so cells across panels are comparable). The panels are laid out in a horizontal flex row with a shared x/y axis label block above them.
  The **`EmptyState` subcomponent** displays "Configure at least one parameter and run the sweep to see results."
  File size: ~450 lines including all four subcomponents, the color scale utility, and the SVG rendering code.

## 7. Implementation approach

The work is ordered so that the pure helpers land first (cartesian product, aggregation), then the parameter catalog (which depends only on Zod from step 01), then the sweep runner (which depends on the helpers and on step 27's worker pool), then the heatmap component (which depends only on the aggregate types), then the sweep form (which composes everything), then the Server Component page wrapper, then the experiments page link, then the CLAUDE.md appends, then the MCP verification script. This order minimizes the window in which the repository is in an inconsistent state and puts the most-tested code (the helpers and the cartesian product kernel) at the bottom of the dependency stack where it can be sanity-checked with `npm test` before any UI surface exists.

**Slice one — cartesian product helper and its tests.** Create `lib/sim/sweep/cartesian-product.ts` with the ten-line reducer implementation and its JSDoc. Immediately create `lib/sim/sweep/cartesian-product.test.ts` with the seven test cases from section 5. Run `npm test -- cartesian-product.test.ts` locally. This helper is the mathematical kernel of parameter sweeps; if it has a bug, every sweep is wrong. Testing it in isolation is the cheapest way to catch the bug.

**Slice two — aggregation helpers and their tests.** Create `lib/sim/sweep/aggregate.ts` with `mean`, `stdDev`, `dominantLabel`, `aggregateCell`, `aggregateSweep`, `getByPath`, and `setByPath`. Create `lib/sim/sweep/aggregate.test.ts` with the known-answer tests from section 5. Run `npm test -- aggregate.test.ts` locally. The stdDev sample-vs-population distinction (divisor `n - 1` not `n`) is easy to get wrong; the known-answer test against `stdDev([1, 2, 3, 4, 5]) === Math.sqrt(2.5)` catches a population-stdDev mistake immediately.

**Slice three — parameter catalog and its runtime assertion.** Create `lib/sim/sweep/parameters.ts` with the ~13 entries from section 5. At the bottom of the file, after the `sweepParameters` constant, add the runtime assertion loop that walks `ExperimentConfig.parse({})` for every entry's path. The assertion runs on first import. Write no test for this file — the assertion itself is the test, and if it throws, the entire module fails to load, which is the correct failure mode.

**Slice four — sweep runner module.** Create `app/(auth)/experiments/sweep/sweep-runner.ts`. This is the composition layer on top of step 27's pool. Declare the types (`SweepSpec`, `SweepState`, `CellState`, `SweepRunner`) first, then the `createSweepRunner` factory function body, then `applyPickOverrides` and `setByPath` helpers (if not already exported from aggregate.ts). The key design choice is **per-cell sequential batches** rather than a single flattened batch: for each cell, call `pool.startBatch(cellSpec)` and await its drain via an `onUpdate`-based polling promise (documented in section 5's file list), then read the per-replicate summaries via `options.loadRunSummary(runId)`, then advance to the next cell. This keeps step 27's pool contract unchanged (the pool still only knows how to run N replicates of one config) and lets step 28's runner add the "iterate over cells" layer above it without touching step 27's code. The pool is constructed **once per sweep** and reused across all cells — do **not** construct a new pool per cell, because that would spawn and tear down workers repeatedly and waste the 1-2 seconds of worker startup time per cell. The pool is torn down on sweep completion (the per-batch drain check inside step 27's pool terminates workers when no more replicates are pending; the runner triggers this by letting the final cell's batch drain fully, at which point the pool's workers are all released).

**Slice five — heatmap component.** Create `app/(auth)/experiments/sweep/heatmap.tsx`. Start with the `EmptyState`, `OneDimensionalBarChart`, `TwoDimensionalGrid`, and `ThreeDimensionalSmallMultiples` subcomponents in that order. The `TwoDimensionalGrid` is the largest subcomponent and deserves careful attention: the SVG layout math (cell width, cell height, axis label positioning, color bar rendering) is straightforward but easy to get wrong; sketching the expected output for a 2×2 grid on paper before coding is a worthwhile five-minute investment. The color scale is the `hsl(240 - t*240, 75%, 50%)` interpolation described in section 5. The `<title>` child of each `<rect>` provides native SVG tooltips that render as browser-provided hover text; no JavaScript tooltip library is needed. The `data-testid` attributes on every cell are the MCP verification hook. Render `metric` and `metricLabel` at the top of the component so the screenshot captures the active metric name.

**Slice six — sweep form Client Component.** Create `app/(auth)/experiments/sweep/sweep-form.tsx`. `'use client'` line 1. This is the largest file in the step. Build it in order: imports → state hooks → derived values (`totalRuns` via `useMemo`) → the runner ref and its `useEffect` cleanup → the callbacks (`handleStart`, `handleCancel`, `handleMetricChange`) → the JSX. The JSX has three phases driven by `phase` state: `'idle'` (the form), `'running'` (the progress grid), `'completed'` / `'cancelled'` (the heatmap + metric dropdown). Each phase is a top-level conditional inside the root element. The parameter picker is the most complex part of the form: it renders a list of zero to three parameter rows, each with a dropdown for the parameter path (filtering out already-picked paths so the user cannot pick the same parameter twice), a kind-specific value editor (numeric range / explicit list / categorical checklist), and a remove button. An "Add parameter" button appends a new row if fewer than 3 are picked. The grid editor for numeric parameters accepts either a range syntax or an explicit list — parse the textarea on blur: if it matches `/^-?\d+(\.\d+)?\s*,\s*-?\d+(\.\d+)?\s*,\s*-?\d+(\.\d+)?$/` it is `start, stop, step` and expand to `[start, start+step, ..., ≤stop]`; otherwise split on comma and parse each as a number. For booleans, the editor is a pair of checkboxes for `true` and `false`. For enums, the editor is a checklist with one box per enum value.

**Slice seven — the Server Component page wrapper.** Create `app/(auth)/experiments/sweep/new/page.tsx`. 30-line Server Component that calls `verifySession()`, `listConfigs()`, `await searchParams`, and renders `<SweepForm ... />`. **Slice eight — experiments page link.** Modify `app/(auth)/experiments/page.tsx` to add a `<Link href="/experiments/sweep/new">New sweep</Link>` adjacent to the step-27 `BatchRunButton`. ~5 lines changed. Add `data-testid="sweep-new-link"` so the MCP script can click it by selector.

**Slice nine — the CLAUDE.md appends per section 11.**

## 9. Unit tests

**Tests shipped in step 28:** two new test files, both running in Vitest's default `node` environment (no `happy-dom` needed — all tests are for pure functions).

### `lib/sim/sweep/cartesian-product.test.ts`

Seven `it(...)` test cases inside a single `describe('cartesianProduct', ...)` block:

1. **Empty axes**. `expect(cartesianProduct([])).toEqual([[]])`. The empty-product convention.
2. **Single axis with three values**. `expect(cartesianProduct([[1, 2, 3]])).toEqual([[1], [2], [3]])`.
3. **Two axes, 2×2**. `expect(cartesianProduct([[0.3, 0.6], [true, false]])).toEqual([[0.3, true], [0.3, false], [0.6, true], [0.6, false]])`. This is literally the input the MCP script uses; correctness here is directly load-bearing for the acceptance test.
4. **Two axes, 2×3**. `expect(cartesianProduct([[1, 2], ['a', 'b', 'c']])).toEqual([[1, 'a'], [1, 'b'], [1, 'c'], [2, 'a'], [2, 'b'], [2, 'c']])`.
5. **Three axes, 2×2×2**. `expect(cartesianProduct([[1, 2], [3, 4], [5, 6]])).toHaveLength(8)` plus explicit ordering assertion.
6. **Empty axis collapses the whole product**. `expect(cartesianProduct([[1, 2], []])).toEqual([])`. The form validation prevents this case at the UI level but the helper must tolerate it.
7. **Large axes performance**. `expect(cartesianProduct([range10, range10])).toHaveLength(100)`. Asserts 100 elements; optional `performance.now()` bound for paranoia.
8. **Mixed types**. `const result = cartesianProduct<number | boolean | string>([[1, 2], [true], ['a']]); expect(result).toEqual([[1, true, 'a'], [2, true, 'a']])`. Confirms the generic signature compiles and runs.

### `lib/sim/sweep/aggregate.test.ts`

Four `describe` blocks — one per helper — each containing 3-5 `it(...)` cases.

- **`describe('mean', ...)`**: empty array returns NaN; single value returns the value; `[1,2,3,4,5]` returns 3; float values return accurate float (`[0.1, 0.2, 0.3]` toBeCloseTo `0.2`).
- **`describe('stdDev', ...)`**: empty returns NaN; single value returns NaN (n < 2); constant values return 0; `[1,2,3,4,5]` returns `Math.sqrt(2.5)` via `toBeCloseTo`.
- **`describe('dominantLabel', ...)`**: empty returns null; single value returns value; strict majority returns the winner; tie broken by first occurrence; all-same returns the single label.
- **`describe('getByPath', ...)`**: top-level key; nested key two levels deep; nested key three levels deep; missing key returns undefined; empty path returns the object itself.
- **`describe('aggregateCell', ...)`**: a hand-built 3-replicate mock `RunSummary[]` whose assimilation index values are `[0.2, 0.4, 0.6]` and classifications are `['segregation', 'mixed', 'segregation']`. Assert `{ mean: 0.4, stdDev: 0.2, n: 3, classification: 'segregation' }`.
- **`describe('aggregateSweep', ...)`**: a `Map` with two cells whose replicates produce different aggregates. Assert the returned map has two entries with the correct values.

Total: ~15 test cases across the two files. Every test is deterministic, has no I/O, no `Date.now()`, and uses only the known-answer pattern.

**Why no test for the sweep runner, the heatmap, the sweep form, or the parameter catalog?**

- **Sweep runner**: testing it would require mocking the step-27 worker pool's `createWorkerPool` factory and the step-26 `persistCompletedRun` / `loadRunSummary` helpers. The `createWorker` injection path from step 27 could be reused to stub the workers, and the `persist` + `loadRunSummary` options on the runner factory already accept injectable closures, so a unit test is **technically feasible** at ~200 lines. It is skipped for step 28 because (a) the runner's logic is straightforward "for each cell, await pool.startBatch, read summaries, advance" — the tricky part is the pool drain polling, which is already tested by step 27's own tests; (b) the MCP verification in section 10 exercises the runner end-to-end against real workers and a real SQLite database; (c) adding a runner unit test would duplicate ~60% of step 27's pool tests. If a future bug report localizes to the runner, a targeted unit test can be added at that time.
- **Heatmap**: the heatmap is mostly SVG markup driven by props. A snapshot test would exercise the JSX output, and a DOM test under happy-dom would verify the rect counts and color values. Both are deferred to v2 in favor of MCP end-to-end verification (which asserts the heatmap's `data-testid="heatmap-cell-*"` elements are present after the sweep completes). The color-scale computation is a pure function (`(value, min, max) => string`) and could be unit-tested in isolation; this is a reasonable v2 addition but not a v1 gate.
- **Sweep form**: the form is mostly orchestration glue. A happy-dom test would exercise `useState` transitions and `fireEvent` simulations, which is high-value in principle but expensive to write for a form with 3 phases and nested parameter pickers. MCP verification covers the end-to-end flow.
- **Parameter catalog**: the catalog's correctness is enforced by the module-load-time runtime assertion (section 5). A test that imports the module from a Vitest test will trigger the assertion; if the catalog has a broken path, `npm test` fails at import time with the clear error message the assertion throws. That effectively **is** a test — no separate test file needed, because the assertion is the gate.

**Summary: two new test files, ~240 lines total, ~15 test cases. Enough to gate the load-bearing pure-function kernels (cartesian product, aggregation) without duplicating coverage already provided by step 27's pool tests or by the MCP verification in section 10.**

## 13. Rollback notes

If step 28 lands in a broken state (e.g., the MCP script passes but a follow-up human review finds a regression in the sweep runner's cell iteration, or step 29's run comparison fails because step 28 introduced a contract drift in the persistence helper calls):

1. `git log --oneline` to find the commit SHA immediately prior to `step 28: parameter sweep`. It will have the subject `step 27: batch queue` (or a normalized variant).
2. `git reset --hard <step-27-sha>` to drop the working tree back to the pre-step-28 state. **This reverts every step-28 file**: `lib/sim/sweep/cartesian-product.ts`, `lib/sim/sweep/cartesian-product.test.ts`, `lib/sim/sweep/aggregate.ts`, `lib/sim/sweep/aggregate.test.ts`, `lib/sim/sweep/parameters.ts`, `app/(auth)/experiments/sweep/sweep-runner.ts`, `app/(auth)/experiments/sweep/heatmap.tsx`, `app/(auth)/experiments/sweep/sweep-form.tsx`, `app/(auth)/experiments/sweep/new/page.tsx`, the one-line edit to `app/(auth)/experiments/page.tsx`, the `CLAUDE.md` append, and `docs/screenshots/step-28-sweep.png`. The step-27 state is the last known-good checkpoint before step 28 began, and all of those files were absent or unchanged at that checkpoint.
3. No `package.json` changes to undo — step 28 adds no dependencies.
4. No database migrations to undo — step 28 touches nothing under `db/` or `db/schema/`. The runs that step 28's MCP verification persisted during its verification run **remain** in the local SQLite database (as regular `runs` rows, indistinguishable from any other completed run); if the rollback is on a researcher's machine and those rows are unwanted, the operator can delete them manually via `sqlite3 data/msksim.db "DELETE FROM runs WHERE config_id = '<sweep-test-config-id>'"` — but this is optional, and the DAL and runs browser tolerate stale rows gracefully.
5. `rm -rf .next` to flush Turbopack's dependency cache — a partial build of the sweep route can leave stale module-graph entries that cause the next `next build` to emit phantom errors about the deleted sweep files.
6. Re-run `npx tsx scripts/run-plan.ts --only 28` to redo the step from the clean step-27 base. Because `scripts/run-plan.ts` checks the commit log for the step marker and the marker is now gone, the orchestrator picks step 28 up as pending.

**Preferred approach: `git reset --hard` to the step-27 SHA.** `git revert` is also available but leaves an asymmetric history (a step 28 commit followed by a revert commit) that confuses the orchestrator's progress detection since the `step 28:` marker is still present in `git log` even though the changes are undone. Hard reset is the clean option when the step is unambiguously broken.

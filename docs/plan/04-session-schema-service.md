## 1. Goal

Add the `sessions` table to the drizzle schema and implement the server-side session service (`lib/auth/sessions.ts`) that backs every authenticated request in the app. The service owns the full lifecycle of a database-backed, opaque-token session: creation (random token + row insert + cookie set), validation (row lookup + expiration check), destruction (row delete + cookie clear), and the thin Next.js 16 async `cookies()` helpers used by steps 06 and 07. The step ends with `sessions.ts` covered by Vitest unit tests and a new checked-in migration under `db/migrations/`. It does **not** touch `proxy.ts`, the Data Access Layer (`lib/auth/dal.ts`), the login UI, or the CLI user-management tool — those are steps 06, 06, 07, and 05 respectively. The invariant this step establishes for later steps is: *anywhere in the server codebase, the single source of truth for "is this cookie bound to a live, non-expired user session?" is `validateSession(token)` in this file, and revocation is a single row delete.*

## 2. Prerequisites

- **Step 02 — drizzle sqlite scaffolding.** The singleton drizzle client at `lib/db/client.ts`, the `db/schema/index.ts` barrel, the `drizzle-kit generate` pipeline, and `scripts/migrate.ts` must already exist. This step imports the client and appends a new schema module to the barrel.
- **Step 03 — user schema and argon2 hashing.** The `users` table (primary key `id: text`) must already exist so the `sessions.user_id` foreign key can reference it with `ON DELETE CASCADE`. This step does not import `@node-rs/argon2` or any password helpers — they are orthogonal to session management.

## 4. Research notes

- `node_modules/next/dist/docs/01-app/02-guides/authentication.md § Database Sessions` — authoritative Next 16 example of exactly the pattern this step implements. The doc shows a Server-Action-based `createSession(id)` that (1) inserts a row into a drizzle `sessions` table with `userId` + `expiresAt`, (2) returns the session ID, (3) calls `(await cookies()).set('session', ..., { httpOnly: true, secure: true, expires: expiresAt, sameSite: 'lax', path: '/' })`. This step follows the structural shape but deviates on one detail: the doc encrypts the session id before putting it in the cookie ("2. Encrypt the session ID"); we store the raw opaque token because our token is already 256 bits of CSPRNG output with no semantic content — encryption adds only ceremony and a key-management burden with no confidentiality win over a random, long, opaque value. This is noted as an intentional deviation, not an oversight.
- `node_modules/next/dist/docs/01-app/02-guides/authentication.md § Deleting the session` — confirms that `deleteSession` is just `(await cookies()).delete('session')` inside an `import 'server-only'` module, and must be called from a Server Action (not a Server Component render path), because the HTTP spec forbids setting cookies after streaming starts. This step's `clearSessionCookie` matches that contract and its docstring reiterates the "call only from a Server Action or Route Handler" caveat for step-07 callers.
- `node_modules/next/dist/docs/01-app/03-api-reference/04-functions/cookies.md` — the definitive reference for the v16 `cookies()` API. Three facts drive the implementation: (1) `cookies()` is an **async** function; every call must be `await`ed and untyped/unawaited code breaks silently (see "Good to know" bullet 1). (2) The `set` and `delete` methods can only be called from a Server Function or Route Handler, not during Server Component rendering. (3) Supported options on `set` include `httpOnly`, `secure`, `sameSite: 'lax' | 'strict' | 'none'`, `path`, `expires: Date`, and `maxAge` — exactly the fields this step needs.
- `node_modules/next/dist/docs/01-app/02-guides/upgrading/version-16.md § Async Request APIs (Breaking change)` — confirms that as of v16 the synchronous compatibility shim from v15 is *fully removed*: `cookies()`, `headers()`, `draftMode()`, `params`, and `searchParams` are now `async`-only. There is no fallback. Forgetting to `await` yields a Promise whose methods are all `undefined`, which crashes with an opaque error well downstream of the mistake. This is already captured as a bullet in CLAUDE.md "Known gotchas"; this step's implementation calls it out in a source comment at the top of every `cookies()` helper.
- `node_modules/next/dist/docs/01-app/02-guides/data-security.md § Preventing client-side execution of server-only code` — authoritative rationale for the `import 'server-only'` directive at the top of `lib/auth/sessions.ts`. The doc explains that the `server-only` package is how Next/React enforces that a module never ends up in the client bundle; importing it from a client component throws a build-time error. Without this guard, Turbopack would try to follow the `better-sqlite3` native binding dependency chain into the browser bundle and fail opaquely. This also matches the convention locked in by CLAUDE.md "Database access patterns": every file that imports `lib/db/client.ts` starts with `import 'server-only'`.
- https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html (WebFetched) — OWASP's session management guidance. Three recommendations drive this step: (a) session tokens must come from a CSPRNG with **≥ 64 bits of entropy**; we use `crypto.randomBytes(32)` (256 bits) hex-encoded to a 64-character string, which blows past the floor by a factor of four and matches the "opaque, meaningless token" guidance. (b) The Secure, HttpOnly, and SameSite flags are all recommended together; we set `httpOnly: true`, `secure: process.env.NODE_ENV === 'production'`, `sameSite: 'lax'`, and `path: '/'` on every cookie write. (c) Server-side invalidation on logout is mandatory; the `destroySession` function does a single `DELETE` on the row, which is exactly the "revocation is a single DELETE on the row" promise in CLAUDE.md "Authentication patterns".
- https://orm.drizzle.team/docs/sql-schema-declaration (WebFetched) — drizzle's reference for declaring sqlite tables with foreign keys. Key fact: `references(() => users.id, { onDelete: 'cascade' })` in an `sqliteTable` column declaration is the correct way to emit `FOREIGN KEY(user_id) REFERENCES users(id) ON DELETE CASCADE` in the generated SQL. Text primary keys are `text('id').primaryKey()`. Integer epoch-ms timestamps are `integer('expires_at', { mode: 'timestamp_ms' })` (or `{ mode: 'number' }` for a raw number column). The `created_at` default-now idiom is `integer('created_at', { mode: 'timestamp' }).default(sql\`(unixepoch())\`).notNull()` (sqlite's `unixepoch()` returns seconds; `unixepoch('subsec')` returns ms) — we use timestamp-ms for `expires_at` because arithmetic with Date.now() is natural, and timestamp (seconds) for `created_at` because the exact resolution is irrelevant for audit trails.
- **Path not taken — JWT (stateless session) instead of a database-backed session row.** The Next 16 authentication doc presents both patterns side by side ("Stateless Sessions" with Jose and "Database Sessions" with a drizzle table). A JWT would remove the database round-trip on every request, which sounds attractive. It is rejected for three reasons specific to msksim: (1) **Revocation.** A JWT is valid until it expires; there is no way to force-log-out a user short of rotating the signing key (which invalidates *every* session). With a database row, `destroySession(token)` is one `DELETE` and the next request on that cookie is unauthenticated. For a two-researcher tool where a stolen laptop is a plausible threat model, immediate revocation is worth one sqlite lookup per request. (2) **Secret management.** A JWT needs a signing secret stored in an env var; losing or rotating it is a footgun. A random opaque token has no secret — the token *is* the credential, stored in the only place that matters. (3) **Next.js v16 guidance.** The authentication doc's "Database Sessions" section is the pattern Vercel recommends when you already have a database, which we do (step 02). The doc is explicit: *"You may opt to use database sessions for more advanced use cases, such as … give users the ability to log out of all devices."* That maps directly onto our "immediate revocation" requirement.
- **Path not taken — store the session id in a JWT-signed cookie anyway, "belt and suspenders".** Tempting because it prevents cookie tampering (an attacker who swaps the cookie value can't forge a valid JWT). Rejected because (a) the token is already unguessable CSPRNG output, so tampering reduces to brute force, which the 256-bit space defeats; (b) a mismatched token simply fails the `validateSession` row lookup and the user gets logged out — the failure mode is identical to a legitimate expired session; (c) adding JWT signing reintroduces the secret-management problem we rejected above for no measurable gain. Plain opaque token, plain cookie.
- **Path not taken — `better-auth`, `lucia-auth`, `next-auth` / `authjs`, or `iron-session`.** All are capable and would shortcut this step, but each brings opinions we do not want: `next-auth` is OAuth-provider-centric and overkill for two researchers with local credentials; `lucia-auth` is close to what we need but adds an adapter layer between our drizzle client and the session table, which hides the schema from our migrations; `iron-session` is stateless (sealed cookie) and therefore inherits the no-revocation problem from the JWT path-not-taken; `better-auth` bundles user management, password hashing, OAuth, and magic links, which is far more surface area than we need. Rolling our own ~80 lines of table + cookie glue against the drizzle client we already own is the lower-risk path and leaves zero library coupling in the auth layer. It is also what the Next 16 authentication doc's "Database Sessions" example is — a do-it-yourself pattern.

Total citations: 9 (4 local Next doc sections, 2 external URLs WebFetched, 3 paths-not-taken). Exceeds the ≥ 5 floor and each sub-floor (≥ 3 local, ≥ 2 external, ≥ 1 path-not-taken).

## 7. Implementation approach

The work is four sequential slices: **(1) schema**, **(2) migration**, **(3) service**, **(4) tests**. Do not reorder — the schema must generate a migration before the service can import the table type, and the service must exist before its tests can import it.

Slice one is the schema file. Declare `sessions` as a `sqliteTable` with four columns. The primary key is `id: text('id').primaryKey()` — no autoincrement, no default, the value is the random token itself supplied on insert. Storing the token *as* the primary key is load-bearing: it makes `validateSession` a single primary-key lookup, it makes `destroySession` a single primary-key delete, and it encodes the "token *is* the credential" design directly into the schema. `user_id` is `text('user_id').notNull().references(() => users.id, { onDelete: 'cascade' })` — the `ON DELETE CASCADE` is the contract that deleting a user row automatically evicts all of that user's sessions, which is how the CLI `users.ts remove` subcommand in step 05 will get logout-everywhere behaviour for free. `expires_at` is `integer('expires_at', { mode: 'timestamp_ms' }).notNull()`; timestamp_ms stores a JavaScript `Date` as a Unix-epoch-ms integer and hands it back as a `Date` on read, which is the ergonomic format for the `Date.now() > expiresAt.getTime()` expiration check in `validateSession`. `created_at` is `integer('created_at', { mode: 'timestamp' }).notNull().default(sql\`(unixepoch())\`)`; seconds-resolution is fine for an audit trail and the sqlite `unixepoch()` default means we do not need to pass a value on insert. Justify the integer-epoch-ms choice for `expires_at` in a comment above the column: ISO strings would require parsing on every read, and integer comparison is cheaper and timezone-free.

Slice two is the migration. Add the new module to the `db/schema/index.ts` barrel, then run `npx drizzle-kit generate` (the same invocation pattern step 02 established). This emits a new SQL file under `db/migrations/` containing the `CREATE TABLE sessions (...)` plus the `FOREIGN KEY(user_id) REFERENCES users(id) ON DELETE CASCADE` clause. Commit the generated file **verbatim** — do not hand-edit it; the file name embeds a timestamp that drizzle uses to order migrations, and the content is the authoritative output of the generator. Run `npx tsx scripts/migrate.ts` against a scratch copy of the dev database to confirm the file applies cleanly, then delete the scratch db. Leave `data/msksim.db` untouched so the next step can pick up the migration during its own startup.

Slice three is the service. Every public function is async and server-only. `createSession(userId: string, ttlSeconds: number = DEFAULT_SESSION_TTL_SECONDS)` generates a token with `randomBytes(32).toString('hex')` (64 hex characters, 256 bits of entropy — well above OWASP's 64-bit floor), computes `expiresAt = new Date(Date.now() + ttlSeconds * 1000)`, inserts a row, and returns `{ token, expiresAt }` so the caller (step 07's login Server Action) can pass both to `setSessionCookie`. Default TTL is seven days (`7 * 24 * 60 * 60 = 604_800` seconds) as a module-level constant; step 07 and step 05 can override per-call. `validateSession(token: string)` does `db.select().from(sessions).where(eq(sessions.id, token)).limit(1)`, returns `null` for an empty result, compares `row.expiresAt.getTime() < Date.now()` and returns `null` for expired (**without** deleting the row — opportunistic cleanup belongs to a future sweeper step, not the request-path hot loop), and otherwise returns `{ userId: row.userId, expiresAt: row.expiresAt }`. Also return `null` for any empty-string or undefined-looking token input, so `getSessionTokenFromCookie()` returning `null` flows cleanly into a null validate result. `destroySession(token: string)` does `db.delete(sessions).where(eq(sessions.id, token))` — one row, unconditionally; idempotent on a missing token.

The three cookie helpers wrap Next.js's async `cookies()` API. **Every call site awaits `cookies()`.** At the top of the file, include a single comment block (not a per-function repeat) that reads roughly: *"cookies() is async in Next.js v16; forgetting `await` yields a Promise whose .get/.set/.delete are undefined and blows up downstream with an opaque error. See CLAUDE.md 'Next.js 16 deltas' and 'Known gotchas'. See also node_modules/next/dist/docs/01-app/02-guides/upgrading/version-16.md § Async Request APIs (Breaking change)."* `setSessionCookie(token, expiresAt)` calls `(await cookies()).set(SESSION_COOKIE_NAME, token, { httpOnly: true, secure: process.env.NODE_ENV === 'production', sameSite: 'lax', path: '/', expires: expiresAt })`. `clearSessionCookie()` calls `(await cookies()).delete(SESSION_COOKIE_NAME)`. `getSessionTokenFromCookie()` returns `(await cookies()).get(SESSION_COOKIE_NAME)?.value ?? null`. The cookie name is the module constant `SESSION_COOKIE_NAME = 'msksim_session'`, also exported so `proxy.ts` in step 06 can check for its presence by name without hardcoding the string.

One subtlety for the cookie-write helpers: the v16 docs note that `set` and `delete` can only be called in a Server Function / Route Handler, not during Server Component rendering. That restriction is a caller obligation — this file just provides the function and cannot enforce it at type level. Document it in each helper's JSDoc (`@remarks Must be called from a Server Action or Route Handler; calling during Server Component render will throw.`) so step 07 and step 06 callers don't get bit.

Slice four is the tests. See §9 for the enumerated list. Test hygiene: (a) each test case gets a fresh in-memory sqlite database built from a small helper that opens `better-sqlite3(':memory:')`, runs `PRAGMA foreign_keys = ON`, and applies the current drizzle schema via `pushSchema` or a direct `CREATE TABLE` emitted from the schema objects (the pattern step 02's own tests establish — reuse it, do not reinvent); (b) the `cookies()` stub is injected per-file via `vi.mock('next/headers', ...)` with an in-module `Map` closed over by the stub; (c) because timestamps are involved, use `vi.setSystemTime(new Date('2025-01-01T00:00:00Z'))` for determinism, and advance with `vi.setSystemTime` for the expiration test rather than real `setTimeout`.

Three gotchas this step must explicitly call out and handle:

1. **Async `cookies()`.** The v16 change is non-negotiable and silent. Every single cookies call in this file must `await`; the code review for this step should grep for `cookies(` in `lib/auth/sessions.ts` and confirm every occurrence is preceded by `await`. This is already in CLAUDE.md "Known gotchas" but the reminder is load-bearing enough to rephrase in a source comment.
2. **SQLite foreign keys are OFF by default.** `better-sqlite3` does not enable `foreign_keys` automatically — the `ON DELETE CASCADE` in the migration SQL will be a no-op unless the client issues `PRAGMA foreign_keys = ON;` after opening the connection. Step 02 is supposed to do this in `lib/db/client.ts`; if it did not, this step's cascade test will fail and the fix is to add the PRAGMA there (inside the singleton initializer) rather than in this step's code. Note this in a comment in the test file so a future reader does not mis-diagnose the test.
3. **Do not import from `lib/auth/sessions.ts` into a client component.** The `import 'server-only'` directive enforces this at build time, but the docstring at the top of the file should state the rule so hand-written tests and future utilities do not inadvertently leak it. This is CLAUDE.md "Database access patterns" restated for the auth layer.

## 8. Library choices

- **Token generation: `crypto.randomBytes(32).toString('hex')` from `node:crypto`.** Node core, zero dependencies, 256 bits of entropy. No alternative considered because there is no meaningful competitor — `nanoid` is URL-safe but shorter-alphabet (collision space lower per char), `uuid.v4` is deliberately *not* full-entropy (bits are reserved for version/variant fields), and no external library improves on the built-in CSPRNG.
- **Database schema: `drizzle-orm/sqlite-core` (already installed in step 02).** No change.
- **Cookie API: `next/headers` `cookies()` (v16 async form, already available).** No change.
- **Server-only guard: `server-only` package (installed by step 02 convention).** No change.
- **Test runner: Vitest (step 00).** No change.
- No new `package.json` entries.

## 9. Unit tests

All live in `lib/auth/sessions.test.ts`. Each test seeds a dedicated in-memory drizzle database and a stub `cookies()` Map before running. The cascade test explicitly enables `PRAGMA foreign_keys = ON`.

1. **create → validate round-trip.** Insert a user row directly. Call `createSession(user.id, 3600)`. Assert the returned `token` is a 64-character lowercase hex string. Assert `expiresAt` is approximately one hour in the future (±1 s window). Call `validateSession(token)` and assert the returned `userId` equals `user.id` and the returned `expiresAt` equals the one from `createSession` (same millisecond). This is the happy path.
2. **validate for an unknown token returns null.** `validateSession('0'.repeat(64))` with no session rows in the database. Assert return value is `null`. Also assert that passing the empty string, a non-hex string, or a short string returns `null` without throwing (defensive but cheap).
3. **validate for an expired session returns null.** Insert a user row. Call `createSession(user.id, 60)` (1-minute TTL). Use `vi.setSystemTime` to advance virtual time by 2 minutes. Call `validateSession(token)` and assert `null`. Re-query the `sessions` table directly with drizzle and assert the expired row is **still present** — this step does not lazily delete; that is deliberate and the test pins the invariant.
4. **destroy makes validate return null.** Create a session, validate it (sanity check non-null), then `destroySession(token)`, then `validateSession(token)` again and assert `null`. Also assert the row count in `sessions` dropped by exactly 1.
5. **two consecutive createSession calls produce distinct tokens.** Call `createSession(user.id)` twice in the same millisecond (no time advance). Assert the two returned `token` values are not equal and both are present as rows in the `sessions` table with the same `user_id`. This pins that token uniqueness comes from the CSPRNG, not from any time component.
6. **foreign-key cascade: deleting a user cascades to sessions.** Insert a user, call `createSession(user.id)` three times. Assert `sessions` has three rows with that `user_id`. Delete the user row (`db.delete(users).where(eq(users.id, user.id))`). Re-query `sessions` and assert zero rows remain for that user. Prerequisites: `PRAGMA foreign_keys = ON` must be set on the test connection — the test helper does this explicitly and a comment in the test body explains why (because `better-sqlite3` defaults to OFF).

Optional (stretch, only if quick): a seventh test that exercises `setSessionCookie` and `getSessionTokenFromCookie` through the stubbed `next/headers` mock to verify the round-trip through the cookie store. Not required to land the step; the primary tests cover the authorization-critical paths.

## 11. CLAUDE.md updates

Append to CLAUDE.md "Authentication patterns" section (under the existing "Established in steps 03-07" bullet list; ≤ 12 lines added):

- The session token is 32 bytes from `crypto.randomBytes`, hex-encoded to a 64-character string. No JWT, no signing, no rotation keys.
- Default session TTL is 7 days, configurable per-call via the second argument to `createSession`.
- Cookie name is `msksim_session`. Flags: `HttpOnly`, `SameSite=Lax`, `Secure` in production, `Path=/`, `expires` set to the row's `expires_at`.
- The row's primary key *is* the cookie value; there is no separate session id.
- `validateSession` does not lazily delete expired rows — a future sweeper step will handle GC. Callers must not rely on post-validation cleanup.

Append to CLAUDE.md "Known gotchas" (≤ 2 lines):

- `better-sqlite3` opens connections with `PRAGMA foreign_keys = OFF` by default; the singleton client must set it `ON` explicitly or `ON DELETE CASCADE` is silently a no-op. (*Rationale*: sqlite's default is off for historical back-compat, and drizzle-kit emits the constraint in SQL but cannot set the pragma for you.)

## 1. Goal

Deliver **F15 — Recorded runs** from `docs/spec.md` §4.4 end-to-end. Step 26 wires the full lifecycle of a simulation run: when a run completes in the playground (either because the configured `tickCount` has been reached or the researcher clicks a "Save run" action), the worker's final metric stream is handed off to a Server Action which opens a single drizzle transaction and writes one row into `runs`, N long-format rows into `tick_metrics`, and K sampled rows into `snapshots`. Once persisted, the run is visible at `/runs` — an authenticated Server Component page that reads via the helpers landed in step 08, shows a filterable / sortable table (by classification, by config, by date range, by `tickCount`), and offers "View" and "Delete" actions per row. Clicking "View" lands on a dynamic Server Component route `app/(auth)/runs/[id]/page.tsx` that re-hydrates the run's time series out of `tick_metrics`, feeds it into the **same** `MetricsDashboard` component step 22 built for the live playground (reused with zero duplication), shows the end-of-run `RunSummary` from step 17 as a classification card, and offers a "Reopen in playground" link that forwards the saved `configId` and `seed` back to `/playground?configId=...&seed=...` so the researcher can replay the exact experimental conditions. Deleting a run invokes another Server Action whose body is a single `deleteRun(id)` call — FK cascade from step 08 handles `tick_metrics` and `snapshots`. This step ships no new simulation semantics, no new metrics, no new chart components, and no new database tables: it is a **persistence + navigation** step whose load-bearing correctness is (a) the atomic transaction pattern for the ~140k-row `tick_metrics` insert, (b) the faithful reuse of `MetricsDashboard` for the detail page, and (c) the round-trip "save → list → reopen → delete" integrity verified by the MCP script in section 10. F15's acceptance criterion "runs survive page reload" is the outcome criterion that matters most, and the whole step exists to give the researchers that guarantee on top of the server-side SQLite substrate the user override already chose in `CLAUDE.md` "Stack and versions".

Scope boundary: step 26 does **not** implement the batch queue (step 27), does **not** implement parameter sweeps (step 28), does **not** implement run comparison (step 29 — comparison view across multiple runs), and does **not** implement CSV/JSON export (step 30 — that is the streaming API route for F16). It also does **not** introduce any new chart library, does **not** add per-world snapshot replay UI beyond a placeholder card (the step-25 config editor's replay is the entry point for deep snapshot inspection in later waves), and does **not** mutate the step-22 `metrics-history.ts` buffer — the run detail page constructs a brand-new `MetricsHistory` from the loaded `tick_metrics` rows and passes it to `MetricsDashboard` as a normal prop, treating the dashboard as the already-built read-only consumer it was designed to be.

## 3. Spec references

- `docs/spec.md` **§4.4 Persistence and export**, specifically **F15 Recorded runs**: *"Every completed run is persisted to browser-local storage (IndexedDB) with its config, seed, tick-by-tick metrics, and a summary snapshot. A runs browser lists them with filter/sort, and each run can be re-opened to any of the live views. Acceptance: Runs survive page reload; the browser can store at least 1000 runs of ~10k ticks each before hitting quota warnings; an explicit 'clear all runs' action is available. Supports: RQ1–RQ5."* This is the authoritative contract step 26 delivers, with the single substrate override per `CLAUDE.md` "Stack and versions": **IndexedDB becomes server-side SQLite via drizzle**, all other F15 requirements remain as stated. The "1000 runs × ~10k ticks" quota target is the source of the acceptance-script "store 10 small runs and verify the browser lists all 10" check in §10 (scaled down for MCP-run speed) and the source of the persistence pattern discussion in §7 (a 10k-tick run × 10 metrics × 2 worlds ≈ 200k rows, so 1000 runs ≈ 200M rows in `tick_metrics` — well within SQLite's comfortable range per the research notes in §4). The "re-opened to any of the live views" phrase directly motivates the detail page's reuse of `MetricsDashboard` and the "Reopen in playground" link.

- `docs/spec.md` **§7.1 Per-tick scalar metrics** — defines the `ScalarMetricsSnapshot` fields that populate `tick_metrics` rows (communication success rate, mean token weight, token weight variance, Nw, matching rate, etc.). Step 26 does not compute these metrics; it **serializes** them from the worker's `TickReport[]` into the long-format `tick_metrics` row shape. The serialization is purely a flattening: each `TickReport` at tick `t` produces one row per `(world, metricName, metricValue)` triple, where `world ∈ {'world1', 'world2', 'both'}` matches the enum from step 08's schema and `metricName` is a stable snake_case string (`'success_rate'`, `'mean_token_weight'`, `'distinct_active_tokens'`, etc.). The exact list of metrics written and their canonical names is documented in §7 slice two as a constant; it is the union of what step 15 and step 16 emit. The step-22 dashboard already reads these paths out of the `TickReport`, so the serialization used here mirrors the dashboard's shaper functions field-for-field to guarantee that a reopened run renders identically to its live-running counterpart.

- `docs/spec.md` **§7.2 Per-tick tensor snapshots** — optional, sampled. Default interval "every 10 ticks" per the spec and the step-01 config schema's `sampleInterval`. Step 26 honors §7.2 by writing snapshots at that interval **only if** they were computed during the run; v1 of the worker (step 20) does not necessarily emit tensor snapshots in its `RunResult` by default (step 20's focus was metric streaming, and snapshot computation is a separate method the worker exposes but which the playground may not call). The decision for step 26: **snapshots are optional**. If the `RunResult` carries a non-empty `snapshots` array, persist it via `saveSnapshot` in a loop (N ≤ 1000 even for long runs at the default sampling interval). If the array is empty or absent, skip snapshot persistence — the run is still a valid F15 artifact because the per-tick metrics give the researcher everything F15 literally promises ("tick-by-tick metrics"). This matches the step-specific context's phrasing: "optionally sampled snapshots".

- `docs/spec.md` **§7.3 Summary metrics (end of run)** — the `RunSummary` shape step 17 produced and step 26 persists into `runs.summary_json` and `runs.classification`. The classification field (`assimilated | segregated | mixed | inconclusive`) is the single most surface-level display on both the runs browser table and the detail page — it is the one-glance answer to "did this experiment produce the outcome I hypothesized?". Section 7 documents a small `formatClassificationLabel(classification)` helper that maps the enum to a human-friendly label with an Okabe-Ito color badge (reusing the same palette step 22 chose for chart series). That label is the column in the runs browser and the headline on the detail page.

- `docs/spec.md` **§1.2 RQ1 — Assimilation vs. segregation thresholds** and **§5.1 US-6/US-7** — *"I want every run to be reproducible from its seed and config"* and *"I want to save a run today and revisit it tomorrow"*. These are the user stories step 26 directly satisfies. Reproducibility from seed is why the runs browser table displays `seed` as a first-class column (so the researcher can visually confirm two side-by-side runs share the same seed) and why the "Reopen in playground" link forwards the seed as a query parameter (so the playground can pin its RNG to the exact value the saved run used). The "save today, revisit tomorrow" requirement is the motivation for persisting the full `tick_metrics` time series rather than just the `RunSummary` — a researcher returning to a run next week may want to re-plot it at a different Y-axis scale or zoom into a specific tick range, and that is only possible if the underlying data is still there.

### Local Next.js 16 documentation

1. **`node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/page.md`**, the entire file (240 lines). This is the canonical reference for the `page.tsx` file convention under the App Router in Next 16. Load-bearing facts for step 26: (a) `page.tsx` is the leaf of a route segment, required to make a route publicly accessible — so `app/(auth)/runs/page.tsx` is what renders at `/runs` and `app/(auth)/runs/[id]/page.tsx` is what renders at `/runs/<id>`. (b) Pages are Server Components by default, which is exactly what step 26 wants for both the index and detail pages: reads go through the DAL (`verifySession()`) and the step-08 helpers in the Server Component body, and any client interactivity (the delete-confirmation dialog, the filter dropdowns) is delegated to small client-component children marked with `'use client'`. (c) The `params` prop for dynamic segments is a **`Promise`** in Next 16 — `params: Promise<{ id: string }>` — and must be `await`ed inside the Server Component. Forgetting the `await` produces a silently Promise-shaped object per the "Known gotchas" entry in `CLAUDE.md`. Step 26's detail page body begins with `const { id } = await params;` immediately after the session check. (d) `searchParams` is also a `Promise` in Next 16 — step 26's runs index page uses `searchParams` to carry filter state through the URL (`/runs?classification=assimilated&configId=xyz`), so the page body awaits it and derives the active filter set before calling `listRuns(...)`. Using URL-driven filter state rather than React state is deliberate: it makes the runs browser bookmarkable and shareable, which is the same affordance `docs/spec.md` F11/US-12 asks for on the config-library side ("share a colleague's exact config via URL").

### External WebFetched references

5. **Drizzle ORM — Transactions**, `https://orm.drizzle.team/docs/transactions` (WebFetched at plan-write time). Load-bearing facts confirmed by the fetch: (a) the drizzle transaction API is `await db.transaction(async (tx) => { ... })` where `tx` is a transaction-scoped query interface with the same surface as the top-level `db` singleton — every read and write inside the callback uses `tx.select(...)` / `tx.insert(...)` / `tx.update(...)` / `tx.delete(...)` instead of `db.*`. (b) The callback is wrapped in a SQLite `BEGIN; ... COMMIT;` (or `ROLLBACK;` on a thrown error) automatically — the implementor does not write those statements by hand. (c) Thrown errors **and** explicit `tx.rollback()` both abort the transaction and re-throw; the caller of `db.transaction(...)` must either handle the error or let it propagate. (d) Nested transactions are supported via SQLite SAVEPOINTs, but step 26 does not use nested transactions — the single top-level transaction in `persistCompletedRun` is sufficient because it is the only caller and it owns the whole payload. (e) **Performance characteristic specific to `better-sqlite3`**: because better-sqlite3 is synchronous under the hood, wrapping 100k+ inserts in a single transaction converts 100k auto-commits into 1 commit, and the total wall-clock time collapses from multiple seconds to well under 100 ms. This is the exact pattern step 08's `insertTickMetrics` already implements; step 26 simply invokes that helper inside the bigger transaction that also calls `createRun` (or `finishRun`) and the per-snapshot saves, ensuring the whole "one completed run" write is atomic — either every row lands or the caller sees a clean rollback and no partial state. The atomicity matters for the "1000 runs per user" F15 acceptance target: a partial write on a crash would leave the browser showing a row that has no metrics, which would be worse than no row at all. Section 7 slice three documents the exact transaction body: `createRun` (or if the run row was created at start, `updateRunStatus` to `'running'`) → `insertTickMetrics` → optional `saveSnapshot` loop → `finishRun` (which updates status to `'completed'`, sets `finishedAt`, writes `summaryJson`, writes `classification`, writes `tickCount`). Any throw inside the callback rolls the whole thing back. Because step 08's `insertTickMetrics` already opens its own inner transaction, **the step 26 implementing claude must verify at execution time whether better-sqlite3 supports nested transactions via SAVEPOINT in the installed version** — the drizzle docs confirm it does, and if it does not, step 26 refactors to call the raw drizzle insert loop directly (without the helper's inner transaction) inside the outer transaction. The expected path is the first one — drizzle + better-sqlite3 handles nesting cleanly in all recent versions.

6. **Drizzle ORM — Select queries and filters**, `https://orm.drizzle.team/docs/select` (WebFetched at plan-write time). Confirms the filter composition pattern step 26's `listRuns(...)` helper extension uses. Load-bearing facts: (a) Multi-condition `WHERE` clauses compose via `and(cond1, cond2, ...)` from `drizzle-orm`. (b) Range queries on timestamp columns use `gte(col, date)` and `lte(col, date)` for the date-range filter in the runs browser. (c) `orderBy(desc(col))` and `orderBy(asc(col))` are the canonical sort helpers. (d) `.limit(n).offset(m)` is the pagination primitive. (e) `inArray(col, values)` is how the runs browser's classification multi-select would filter to a subset of classifications (used in v2 if the researchers want multi-classification filtering; v1 uses a single-select per §5). Section 7 slice five documents the specific query the server component issues: `db.select().from(runs).where(and(eq(runs.status, 'completed'), /* optional classification filter */, /* optional configId filter */, /* optional date-range filter */)).orderBy(desc(runs.finishedAt)).limit(50).offset(pageOffset)`. Step 08 shipped a `listRuns(opts)` helper with `{ limit, configId, status, createdBy }` filters; step 26 extends it **in a small additive way** to also accept `{ classification?: RunClassification; finishedAfter?: Date; finishedBefore?: Date; orderBy?: 'finishedAt' | 'tickCount'; offset?: number }`, without breaking any existing caller signatures. The extension is recorded in §6 as a modification to `lib/db/runs.ts`.

### Path not taken

7. **Storing full agent-state snapshots at every tick in a dedicated table, so the detail page can replay the lattice canvas tick-by-tick — rejected.** The obvious "most satisfying" design is: every tick writes its full `Map<Language, Map<Referent, Map<Token, Weight>>>` inventory for every agent into a `snapshots` row, so a researcher revisiting the run next week can scrub through the lattice canvas at any tick. **Rejected for three concrete reasons**: (a) *Size*: at N = 500 per world × 2 worlds × 10k ticks × ~10 token entries per agent × ~40 bytes per JSON entry = ~4 GB per run. The F15 acceptance target "1000 runs per user" would demand 4 TB per user. SQLite can hold it, but `better-sqlite3` would spend many seconds serializing each row and the page file would dwarf the researcher's laptop SSD. (b) *Spec alignment*: `docs/spec.md` §7.2 explicitly says snapshots are *"sampled"* at a configurable interval (default every 10 ticks), not every tick — the spec already rejects this design for the same size reason. (c) *Use case*: no user story in §5 asks for tick-by-tick replay; US-7 ("save a run today and revisit it tomorrow") and US-8 ("pause and inspect agent inventories") are satisfied by sampled snapshots every 10 ticks plus the cumulative time-series metrics. Step 26 honors the spec: snapshots are persisted only when the worker supplied them, at whatever interval the config specified, and the detail page uses them **only** to render a summary card or a small timeline of "sampled inventories at ticks [10, 20, 30, ...]" rather than to drive a full tick-by-tick canvas replay. Tick-by-tick canvas replay, if ever wanted, is a v2 feature built on top of a different storage substrate (probably a compact binary snapshot format and memory-mapped reads), not on top of the JSON TEXT snapshots table this step consumes.

8. **Running the transaction body inside `db.transaction` on the client side via a Comlink-proxied call to a server-thread transaction runner — rejected.** Considered briefly because the worker already has the `RunResult` in memory and "just shipping it to the DB" in-place would feel symmetric. **Rejected** because (a) the worker runs in the browser and has no network access to the SQLite file (the DB is on the Node server, not the client), (b) a "Comlink to server" bridge would require a WebSocket or server-sent events channel that is out of scope for v1, and (c) Server Actions are the canonical Next 16 pattern for exactly this handoff (client → server, typed, authenticated, transactional). Step 26 uses Server Actions as documented in research note 3, not a bespoke RPC layer. Recording the rejection so a future agent does not revisit the design without reading this rationale.

### Additional informational citation

9. **SQLite WAL mode and concurrent read performance**, `https://www.sqlite.org/wal.html` (referenced for context, not WebFetched as a hard dependency). Informational only: WAL mode would let the runs browser's read queries proceed concurrently with an in-flight `persistCompletedRun` write, which matters at very high ingest rates. **Deferred**: step 02's `lib/db/client.ts` initializer may or may not have enabled WAL — the step-26 implementing claude checks and enables `PRAGMA journal_mode=WAL;` inside the client singleton **only if** it is not already on. At v1 scale (one researcher, one config-running session at a time) the default rollback-journal mode is fine. This is not a strict prerequisite; it is recorded here so a future performance investigation has a pointer.

- **`app/(auth)/playground/actions.ts`** — **extend or create** (depends on whether step 25 already created this file for config save actions). If step 25 created it, append the new Server Actions to the existing file. If not, create a fresh file with `'use server'` at the top and the imports for `@/lib/auth/dal`, `@/lib/db/runs`, `@/lib/db/tick-metrics`, `@/lib/db/snapshots`, `@/lib/db/configs`, `@/lib/db/client` (for the transaction runner), `revalidatePath` from `next/cache`, and the `RunResult` / `TickReport` / `RunSummary` types from `@/lib/sim/worker-client` (which re-exports from the worker per step 20). Exports:
  - **`persistCompletedRun(payload: PersistRunPayload): Promise<{ runId: string }>`** — the primary action this step ships. `PersistRunPayload` is a locally declared interface: `{ configId: string; seed: number; tickCount: number; result: RunResult; snapshots?: Array<{ tick: number; kind: 'inventory' | 'interaction_graph'; content: unknown }> }`. Body: (1) `const session = await verifySession();` (2) `return await db.transaction(async (tx) => { ... })` where the callback: (a) inserts a new `runs` row via `tx.insert(runs).values({ configId, seed, status: 'pending', createdBy: session.userId }).returning()` (or reuses an existing pending run row if the playground created one at run start — see §7 slice two for the "create early vs. create late" design decision), (b) serializes the `result.metricsTimeSeries` into long-format rows using the `serializeTickReportsToMetricRows(runId, timeSeries)` pure helper (see §5 below), (c) bulk-inserts via `tx.insert(tickMetrics).values(rows)` — this is a raw drizzle insert inside the outer transaction, **not** a call to `insertTickMetrics` (which opens its own inner transaction; nesting works via SAVEPOINT but calling the helper twice per payload is wasteful and conceptually muddled). The raw insert respects drizzle's parameter-count limit — for 200k rows it may need to be chunked into batches of ~500 per `.values([...])` call to avoid SQLite's `SQLITE_MAX_VARIABLE_NUMBER` limit (default 32766 in recent SQLite builds, so 500 rows × 4 columns = 2000 parameters per chunk is safely under). The chunking is a simple `for (let i = 0; i < rows.length; i += 500) tx.insert(tickMetrics).values(rows.slice(i, i + 500));`. (d) Iterate `payload.snapshots ?? []` and `tx.insert(snapshots).values({ runId, ...s, contentJson: JSON.stringify(s.content) });` for each. (e) Finalize the run via `tx.update(runs).set({ status: 'completed', finishedAt: new Date(), tickCount: payload.tickCount, summaryJson: JSON.stringify(result.summary), classification: result.summary.classification }).where(eq(runs.id, runId)).returning();` (f) Call `revalidatePath('/runs');` inside the Server Action (not inside the transaction — `revalidatePath` is a cache-invalidation call, not a DB write). (g) Return `{ runId }`. Total function body: ~60 lines including comments.
  - **`deleteRun(id: string): Promise<void>`** — (1) `const session = await verifySession();` (2) Optionally verify the run belongs to the user (skip for v1 per the "every user sees every run" simplification). (4) `revalidatePath('/runs');` (5) Return void. The FK cascade from step 08 handles `tick_metrics` and `snapshots` deletion automatically.

### Pure helper modules (shared, no React)

- **`lib/sim/metrics/serialize.ts`** — **new file**, pure TypeScript, no React, no DB, no `'use server'`. Client-safe and worker-safe per the convention established in step 17 (no `import 'server-only'` because this helper is used on both sides of the Comlink boundary: the server writes it, the client might later inspect it via the run detail page). Exports:
  - `serializeTickReportsToMetricRows(runId: string, timeSeries: TickReport[]): NewTickMetric[]` — the forward transform. Walks `timeSeries` and for each `report` emits one row per `(world, metricName, metricValue)` triple. The metric names it emits are a **stable constant list** declared at the top of the file as `const METRIC_SHAPES: ReadonlyArray<{ name: string; path: (r: TickReport, world: 'world1' | 'world2' | 'both') => number | null }> = [...]`. Each entry is a tuple of `(metricName, extractor)`. The extractor returns the numeric value for that metric at that world, or `null` if the metric is not defined (e.g., assimilation index may be null on ticks with no cross-class interactions). Null values are **skipped** (no row emitted) so the resulting `tick_metrics` table has genuine absence for missing observations, which matches how the dashboard shapers treat `null` as "line gap" per step 22. The exact `METRIC_SHAPES` list is enumerated in §7 slice two below.
  - `materializeTickReports(rows: TickMetric[]): TickReport[]` — the inverse transform. Walks the sorted `TickMetric[]` and groups by `tick`. For each tick, it reconstructs a `TickReport` by de-serializing `(world, metricName, metricValue)` triples back into the nested `scalar` and `graph` object shapes. The nested shapes are built by walking a parallel metadata table that maps each metric name back to its path in the `TickReport`. Any metric present in the stored rows but not in the known shape list is ignored with a console warning (for forward compatibility with future metric additions). Metrics present in the shape list but not in the rows are left as `undefined` in the materialized report — the dashboard's shaper functions already handle `undefined` (they fall through to `null` line gaps).
  - `formatClassificationLabel(classification: RunClassification | null): { label: string; color: string }` — the display helper. Maps `'assimilated' → { label: 'Assimilated', color: '#009E73' }`, `'segregated' → { label: 'Segregated', color: '#D55E00' }`, `'mixed' → { label: 'Mixed', color: '#E69F00' }`, `'inconclusive' → { label: 'Inconclusive', color: '#56B4E9' }`, `null → { label: 'Pending', color: '#9ca3af' }`. Colors are from the Okabe-Ito palette step 22 picked.
  - File size: ~150 lines including the `METRIC_SHAPES` constant, the two folding helpers, and the label formatter. JSDoc documents the serialization contract so future metric additions (step 27, step 28, v2) know to extend the shape list in lockstep with the producer modules in `lib/sim/metrics/*`.

Also grep for `RunResult`, `appendTick`, and `MetricsHistory` to confirm step 20, 21, and 22 landed their exports where this step expects. And inspect step 25's files (when it lands) to learn how the config-editor page routes to the playground — the "Reopen in playground" link in the detail page must match that convention exactly.

**Slice two — define the metric shape list and write the serialize helper.** Create `lib/sim/metrics/serialize.ts`. Declare the `METRIC_SHAPES` constant at the top with one entry per metric the step-15/step-16 producers emit. The canonical v1 list, matching the paths step 22's dashboard shapers read:

```typescript
const SCALAR_METRICS: ReadonlyArray<{
  name: string;
  world: 'world1' | 'world2' | 'both';
  extract: (r: TickReport) => number | null;
}> = [
  { name: 'success_rate',            world: 'world1', extract: r => r.scalar.world1.successRate.rate },
  { name: 'success_rate',            world: 'world2', extract: r => r.scalar.world2.successRate.rate },
  { name: 'success_rate',            world: 'both',   extract: r => r.scalar.overall.successRate.rate },
  { name: 'mean_token_weight',       world: 'world1', extract: r => meanOfLanguages(r.scalar.world1.perLanguage) },
  { name: 'mean_token_weight',       world: 'world2', extract: r => meanOfLanguages(r.scalar.world2.perLanguage) },
  { name: 'token_weight_variance',   world: 'world1', extract: r => varianceOfLanguages(r.scalar.world1.perLanguage) },
  { name: 'token_weight_variance',   world: 'world2', extract: r => varianceOfLanguages(r.scalar.world2.perLanguage) },
  { name: 'distinct_active_tokens',  world: 'world1', extract: r => r.scalar.world1.distinctActiveTokens },
  { name: 'distinct_active_tokens',  world: 'world2', extract: r => r.scalar.world2.distinctActiveTokens },
  { name: 'matching_rate',           world: 'world1', extract: r => r.scalar.world1.matchingRate },
  { name: 'matching_rate',           world: 'world2', extract: r => r.scalar.world2.matchingRate },
];

const GRAPH_METRICS: ReadonlyArray<{ name: string; world: 'world1' | 'world2' | 'both'; extract: (r: TickReport) => number | null }> = [
  { name: 'largest_cluster_size',      world: 'world1', extract: r => r.graph.world1.largestClusterSize },
  { name: 'largest_cluster_size',      world: 'world2', extract: r => r.graph.world2.largestClusterSize },
  { name: 'cluster_count',             world: 'world1', extract: r => r.graph.world1.clusterCount },
  { name: 'cluster_count',             world: 'world2', extract: r => r.graph.world2.clusterCount },
  { name: 'interaction_modularity',    world: 'both',   extract: r => r.graph.interactionGraphModularity },
  { name: 'assimilation_index',        world: 'both',   extract: r => r.graph.assimilationIndex },
  { name: 'segregation_index',         world: 'world2', extract: r => r.graph.segregationIndex },
];
```

The `meanOfLanguages` and `varianceOfLanguages` helpers are tiny local functions that walk the `perLanguage: Record<string, { meanTokenWeight: number; ... }>` object and average over its values; they match what step 22's dashboard shaper does for the "mean token weight" chart, so the serialized value is identical to what was displayed in the live playground. The **exact paths** in the extract functions must match whatever step 15 and step 16 actually ship — the implementing claude verifies at execution time by opening those files and confirming the property names. If a property name differs (e.g. `successRate.rate` vs. `successRate.ratio`), honor the actual name. The `serializeTickReportsToMetricRows` function then walks `timeSeries`, and for each `report` and each `{ name, world, extract }` entry, computes `const value = extract(report)` and (if `value !== null && !Number.isNaN(value) && Number.isFinite(value)`) emits a row `{ runId, tick: report.tick, world, metricName: name, metricValue: value }`. NaN and infinity are treated as "missing observation" and produce no row — these values are typically SQLite-hostile (NaN stored as REAL round-trips as NULL in some drivers, which would silently corrupt the data), and treating them as absent is consistent with the dashboard's null-as-gap handling.

The inverse `materializeTickReports(rows)` walks the sorted `TickMetric[]` and reconstructs the `TickReport[]`. It groups rows by `tick`, and for each group it builds a nested object structure that matches the `TickReport` type. ## 9. Unit tests

All tests live in `lib/sim/metrics/serialize.test.ts` (new file) and in an extension to `lib/db/persistence.test.ts` (step 08's existing test file). Each test runs under Vitest's default `node` environment.

**In `lib/sim/metrics/serialize.test.ts`:**

1. **Round-trip `serializeTickReportsToMetricRows` → `materializeTickReports` preserves the time series.** Construct a synthetic `TickReport[]` of length 5 with hand-crafted, known scalar and graph values. Call `serializeTickReportsToMetricRows('test-run-id', reports)` to get rows. Call `materializeTickReports(rows)` to get `reports2`. Deep-equal `reports2` against `reports`, allowing for fields that are intentionally `undefined` vs. `null` (normalize by comparing `JSON.parse(JSON.stringify(x))` on both sides). Assert length matches. Assert every tick, world, and metric name appears in the rows exactly once. Assert the `runId` on every row equals `'test-run-id'`.

2. **Serialization skips null, NaN, and Infinity metric values.** Construct a `TickReport[]` where one tick has `graph.assimilationIndex === null` (simulating the "no cross-class interactions" case), another has `graph.interactionGraphModularity === NaN` (simulating a degenerate graph), and another has `scalar.world1.successRate.rate === Infinity` (simulating a divide-by-zero). Call `serializeTickReportsToMetricRows`. Assert the returned rows do **not** contain any row for those null/NaN/Infinity cells. Assert the rows for other metrics at the same ticks are still present (i.e., one bad metric does not poison the whole tick).

3. **`formatClassificationLabel` returns the expected label and color for every enum value.** Test all five inputs: `'assimilated'`, `'segregated'`, `'mixed'`, `'inconclusive'`, `null`. Assert each returns the documented `{ label, color }` pair (with hex colors matching the Okabe-Ito palette).

4. **Serialization produces rows in a stable order that matches the composite primary key.** Call `serializeTickReportsToMetricRows('run-id', reports)` where `reports` has 3 ticks. Assert the returned array is sorted such that applying `drizzle`'s insert on it would not produce PK collisions (i.e., no two rows share the same `(tick, world, metric_name)` for the same `runId`). This is a pre-insert sanity check: the test iterates the rows, builds a `Set<string>` of `\`\${tick}-\${world}-\${metricName}\`` keys, and asserts `set.size === rows.length`.

**In `lib/db/persistence.test.ts` (extension to the existing file):**

5. **`listRunsWithConfig` joins runs and configs correctly.** Reuse the existing in-memory drizzle test helper. Insert a config, insert two runs referencing the config (different seeds, both `completed`), then call `listRunsWithConfig({ limit: 10 })`. Assert the returned length is 2, every row has the config's name inlined, and both runs have the expected seeds. Also verify the `configHash` field is present (pulled from the join).

6. **`countRuns(opts)` returns accurate counts under filters.** Insert one config + three runs: two `classification: 'assimilated'` and one `'segregated'`. Call `countRuns({ status: 'completed' })` → expect 3. Call `countRuns({ status: 'completed', classification: 'assimilated' })` → expect 2. Call `countRuns({ status: 'completed', classification: 'segregated' })` → expect 1. Call `countRuns({ configId: 'nonexistent' })` → expect 0.

7. **Full persistCompletedRun round trip against an in-memory DB.** (Integration test.) Build a synthetic `RunResult` with 5 ticks and 7 metrics per tick, wrap the `db.transaction` pattern from the Server Action in a plain async function (the test cannot easily import a `'use server'` module without a Next.js runtime, so the test re-implements the transaction body inline to exercise the same sequence: insert run → chunk-insert metrics → update run). Call it. Query `runs` → expect 1 row with `status: 'completed'` and non-null `summaryJson`. Query `tick_metrics` → expect `5 * number_of_non_null_metrics` rows. Query `snapshots` → expect 0 rows (synthetic payload had no snapshots). Verify no orphan rows exist (every `tick_metrics.run_id` maps to the inserted run). This test is the most load-bearing integration test in the step because it exercises the exact write path the Server Action will use in production.

All seven tests are deterministic, idempotent, and run in under 1 second combined.

## 11. CLAUDE.md updates

Append to the **"Database access patterns"** section. The section's hard cap is 50 lines; step 02 and step 08 combined populate it to ~25 lines. This step appends ≤ 10 new lines documenting the batch-insert transaction pattern for `tick_metrics`:

- Bulk writes of `tick_metrics` from a completed simulation run happen inside a single `db.transaction(async (tx) => { ... })` block in the `persistCompletedRun` Server Action. The block creates or updates the `runs` row, chunks the `tick_metrics` insert into batches of 500 rows per `tx.insert(tickMetrics).values([...])` call (to stay under SQLite's `SQLITE_MAX_VARIABLE_NUMBER` limit), optionally saves sampled snapshots, and finalizes the run row. A throw anywhere inside the callback rolls the whole write back — there is no partial state on disk. A typical 10k-tick run writes ~150-200k `tick_metrics` rows; better-sqlite3's synchronous transaction runner processes this in well under 100 ms because the single commit replaces what would otherwise be 150k-200k auto-commits.

That is the exact text budget. The implementing claude verifies the section remains under its 50-line cap after the append; if not, promote the content into a new dedicated section per `CLAUDE.md` "Living-document rules" rather than truncating existing bullets.

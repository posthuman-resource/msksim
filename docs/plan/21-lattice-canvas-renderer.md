## 1. Goal

Ship the **first UI consumer** of the simulation Web Worker: a `SimulationShell` client component that owns the worker lifecycle for `/playground`, plus a per-world HTML5 Canvas 2D **lattice renderer** that implements feature **F7** (live lattice view) from `docs/spec.md` §4.2. This step replaces the step-07 stub at `app/(auth)/playground/page.tsx` (currently five lines of placeholder JSX) with a real, interactive rendering surface that draws a 2D grid of colored cells — one cell per agent in either `world1` or `world2` — and lets the researcher toggle between three F7 projections: (a) **class** (blue/purple/green/orange for the four `AgentClass` values, gray for empty lattice cells), (b) **dominant token** (distinct hues per top-weighted token for the first referent in the config), and (c) **matching rate** (a red-to-green gradient based on the fraction of each cell's neighbors whose top-weighted token for the same referent matches). Hover over a cell reveals a tooltip containing the agent's `id`, `class`, `position`, and full nested-Map inventory rendered as a JSON-ish text block — this is the F7 "hover reveals the agent's full inventory" acceptance clause. The rendering loop runs inside a `requestAnimationFrame` callback that redraws whenever the cached snapshot or the projection kind changes, and is bound by the F7 acceptance criterion: **30+ FPS for N ≤ 500 agents per world**. A world selector (a two-button toggle above the canvas) flips between `world1` and `world2`; the two worlds are *not* rendered side-by-side in step 21 (that is step 24's full-controls concern), one canvas at a time is sufficient to satisfy F7. Because the worker (step 20) does **not** push full agent states every tick (a `TickReport` carries only metric maps — the spec's §8 "rendering is decoupled from simulation" commitment), this step introduces a new worker method `getLatticeProjection(worldId, kind): Promise<CellData[]>` that returns only the minimum information each projection needs, and the main thread calls it whenever (i) the projection toggle changes, (ii) the world toggle flips, or (iii) a batch of ticks has just completed. This avoids the worst-case "50 ticks per second × 500 agent inventories per tick" message-passing load that would kill frame rate. The step 20 plan file already documented the worker's `init`/`step`/`run`/`getMetrics`/`getSnapshot`/`reset` surface; step 21 extends that surface with the projection method in a forward-compatible way (section 6 documents the mechanics). Step 21 ships no metrics dashboard (step 22), no network view (step 23), no interactive-parameter sliders (step 24), no persistence (step 26). Its acceptance surface is the F7 renderer plus the MCP script that drives it.

## 3. Spec references

- `docs/spec.md` **§4.2 F7 "Live lattice view"** — the primary feature this step implements. The F7 description says: "A per-world canvas rendering of the lattice, with each cell colored by (a) agent class, (b) dominant token for a selected referent, or (c) matching rate with neighbors. The researcher chooses which projection." F7's acceptance criteria are: "Renders at 30+ FPS for N ≤ 500 per world; projection toggle takes effect within one tick; hover reveals the agent's full inventory." Every implementation decision in this step is traceable to these three acceptance clauses. The "per-world canvas" wording clarifies that each world gets its own canvas rather than a combined view — step 21 renders one canvas at a time (with a world toggle) rather than two stacked canvases, on the grounds that the toggle is a one-line state change, the screen real estate matters for the hover tooltip, and step 24 will revisit multi-world layout once the parameter-control panel arrives.
- `docs/spec.md` **§4.2 F10 "Interactive controls"** — **not** implemented in step 21. F10 is step 24's concern and this plan file explicitly carves it out. The only play/pause affordance step 21 ships is a minimal two-state button for the MCP demo, sufficient to exercise the worker's `step()` and `run()` methods during verification. Slider-driven hot-parameter changes, variable tick-rate, RNG seed entry, and reset — all step 24. This is called out because it is tempting to ship "just one slider" during step 21, and the scope boundary must hold for the step to fit its 40-minute budget.
- `docs/spec.md` **§7.1 row 5 "Matching rate"** — "Fraction of agent pairs whose top-weighted token for a given referent agrees, averaged over referents." Step 15 implements this as a world-level scalar. Step 21's `matching-rate` projection re-uses the concept per-cell: for each agent, look at its neighbors (per the topology's `neighbors()` call), count how many share the same top-weighted token for the first referent, divide by the neighbor count, and emit that `[0, 1]` fraction as the cell's color intensity. This is a *view-layer derivation* — the per-cell matching rate is not a first-class metric in `ScalarMetricsSnapshot`, because averaging it back to a scalar loses the spatial information that F7 is designed to expose.
- `docs/spec.md` **§8 Architecture Sketch** — the ASCII diagram shows "Lattice canvas (per world)" as one of three downstream consumers of the `SimulationShell` client component, and the "Key architectural commitments" bullet "Rendering is decoupled from simulation: the worker emits metrics and snapshots at the sampling interval; the main thread chooses when to redraw. A slow renderer cannot stall the simulation." is the design contract step 21 honors by calling `getLatticeProjection` on demand rather than subscribing to a per-tick stream. A slow renderer in step 21 cannot stall the worker because the worker advances ticks via `step()`/`run()` calls driven by the main thread, and the renderer is a separate concern that polls when it is ready.
- `docs/spec.md` **§3.2 Agent state** — "`position`: lattice coordinate or network-node id. Determines who the agent can interact with. Fixed for v1 (agents do not migrate within a world)." The fixed-position invariant is load-bearing for the renderer: once `init()` has run, every agent's `(x, y)` coordinate is stable for the life of the run, so the renderer can compute cell-rect positions once at snapshot time and reuse them across every rAF tick until a `reset()` happens.
- `docs/spec.md` **§3.4 Initial conditions** — the researcher sees a *seeded, deterministic* initial lattice, so step 21's initial render matches the agent placements step 11 produces for the default config + seed. This is what makes the MCP screenshot reproducible across pipeline runs.
- `docs/spec.md` **§11 Open Question 2 "Agent count"** — "default to N = 50–500 per world for interactive playground mode." The F7 acceptance criterion of 30+ FPS is calibrated against this range; step 21 does not need to push beyond 500, and any architecture that would require e.g. a scene graph or WebGL to meet the FPS target is over-engineered for the scale. Section 4's "paths not taken" records PixiJS and SVG rejections against this ceiling.

**External references (WebFetched):**

5. **MDN `CanvasRenderingContext2D`** — `https://developer.mozilla.org/en-US/docs/Web/API/CanvasRenderingContext2D` (WebFetched during research). Load-bearing facts:
   - `fillRect(x, y, w, h)` is the fast path for filled rectangles. The fetch confirmed: "Avoid Path-Based Drawing — Don't use `beginPath()`, `rect()`, `fill()` for hundreds of rectangles; `fillRect()` is significantly faster." Step 21's renderer uses `fillRect` exclusively for cell drawing; it never calls `beginPath`/`rect`/`fill` per-cell.
   - `fillStyle` is a state property, and changing it has per-call overhead. The fetch's recommended optimization: "Batch Style Changes — Group rectangles by color; change `fillStyle` once per color group." Step 21's renderer sorts the cell list by color (projection hue) before drawing, sets `fillStyle` once per color group, and loops `fillRect` inside each group. For the `class` projection this caps the number of `fillStyle` writes at 5 (four classes + empty), regardless of N. For `dominant-token` it caps at the number of distinct tokens (typically 4-8 for v1's two-referent default). For `matching-rate` the color space is continuous, so the renderer quantizes to ~32 color bins (one `fillStyle` write per bin) rather than one per cell — this is an explicit perf optimization and section 7 documents the quantization step count as a tuning knob.
   - `getBoundingClientRect()` is how a canvas converts page-space pointer events (`event.clientX`, `event.clientY`) to canvas-local coordinates for hit testing. Step 21's `onMouseMove` handler on the canvas computes `const rect = canvas.getBoundingClientRect(); const localX = event.clientX - rect.left; const localY = event.clientY - rect.top;` then divides by the cell size (accounting for device pixel ratio) to get an `(x, y)` cell index. The hit-test is O(1) — no iteration over cells.
   - `devicePixelRatio` scaling: high-DPI displays need the canvas backing store to be `devicePixelRatio × cssPixels` wide and tall, with a corresponding `ctx.scale(dpr, dpr)` at the top of the draw call to match the CSS coordinate space. Without this, the rendering is blurry on retina / 4K screens. The fetch confirmed the pattern: `canvas.width = width * dpr; canvas.height = height * dpr; ctx.scale(dpr, dpr);`. Step 21's renderer applies this pattern inside a `useEffect` that reads the parent element's `clientWidth`/`clientHeight` via a `ResizeObserver`, sets the canvas backing-store dimensions, and resets `ctx.scale`. The `ResizeObserver` cleanup is part of the effect's return function.
   - Text drawing for the hover tooltip is deliberately **not** done on the canvas. `ctx.fillText` does not wrap, does not support CSS styling, and is hard to position precisely relative to the pointer. Instead, the `AgentTooltip` client component is a Tailwind-styled `<div>` positioned absolutely over the canvas via React state (mouse coordinates), with the tooltip's content rendered as normal React JSX (pre-formatted text inside a `<pre>` element). This matches the fetch's principle: "`fillText` is hard to position precisely and does not wrap," and keeps the tooltip styling consistent with the rest of the Tailwind 4 shell.

6. **MDN `Window.requestAnimationFrame()`** — `https://developer.mozilla.org/en-US/docs/Web/API/Window/requestAnimationFrame` (WebFetched during research). Load-bearing facts:
   - `requestAnimationFrame(callback)` is **one-shot**; the callback must re-schedule itself for the next frame if a continuous loop is desired. The returned `number` handle is passed to `cancelAnimationFrame(handle)` to stop the loop — this is the cleanup primitive the `useEffect` return function uses.
   - Using the `timestamp` argument the callback receives (rather than `performance.now()` or `Date.now()`) is the correct way to compute deltas for refresh-rate-independent animation. Step 21 does not run a time-driven animation — cells do not tween — so the renderer uses rAF as a **repaint scheduler** rather than a time pump: the callback is scheduled whenever the cached snapshot or projection changes, it runs once, it redraws the canvas, and it does **not** re-schedule unless further state changes invalidate the canvas. This minimizes CPU cost when the simulation is paused — the canvas is not being repainted 60 times per second for no reason — and still hits the F7 30+ FPS target when the simulation is running at high tick rates (each tick marks the cache as dirty and schedules one rAF callback).
   - React `useEffect` cleanup pattern for rAF: save the handle in a mutable ref (`frameRef.current = requestAnimationFrame(tick)`), return a cleanup function that calls `cancelAnimationFrame(frameRef.current)` if the handle is set. This is the pattern the fetch documented verbatim and it is also the pattern `CLAUDE.md` "Worker lifecycle" enforces for idempotent React 19 strict-mode effects. Step 21's renderer mirrors the worker-lifecycle pattern from step 20: construct in `useEffect`, clean up in the return function, tolerate strict-mode double-invocation.
   - Frame-skip when nothing has changed: the rAF callback checks a `dirty` flag (a React ref, not state, so mutation doesn't trigger a re-render) and bails without drawing when it is false. This is the "render only on demand" optimization the fetch recommended; step 21 adopts it.

**Paths not taken:**

7. **PixiJS or a scene-graph WebGL renderer.** Considered and rejected. PixiJS is a mature 2D WebGL renderer that would scale gracefully past N = 10⁴ cells. But F7's ceiling is N ≤ 500 per world and `docs/spec.md` §11 Open Question 2 pins the interactive playground upper bound at 500. At that scale, Canvas 2D's `fillRect` loop clocks in at well under 1ms per frame on any hardware the researchers will use (500 `fillRect` calls on a modern browser is ~0.1ms; the DOM/tooltip update dominates). Adding PixiJS would bring in a 300KB+ dependency, a scene-graph abstraction that the simple cell grid does not need, and a WebGL context whose loss handling must be considered across tabs and GPU driver restarts. The cost-benefit is decisively against it for v1. **Recorded as an explicit rejection** so a future agent does not revisit the design without reading this rationale first. If v2 research demands N > 10⁴ interactive rendering, Pixi (or raw `WebGL2RenderingContext`) can be introduced then, behind a feature flag that keeps the Canvas 2D path for smaller simulations.

8. **SVG rendering with one `<rect>` per cell.** Considered and rejected. SVG has compelling ergonomics — declarative React JSX, CSS hover via `:hover`, no manual rAF loop, no pointer-math hit-testing. But the DOM overhead of 500 `<rect>` elements plus per-tick attribute updates (the `fill` attribute changes when the `dominant-token` projection refreshes, for example) is **precisely** the anti-pattern the React DOM documentation warns about: `reconciler` has to diff 500 virtual DOM nodes every tick, and the browser has to re-layout the SVG for every attribute change. Empirical reports from similar grid renderers (e.g. the FreeCodeCamp ABM tutorials cited by `docs/spec.md` §9's "FlowingData 2022" link) show SVG falls below 30 FPS at roughly N = 200 on mid-range hardware — so SVG would miss the F7 acceptance criterion at the upper half of the allowed range. **Rejected categorically.** The Canvas 2D implementation has direct `fillRect` writes, no DOM churn, and a single event listener — it is faster by construction at the N ≤ 500 scale.

9. **Pushing full agent states from the worker on every tick.** Considered and rejected. The worker already has a `getSnapshot()` method from step 20 that returns a `FullStateSnapshot` with all agent inventories serialized as `[language, referent, lexeme, weight][]` quadruples. Calling that method every tick at 50 Hz would serialize ~500 agents × ~8 quadruples per inventory × 4 fields per quadruple = ~16,000 JSON primitives per tick, roughly 0.5-1 MB of `structuredClone`-able data per second at peak tick rate. This is not a crushing load in absolute terms, but it is entirely unnecessary when the class projection only needs `(id, class, position)` (~12 bytes per agent) and the dominant-token projection only needs `(id, position, topTokenForReferent0)` (~20 bytes per agent). Step 21 introduces `getLatticeProjection(worldId, kind)` that returns exactly the fields each projection needs — a *view-specific projection query* rather than a firehose. This reduces the per-request payload by ~30-50x for the class projection and allows the renderer to call the method only when the cached snapshot is stale (every N ticks, not every tick). **Recorded here as the primary architectural reason for the new worker method.**

10. **Unifying the three projections into a single `getProjectionData(worldId)` call that returns the union of fields.** Considered and rejected. A union-shaped response would simplify the worker API (one method, one call per refresh) but would pay the per-cell matching-rate computation cost on *every* refresh even when the matching-rate projection is inactive. Matching-rate is the most expensive of the three — it walks each cell's neighbor list via the topology and compares top-weighted tokens, which is O(N × avg_neighbors) ≈ O(N × 8) per call for Moore-neighborhood lattices. Doing this work when the user is viewing the cheap `class` projection is wasteful. **Rejected in favor of a kind-parameterized method** that lets the worker skip the computation for inactive projections. The three kinds share 80% of the iteration shape (walk all agents in the chosen world, extract the relevant fields), so the worker's implementation of `getLatticeProjection` is a switch-on-kind dispatch with shared agent enumeration, ~50 lines total.

11. **Rendering both worlds side-by-side with two canvases.** Considered and deferred to step 24 (interactive controls). The F7 description says "A per-world canvas rendering of the lattice" — singular "a" per world — and the acceptance criterion does not require simultaneous display of both worlds. Rendering both worlds in step 21 would double the screen-real-estate cost, force a decision about responsive stacking vs. side-by-side, and add a second pointer-event target to reason about. Step 21 ships a world selector (two buttons: "World 1" / "World 2") above a single canvas; step 24 will revisit multi-world layout once the parameter-control panel lands. **Deferred, not rejected.** The `SimulationShell` state machine already tracks `selectedWorldId` so step 24 can swap in a two-canvas layout without refactoring the shell's effect wiring.

12. **Colorblind-safe palette sourced from a formal reference (Wong 2011, or the Viridis family).** Considered and partially adopted. The four `AgentClass` colors (W1-Mono blue, W1-Bi purple, W2-Native green, W2-Immigrant orange) overlap with Wong's 7-color colorblind-friendly palette (Okabe-Ito 2002, popularized by Wong Nature Methods 2011) at blue (`#0072B2`), bluish green (`#009E73`), vermillion (`#D55E00`), and reddish purple (`#CC79A7`). Step 21's palette uses Tailwind-compatible approximations of these (`bg-blue-500`, `bg-purple-500`, `bg-green-500`, `bg-orange-500`) rather than the raw Okabe-Ito hex values, because (a) Tailwind 4's default palette is already designed to be broadly perceptible, (b) the presentation target is the researchers' own browser rather than a published figure (exports happen in step 30 via CSV for R/Python plotting, where colorblind-safety is a downstream concern), and (c) the `dominant-token` projection generates distinct hues programmatically via an HSL ring rather than from a fixed palette, so a formal colorblind reference does not apply uniformly across all three projections anyway. The `matching-rate` red-to-green gradient is a conscious departure from colorblind best practice (red-green is the canonical colorblind confusion axis), but is retained because (a) F7's description uses "matching rate" semantics where the red/green gradient is the field-standard encoding (high agreement = green, low = red), (b) the researchers are not red-green colorblind, and (c) a future v2 feature could swap in a viridis-style gradient behind a user preference. This is documented here so the palette decision is traceable.

## 5. Files to create

- `app/(auth)/playground/simulation-shell.tsx` — **`SimulationShell`** client component. Starts with `'use client'` on its own first line. Imports: `useEffect`, `useRef`, `useState`, `useCallback` from `'react'`; `createSimulationWorker` from `'@/lib/sim/worker-client'`; `type { Remote, TickReport, CellData, ProjectionKind, WorldId } from '@/lib/sim/worker-client'` (type-only); `LatticeCanvas` from `'./lattice-canvas'`; `ProjectionToggle` from `'./projection-toggle'`; `AgentTooltip` from `'./agent-tooltip'`. The component owns the following state via `useState`: `isRunning: boolean` (play/pause), `currentTick: number`, `selectedWorld: WorldId`, `projectionKind: ProjectionKind`, `cells: CellData[]`, `hoveredAgent: HoveredAgentInfo | null`. It holds the worker factory's return in refs via `useRef<{ api: Remote<SimulationWorkerApi>; terminate: () => void } | null>(null)`. The **primary** effect is a single `useEffect` with empty deps that (a) constructs the worker via `createSimulationWorker()` (SSR-safe because effects run client-side only), (b) calls `api.init(defaultConfig, seed)` with the hard-coded default config (a forward reference to whatever `defaultConfig` step 21 imports from `@/lib/schema/experiment` — the step-01 schema — or synthesizes inline from the PDF's 3:2 ratio, lattice 10×10, Moore neighborhood, and the two default referents), (c) calls `api.step(5)` to run five warm-up ticks so the initial render shows non-trivial state, (d) calls `api.getLatticeProjection(selectedWorld, projectionKind)` to fetch the first cell list, (e) sets `cells`, `currentTick`, and clears any loading flag, (f) returns a cleanup function that calls `terminate()` (which internally calls `api[Comlink.releaseProxy]()` then `worker.terminate()` per step 20). A **second** effect watches `selectedWorld` and `projectionKind` and re-fetches the projection whenever either changes — this is the "projection toggle takes effect within one tick" F7 acceptance clause. A **third** effect drives the play/pause loop: when `isRunning` is true, set up a `setInterval` (or a rAF-driven loop, either works; `setInterval(100ms)` is fine because the worker ticks are bounded by the `step()` call's round-trip not by the interval period) that calls `api.step(1)`, increments `currentTick`, then re-fetches the projection; the interval is cleared when `isRunning` flips to false or the component unmounts. The tooltip callback (`onHoverCell(cellIndex)`) is a `useCallback` that calls `api.getSnapshot()` to retrieve the full inventory for the hovered agent (lazily — only when the user actually hovers, not on every render) and updates the `hoveredAgent` state with a structured `{ id, class, position, inventory }` object; a null `cellIndex` clears it. The JSX structure is a top-level `<div>` with Tailwind classes for layout (`p-6 flex flex-col gap-4`), a header row containing the play/pause button, the world selector (two buttons that set `selectedWorld`), the `<ProjectionToggle projectionKind={...} onChange={...} />`, and a `<span>` showing `Tick: {currentTick}` and `World: {selectedWorld}` for the MCP verification script to read. Total size: ~200 lines with JSDoc and blank lines.

  1. Null-guard `state` (same pattern as the other worker methods — throw if `init()` has not been called).
  2. Select `world = worldId === 'world1' ? state.world1 : state.world2`.
  3. Switch on `kind`:
     - `'class'`: map `world.agents` to `{ agentId: a.id, class: a.class, position: a.position }` (no `topToken`, no `matchingRate`). Return the array.
     - `'dominant-token'`: for each agent, compute the top-weighted token for the **first referent** in the config. For each neighbor that exists, compute its top-weighted token for the same first referent (reusing the dominant-token computation). **Slice two — implement `getLatticeProjection` in the worker.** Add a new async arrow function inside the worker module, near the other method implementations. Body sketch:

```typescript
const getLatticeProjection = async (
  worldId: WorldId,
  kind: ProjectionKind,
): Promise<CellData[]> => {
  if (state === null) {
    throw new Error('simulation worker: getLatticeProjection() called before init().');
  }
  const world = worldId === 'world1' ? state.world1 : state.world2;
  const firstReferent = world.referents[0];

  const topTokenFor = (agent: Agent): string | undefined => {
    let best: { lex: string; w: number } | null = null;
    for (const langMap of agent.inventory.values()) {
      const lexMap = langMap.get(firstReferent);
      if (!lexMap) continue;
      for (const [lex, w] of lexMap.entries()) {
        if (best === null || w > best.w) best = { lex, w };
      }
    }
    return best?.lex;
  };

  if (kind === 'class') {
    return world.agents.map(a => ({
      agentId: a.id,
      class: a.class,
      position: a.position,
    }));
  }

  if (kind === 'dominant-token') {
    return world.agents.map(a => ({
      agentId: a.id,
      class: a.class,
      position: a.position,
      topToken: topTokenFor(a),
    }));
  }

  // kind === 'matching-rate'
  const byPosition = new Map<number, Agent>();
  for (const a of world.agents) byPosition.set(a.position, a);
  return world.agents.map(a => {
    const myTop = topTokenFor(a);
    let matches = 0;
    let total = 0;
    for (const npos of world.topology.neighbors(a.position, state!.rng)) {
      const neighbor = byPosition.get(npos);
      if (!neighbor) continue;
      total += 1;
      if (topTokenFor(neighbor) === myTop && myTop !== undefined) matches += 1;
    }
    return {
      agentId: a.id,
      class: a.class,
      position: a.position,
      matchingRate: total === 0 ? 0 : matches / total,
    };
  });
};
```

Then add `getLatticeProjection` to the `api` object: `const api: SimulationWorkerApi = { init, step, run, getMetrics, getSnapshot, reset, getLatticeProjection };`. The method re-uses step 10's `topology.neighbors()` iterator, which for lattices returns a small `number[]` and for well-mixed is a lazy generator — both work with the `for (const npos of ...)` loop. A defensive note on the `topology.neighbors()` `rng` argument: step 10 passes the RNG through for symmetry with variants that might need it at iteration time; the lattice and well-mixed implementations do not consume it for `neighbors()` enumeration, but the step-21 call passes `state.rng` anyway to satisfy the interface.

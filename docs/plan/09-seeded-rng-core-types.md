## 1. Goal

Lay the foundation of the **pure simulation core** in `lib/sim/`. This step delivers two modules and their test files: (a) a **deterministic seeded RNG** wrapping `pure-rand` with a narrow, simulation-shaped interface (`nextInt`, `nextFloat`, `pick`, `pickWeighted`, `shuffle`), and (b) the **core simulation types** ŌĆö branded `AgentId`, `Language`, `Referent`, `TokenLexeme`, a `Token` value type, the `AgentClass` union from `docs/spec.md` ┬¦3.1, the nested `Inventory` type from ┬¦3.2, and the `AgentState` / `InteractionRecord` records consumed by steps 10-18. Utility helpers on `Inventory` (`inventoryGet`, `inventorySet`, `inventoryIncrement`) are pure functions that return new `Map`s rather than mutating inputs, so that later code can rely on structural immutability when reasoning about snapshot/reset semantics. The RNG is the **single source of entropy** for the entire simulation ŌĆö every later step (10 placement, 11 vocabulary seeding, 12 language policies, 13 the tick loop, 14 preferential attachment, 17 summary classifiers) accepts an `RNG` argument and **never** calls `Math.random()`. This "no `Math.random`" rule is a hard invariant enforced by code review and by a lint-friendly grep test in step 18's smoke test. Without this step, nothing downstream in Phase 2 is reproducible, and the whole research instrument loses its primary epistemic guarantee: that a run with the same seed and config is bit-identical today, tomorrow, and on a collaborator's machine.

- `https://en.wikipedia.org/wiki/Xoroshiro128%2B` ŌĆö confirmed xoroshiro128+ is a 128-bit-state PRNG by David Blackman and Sebastiano Vigna, designed as a fast and statistically strong successor to xorshift. Known caveat: the lowest bits have mildly poor quality; standard practice is to use upper bits for integer work and to use the full 53-bit mantissa path for floats. `pure-rand`'s `uniformInt`/`uniformIntDistribution` does the right thing here ŌĆö it samples from the high-quality bits via rejection sampling under the hood, so the wrapper in `lib/sim/rng.ts` does not need to compensate. **Not cryptographically secure** ŌĆö this is fine because simulation RNGs must be reproducible, not unpredictable. Period is `2^128 - 1`, which is several orders of magnitude beyond anything a tick loop could consume even in pessimistic batch sweeps (N = 10^4 agents ├Ś 10^4 ticks ├Ś 10 random draws per interaction = 10^9 draws per run; the 128-bit period is vastly larger).

Paths not taken:

- **`seedrandom`** (`https://github.com/davidbau/seedrandom`) ŌĆö the library `docs/spec.md` ┬¦9 first named as the "leading candidate" alongside `pure-rand`. Rejected here because it is an older JS-first library that monkey-patches `Math.random` by default (opt-out), exposes a less structured API, and its ESM story is less clean under Turbopack. Pure-rand's explicit "pure" design (returning new generators rather than mutating) makes determinism failures loud: if you forget to thread the new generator through, your next call uses the same state. This is a useful safety net when writing a simulation where thousands of entropy draws happen per tick.
- **A hand-rolled Mulberry32 or splitmix64** ŌĆö viable and small (~20 lines). Rejected because (a) pure-rand gives us multiple generators, distributions, and an immutable API for free, (b) the tested generator implementations in pure-rand have survived statistical test suites (TestU01, PractRand) that we do not want to re-run, and (c) a five-line dependency is cheaper to maintain than a custom PRNG whose correctness the team would have to argue for in the research write-up.
- **`Math.random` with a seeding polyfill** ŌĆö **rejected**. `Math.random` is a **global** mutable singleton; there is no portable way to reset its state, and two modules that both call it interfere with each other in ways that are undetectable until a run is not reproducible. Worse, V8's `Math.random` implementation is explicitly allowed to change between Node versions. This step codifies the ban as a project invariant; see section 11 below.
- **`@types/seedrandom` + seedrandom** ŌĆö even if we chose seedrandom, we would pay the type-only dependency cost. `pure-rand` ships its own types, so this is moot.
- **Defining `AgentClass` as a `const enum`** ŌĆö rejected because `const enum` has known `isolatedModules` and `verbatimModuleSyntax` pitfalls under Turbopack's SWC transforms, and the Zod schema in step 01 already uses a literal union (`z.enum([...])`). A literal-union type alias in `lib/sim/types.ts` mirrors step 01's choice and avoids the double-declaration problem.

## 5. Files to create

- `lib/sim/rng.ts` ŌĆö the deterministic RNG wrapper. Exports: the `RNG` interface (with methods `nextInt(min: number, max: number): number`, `nextFloat(): number`, `pick<T>(items: readonly T[]): T`, `pickWeighted<T>(items: readonly T[], weights: readonly number[]): T`, `shuffle<T>(items: readonly T[]): T[]`), the `createRNG(seed: number): RNG` factory, and a small `RNGState` type alias that covers the serialized state of the underlying pure-rand generator for optional snapshot resumption. If exposing the state turns out to depend on internal pure-rand shapes that are not part of its public API, leave `state()` off the interface in v1 and note the gap in the step-20 worker plan (which is the first caller that would need resumption). No classes ŌĆö the `RNG` type is an object literal of method functions, and `createRNG` closes over a mutable `current` reference to the immutable pure-rand generator so each method call advances the closure's reference. This keeps the public API **stateful and imperative** (the four methods simply return values) while the underlying generator remains functionally pure ŌĆö consumers never touch the pure-rand type at all.
- `lib/sim/types.ts` ŌĆö the branded primitive and record types. Exports: `AgentId`, `Language`, `Referent`, `TokenLexeme` (all branded strings via the `string & { readonly __brand: unique symbol }` pattern wrapped in a tiny `Brand<Base, Tag>` utility), `Token` (= `{ readonly language: Language; readonly lexeme: TokenLexeme }`), `TokenKey` (alias for `string`, produced by the `tokenKey` helper), `Weight` (alias for `number`), `AgentClass` (literal union, re-exported from or locally mirrored to match `@/lib/schema/primitives`), `Inventory` (= `ReadonlyMap<Language, ReadonlyMap<Referent, ReadonlyMap<TokenLexeme, Weight>>>`), `AgentState` (= `{ readonly id: AgentId; readonly class: AgentClass; readonly position: number; readonly inventory: Inventory; readonly interactionMemory: readonly InteractionRecord[] }`), `InteractionRecord` (= `{ readonly tick: number; readonly partnerId: AgentId; readonly language: Language; readonly referent: Referent; readonly token: TokenLexeme; readonly success: boolean }`). The file also exports the pure utility helpers `makeAgentId(raw: string): AgentId`, `makeLanguage(raw: string): Language`, `makeReferent(raw: string): Referent`, `makeTokenLexeme(raw: string): TokenLexeme`, `makeToken(lang: Language, lex: TokenLexeme): Token`, `tokenKey(token: Token): TokenKey`, `emptyInventory(): Inventory`, `inventoryGet(inv: Inventory, lang: Language, ref: Referent, lex: TokenLexeme): Weight | undefined`, `inventorySet(inv: Inventory, lang: Language, ref: Referent, lex: TokenLexeme, weight: Weight): Inventory`, and `inventoryIncrement(inv: Inventory, lang: Language, ref: Referent, lex: TokenLexeme, delta: number, floor?: number): Inventory`. The `make*` functions are thin type-asserting wrappers (runtime no-op casts); their job is readability at call sites, not validation ŌĆö validation already happens at the Zod boundary in `lib/schema/`. `inventorySet` and `inventoryIncrement` **must not mutate** their input: they clone each `Map` along the path from the root down to the leaf using `new Map(existing)`, then replace the touched leaf. This is what makes snapshotting cheap (shallow structural sharing by key until a write path touches a branch) and reasoning about ticks safe (the step-13 tick loop can hold a reference to the pre-tick inventory and still have it unchanged after weight updates land).
- `lib/sim/index.ts` ŌĆö a small barrel that re-exports everything from `./rng` and `./types`. Optional but recommended: downstream files import from `@/lib/sim` and do not have to thread deep paths. This barrel is also the single place step 20's worker entrypoint imports from, which keeps the worker's top-of-file diff narrow.
- `lib/sim/rng.test.ts` ŌĆö Vitest suite for the RNG invariants, enumerated in section 9. Default `node` environment.
- `lib/sim/types.test.ts` ŌĆö Vitest suite for branded-type compile-time safety and inventory-helper immutability, enumerated in section 9. Default `node` environment.

## 7. Implementation approach

Start by pulling the dependency, then build the RNG, then build the types, then write the two test files. The ordering matters because the types module is pure TypeScript with no runtime dependency and can be written blind, but the RNG file wants to be written against the freshly installed `pure-rand` API to catch any mismatch between the research notes and the library's actual shape before the tests are authored.

First, **install `pure-rand`** via `npm install pure-rand@^8.4.0`. Confirm the resolved version is 8.x and the `node_modules/pure-rand/package.json` shows `"main"` and `"types"` fields pointing at `.cjs` and `.d.ts` entrypoints. Run `npm ls pure-rand` to sanity-check there is no accidental duplicate. If any peer-dependency warnings are emitted, record them in the step log but do not act on them ŌĆö pure-rand has no peer dependencies in 8.x.

Second, **write `lib/sim/rng.ts`**. Import `xoroshiro128plus` from `pure-rand/generator/xoroshiro128plus` (or from the top-level `pure-rand` barrel if the top-level export is the documented one at the resolved version ŌĆö the agent checks by reading `node_modules/pure-rand/lib/pure-rand.d.ts` and picks the shorter path that still resolves cleanly). Import `uniformInt` (or `unsafeUniformIntDistribution` / `uniformIntDistribution` ŌĆö again the agent verifies which names are exported in the installed version; prefer the higher-level `uniformInt(rng, min, max)` if available because it returns `[value, newGenerator]` in the functional style, falling back to `unsafeUniformIntDistribution` only if the friendlier wrapper is gone). The `createRNG(seed)` factory creates an initial immutable generator `let current = xoroshiro128plus(seed)` captured in the closure. Each public method advances `current` by reassigning it to the new generator returned by the underlying distribution call, then returns the value to the caller. `nextInt(min, max)` delegates directly to `uniformInt`. `nextFloat()` produces a double in `[0, 1)` by generating a 53-bit integer (draw two 32-bit ints and combine the high 26 and high 27 bits, then divide by `2^53`) ŌĆö this is the textbook "IEEE 754 double from a uniform integer" technique and is necessary because pure-rand's integer-only distributions do not directly provide a `[0, 1)` float. Alternatively, use pure-rand's `uniformIntDistribution(0, 2^53 - 1, rng)` and divide. Pick whichever the installed version exposes; the concrete choice is recorded in the module's doc comment so a future reader knows which formula is authoritative. `pick(items)` returns `items[nextInt(0, items.length - 1)]`. `pickWeighted(items, weights)` validates lengths match, sums the weights to `total`, draws a uniform float in `[0, total)` via `nextFloat() * total`, walks the weights prefix-summed until the bin containing the draw is found, and returns the corresponding item; this is the standard O(n) roulette-wheel sampling and is sufficient at simulation tick scales (agent population Ōēż 10^4, weight vectors per agent Ōēż tens of entries). `shuffle(items)` copies the input into a fresh array and applies the standard Fisher-Yates in-place on the copy using `nextInt`, then returns the copy; the input is guaranteed untouched because the copy is taken first. Defensive-programming checks (`throw` on `min > max`, on empty arrays in `pick`, on mismatched `items.length !== weights.length` in `pickWeighted`, on negative weights, on all-zero weights) are included and documented in the doc comments. The error messages include the method name and the offending argument summary so failures mid-tick are debuggable.

Third, **write `lib/sim/types.ts`**. Define a tiny generic `Brand<Base, Tag extends string>` type (`type Brand<Base, Tag extends string> = Base & { readonly __brand: Tag }`) and derive `AgentId = Brand<string, 'AgentId'>`, `Language = Brand<string, 'Language'>`, `Referent = Brand<string, 'Referent'>`, `TokenLexeme = Brand<string, 'TokenLexeme'>`. The brand key is `__brand`; using the same key with different tag literals is enough to prevent cross-assignment under TypeScript's structural subtyping because the tag is a discriminating string literal. Define `Token = { readonly language: Language; readonly lexeme: TokenLexeme }`, `TokenKey = string`, `Weight = number`, `AgentClass = 'W1-Mono' | 'W1-Bi' | 'W2-Native' | 'W2-Immigrant'` (and add a `re-export` line `export type { AgentClass as AgentClassFromSchema } from '@/lib/schema/primitives'` with a type-level `satisfies`-style check that the two unions are structurally identical; if the import path differs because step 01's module layout changes, the agent adjusts to whatever step 01 actually shipped rather than invoking module names that may not exist ŌĆö the existence check happens by reading `lib/schema/` at the start of this step), and `Inventory = ReadonlyMap<Language, ReadonlyMap<Referent, ReadonlyMap<TokenLexeme, Weight>>>`. `InteractionRecord` and `AgentState` records follow the shapes in section 5. Implement the helpers: `makeAgentId(raw) { return raw as AgentId }` and the other three `make*` wrappers (all trivial casts); `makeToken(lang, lex)` returns a frozen object literal (optional `Object.freeze` for insurance under strict mode); `tokenKey(t)` returns `${t.language}:${t.lexeme}`; `emptyInventory()` returns `new Map()` typed as `Inventory`; `inventoryGet` walks the three levels with optional-chaining and returns `undefined` if any level is missing. `inventorySet` copies each level along the write path ŌĆö `const l1 = new Map(inv); const l2 = new Map(l1.get(lang) ?? new Map()); const l3 = new Map(l2.get(ref) ?? new Map()); l3.set(lex, weight); l2.set(ref, l3); l1.set(lang, l2); return l1` ŌĆö and returns the top copy. `inventoryIncrement(inv, lang, ref, lex, delta, floor = 0)` reads the current weight with `inventoryGet` (defaulting to `0` if absent), computes `Math.max(floor, current + delta)`, and delegates to `inventorySet`. The floor parameter is what lets step 13 use a common helper for both ╬öŌü║ (add positive, floor at 0 is inactive) and ╬öŌü╗ (add negative, floor at 0 clamps to the minimum), matching the weight-update description in ┬¦3.3.

Fourth, **write `lib/sim/index.ts`**: a two-line barrel re-exporting everything from `./rng` and `./types`. No side effects.

Fifth, **write `lib/sim/rng.test.ts`**. Import from `@/lib/sim/rng` (the path alias from step 00). The tests are enumerated in section 9; each uses a pinned seed and asserts exact outputs. ## 8. Library choices

- **`pure-rand`** pinned to **`^8.4.0`** (latest stable 8.x at plan-write time, 2026-03-27 publish date, verified via `npm view pure-rand version`). Justification: (a) **TypeScript-first**, ships `.d.ts` files, no `@types/*` package needed; (b) **zero runtime dependencies** ŌĆö the install footprint is a single package; (c) **four verified generators** including `xoroshiro128plus` with the author's explicit recommendation ("Our recommendation is `xoroshiro128plus`"); (d) **functional/immutable API** ŌĆö each distribution call returns a new generator, which forces the wrapper to be explicit about state advancement and surfaces forgotten-state bugs loudly at code-review time; (e) **MIT licensed**, compatible with the project's future licensing choices; (f) **proven in the field**: pure-rand is the RNG used by `fast-check` (the de-facto property-based testing library in the TypeScript ecosystem), so it has millions of runs of battle testing against its own determinism guarantees. No alternative library is installed.
- **Generator choice: `xoroshiro128plus`.** 128-bit state, period `2^128 ŌłÆ 1`, fast on 64-bit CPUs, pure-rand's own recommendation. The known weakness in the low bits is irrelevant here because simulation draws go through `uniformInt` / `nextFloat` which use upper bits. See ┬¦4 research notes for the Wikipedia citation on the generator's properties.
- **No additional testing libraries.** Vitest is already installed per step 00. No `fast-check` in this step ŌĆö property-based tests are nice-to-have but would double the scope.

## 9. Unit tests

**`lib/sim/rng.test.ts`:**

1. **Determinism across construction.** Create two `createRNG(42)` instances, call `nextInt(0, 99)` 100 times on each, and assert the two sequences are `.deepEqual`. This confirms the factory is a **pure function of seed**.
2. **Deterministic locked sequence.** With `createRNG(42)`, capture the first 10 `nextInt(0, 99)` values into an array and assert it equals a fixture array hard-coded in the test file. The fixture is the output of running the same sequence once during development and pasting it in. If a future refactor swaps generators or changes `nextInt`'s internal bit-consumption pattern, this test fails loudly with a readable diff ŌĆö serving as a change detector beyond the simple same-seed-same-sequence property.
3. **Different seeds produce different sequences.** `createRNG(0)` and `createRNG(1)` both generate 100 `nextInt(0, 999999)` values; assert the two arrays are **not** `.deepEqual`. (Collision is astronomically unlikely with two distinct 32-bit seeds on xoroshiro128+, but if it ever fires it is a strong signal of a bug.)
4. **`nextFloat` is in `[0, 1)`.** Draw 10_000 `nextFloat()` values and assert every sample satisfies `0 <= x && x < 1`.
5. **`nextFloat` determinism.** `createRNG(7)` followed by 50 `nextFloat()` draws produces the same array twice across two constructions.
6. **`pick` returns only elements from the input.** Over 1000 `pick(["a", "b", "c", "d"])` calls with `createRNG(123)`, assert every returned value is one of the four inputs and that every input appears at least once (uniform-ish coverage sanity check; the 1000-draw threshold gives the probability of missing any element < 10^{-100}).
7. **`pickWeighted` biases correctly.** Over 10_000 calls of `pickWeighted(["a", "b"], [1, 99])` with `createRNG(5)`, assert that `"b"` is returned **more than 9500 times and fewer than 9900 times** (tight Chernoff-bound-justified window around the expected 9900 minus std-dev slack). This guarantees the weights actually matter and that the roulette-wheel walk has not collapsed to uniform sampling.

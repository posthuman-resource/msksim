## 1. Goal

Implement the **beating heart of the simulation**: the per-tick speaker-chooses-partner / picks-language / picks-referent / utters-token / hearer-guesses / weight-update loop from `docs/spec.md` §3.3. Concretely, this step delivers a single public entrypoint `tick(state: SimulationState, rng: RNG): TickResult` at `lib/sim/engine.ts`, plus a small companion `updateWeight(...)` helper at `lib/sim/engine/weight-update.ts` that every engine call routes inventory mutations through. Nothing else in `lib/sim/` was able to actually advance simulated time before this step; after this step, steps 15-17 (scalar and graph metrics) can consume the emitted `InteractionEvent[]`, step 14 (preferential attachment) can plug a biased strategy into the partner-selection seam, and step 18 (simulation smoke test) can demonstrate that a 50-agent-per-world simulation runs 200 ticks and reports plausible success rates. This step implements feature **F3** from `docs/spec.md` §4.1 and supports **RQ1** (assimilation/segregation thresholds), **RQ3** (token-weight dynamics), and **RQ4** (emergent social cohesion) per the spec's research-goal matrix in §6. Determinism is the single most important invariant: one run with seed `X` must produce bit-identical `TickResult[]` across invocations, across machines, and across Node patch versions — this property is what makes `msksim` a research instrument instead of a toy, and it is the promise every later step is built on top of. The tick loop is also the first hot path in the project (10^4 ticks × 10^3 agents = 10^7 interactions per run in the pessimistic case), so inventory lookups must be O(1) and per-agent bookkeeping must avoid quadratic scans — but micro-optimizations beyond that are deliberately scoped out of v1 to keep this step focused and reviewable.

## 3. Spec references

- `docs/spec.md` **§3.3 Interaction rules (from the PDF, formalized)** — the authoritative source for the 7-step tick algorithm this file implements verbatim. Step 13 is "how these words become code" for §3.3 in the same sense step 12 was for §3.3 step 2. The relevant sub-steps are:
  1. **Partner selection.** "On a lattice: a neighbor in the configured neighborhood. Well-mixed: any agent in the same world (uniform random). Network: a neighbor in the interaction graph. Once preferential attachment (F6) is engaged, the choice is biased toward partners with similar token-weight profiles." Step 13 delegates structural reachability to `topology.pickNeighbor` and delegates bias to the `selectPartner(strategy)` seam that step 14 will plug into.
  2. **Language selection.** Delegated to `LanguagePolicy` from step 12. Step 13 does not interpret `Language` strings.
  3. **Referent selection.** "The speaker picks a referent to talk about (uniform random among referents the speaker knows in the chosen language)." Step 13 uses `rng.pick` over the keys of `speaker.inventory.get(chosenLanguage)`.
  4. **Token utterance.** "The speaker picks a token for that referent from its inventory in the chosen language, weighted by current token weights." Step 13 uses `rng.pickWeighted`.
  5. **Guessing.** "If the hearer has the token associated with the same referent, the interaction is a success. Otherwise it is a failure." Step 13 implements this as a direct `hearer.inventory.get(lang)?.get(ref)?.get(lex)` lookup, with a same-referent check.
  6. **Weight update.** "On success: both speaker and hearer increase the weight of that (referent, token) pair by a configurable step Δ⁺. On failure: ... no weight decrement in the minimal Naming Game; v1 follows this convention, with an optional penalty Δ⁻ exposed as a parameter. The speaker then selects a new partner in the same tick (bounded by a retry limit to avoid runaway tick cost)." Step 13 implements both the Δ⁺ success update and the optional Δ⁻ failure update, and enforces the retry limit as a per-speaker counter reset at the top of each activation.
  7. **Preferential attachment update (F6).** The interaction memory is updated. Step 13 writes into the bounded FIFO queue (`config.interactionMemorySize`) but does not yet bias partner selection on it — that is step 14's job. Step 13 **creates the memory data**; step 14 **consumes** it.
- `docs/spec.md` **§3.4 Initial conditions** — step 13 consumes `config.deltaPositive`, `config.deltaNegative`, `config.retryLimit`, `config.weightUpdateMode`, `config.interactionMemorySize`, and `config.schedulerMode`. The defaults (additive Δ⁺ = 1.0, Δ⁻ = 0, retryLimit = 1, weightUpdateMode = 'additive', interactionMemorySize = 50, schedulerMode = 'random') come from step 01's Zod schema; step 13 does not re-validate them and never supplies defaults of its own — the schema layer already did.
- `docs/spec.md` **§11 Open Question 4 (noise / mishearing)** — the spec recommends "no by default in v1, but expose a hook in F3 so it can be added as an ablation parameter later without refactoring." Step 13 does not implement noise; it documents the seam as "the place to apply a per-interaction mishearing probability would be between the hearer's inventory lookup and the success/failure branch." No hook is actually wired for noise in v1, but the comment marks the location so a future noise step has a clear insertion point.
- `docs/spec.md` **§11 Open Question 5 (additive vs multiplicative vs normalized weight update)** — "Recommendation: additive as the default, with optional L1-normalization per (agent, referent) as a configurable alternative. Both are exposed in F3." Step 13 supports exactly two modes: `'additive'` (the default) and `'l1-normalized'` (after incrementing, renormalize so that `Σ weights for (agent, language, referent) = 1.0`). Multiplicative is explicitly not in v1 per the open question's phrasing.
- `docs/spec.md` **§11 Open Question 7 (preferential attachment)** — "Recommendation: use a softmax over cosine similarity between top-weighted token vectors." Step 13 does **not** implement the softmax; it leaves the `selectPartner(strategy)` seam for step 14 to fill. Step 13's `'uniform'` strategy is the only strategy in v1.
- `docs/spec.md` **F3 (Interaction engine)** — the acceptance criterion is: "Unit-testable pure functions for each rule; deterministic given a seed; configurable Δ⁺, Δ⁻, retry limit; scheduler can be sequential, random, or priority-based." Step 13 is the concrete realization of this feature. "Pure functions" here means "deterministic given (state, rng)," i.e. side-effect-free except for the RNG's state advancement and the inventory returns; see section 7 for the immutable-return discipline.
- `docs/spec.md` **§1.2 RQ1, RQ3, RQ4** — the research questions F3 supports. Step 13's `InteractionEvent[]` output is the raw data every metric in §7.1 derives from: communication success rate (RQ1, RQ4), mean token weight and variance (RQ3), matching rate (RQ1, RQ4). Without step 13, none of these metrics have anything to read.

**External references (WebFetched):**

5. **Baronchelli (2016), "A gentle introduction to the minimal Naming Game"** — <https://arxiv.org/abs/1701.07419> (WebFetched). The abstract confirms the model describes "how networks of locally interacting individuals can spontaneously self-organize to produce global coordination" and signals that "slight modifications of the basic interaction rules give origin to a richer phenomenology in which more conventions can co-exist indefinitely." This is the canonical reference for the **minimal Naming Game rules** step 13 implements: a randomly chosen speaker picks a hearer, picks a token from its inventory for a shared referent, the hearer accepts or rejects, and on success both parties reinforce the matching token while on failure the minimal variant simply adds the token to the hearer without decrementing anything. `docs/spec.md` §3.3 is a condensation of exactly those rules (with the bilingual extension layered on top), and `docs/spec.md` §12.1 cites this paper as "the source for the 'minimal Naming Game' rules this spec follows." Step 13's Δ⁻ defaulting to 0 (per the step brief) is the faithful encoding of the minimal variant. The `config.retryLimit` defaulting to 1 (speaker attempts exactly one partner per activation before giving up) mirrors the per-tick budget the original paper's simulations use to keep the per-tick cost bounded on large lattices.
6. **Dall'Asta, Baronchelli, Barrat, Loreto (2008), "In-depth analysis of the Naming Game dynamics: the homogeneous mixing case"** — <https://arxiv.org/abs/0803.0398> (WebFetched). The specific claim relevant to step 13 is the **mean-field prediction that success rate climbs monotonically over time** in a well-mixed population with a small initial vocabulary — this is what the "Success rate climbs over time" unit test in section 9 asserts at the level of a small (50-agent) pure-L1 configuration. The paper also studies **normalized vs additive updates** analytically; the spec's §11 Open Question 5 recommendation for "additive as default, L1-normalization as configurable alternative" reads this paper's phase-diagram treatment as justification. Step 13 ships exactly those two modes and routes them through a single `updateWeight(...)` helper so a third mode (e.g. multiplicative, v2+) could be added by extending the discriminated union rather than by branching the tick loop.

**Paths not taken:**

7. **Mutable in-place inventory updates** (the "everything is a plain object you write into" design). Tempting because it is the simplest possible encoding and avoids all the Map-copy allocations the immutable approach pays at every weight update. **Rejected** for three independent reasons. (a) Snapshot/reset semantics: step 16's graph metrics and step 17's summary classifiers want to hold a reference to the pre-tick inventory and diff it against the post-tick inventory; with mutable updates, the pre-tick reference is silently overwritten and the diff becomes impossible. (b) Testing: the determinism test in section 9 compares `JSON.stringify(state)` before and after a roundtrip, which requires the "before" snapshot to still exist after the tick; a mutable design breaks this comparison without deep-cloning at every tick boundary, which is strictly more expensive than the shallow-copy-along-write-path approach step 09's `inventorySet` already provides. (c) The step-09 `inventoryIncrement` helper is already immutable by design, and step 13 is meant to compose with it, not to bypass it. The chosen design uses `updateWeight(...)` as a thin wrapper over `inventoryIncrement` that also handles the normalization branch, and returns a new `Inventory` on every call; the tick loop replaces `agent.inventory` with the returned value before proceeding. Shallow structural sharing keeps the allocation cost bounded to the one write path per weight update (typically 3 `new Map(...)` calls), which at 10^7 interactions per run is ~3 × 10^7 Map constructions — well within the single-digit-GB heap budget of a browser worker tab.
8. **Eagerly recomputing `tokenKey` caches per tick**. Considered: the `(language, referent, lexeme)` triples could be hashed once at bootstrap time and the engine could carry around integer keys instead of triple-nested Map lookups. **Rejected** because step 09 already committed to the nested-Map shape and because profiling has not yet demonstrated that triple-Map lookup is the hot spot. Documented as "out of scope for v1" in section 8; a future optimization step can revisit once step 18's smoke test identifies a real bottleneck. The alternative would also break `JSON.stringify` snapshots (integer-keyed caches are less human-readable in test failures).
9. **A class-based `Engine` with methods `.step()`, `.run(n)`, `.getState()`**. Ergonomic for stateful callers, but **rejected** because it hides the immutability contract behind instance mutation. The pure-function design `tick(state, rng): TickResult` is explicit about what changes and what doesn't: the only thing the function can mutate is the RNG state (by design, per step 09's RNG wrapper), and the only thing it returns is a new state plus events. Callers that want a stateful facade can wrap `tick` in two lines; the engine module itself stays pure.
10. **Embedding preferential attachment directly in step 13** rather than leaving a seam. **Rejected** because (a) step 14 is already scoped for preferential attachment and has its own research-quality-gate budget for the softmax-over-cosine-similarity work, (b) mixing the two concerns in one commit would balloon this step's test surface, and (c) the partner-selection seam is exactly the "pluggable" property F3 requires in `docs/spec.md` §4.1. The `selectPartner(speaker, world, rng, strategy)` function signature is the contract step 14 will honor.

Total research items: **4 local Next doc citations** + **2 external URLs WebFetched** (Baronchelli 2016, Dall'Asta 2008) + **4 paths not taken** = **10 links**, comfortably above the ≥ 5 floor mandated by the plan-file contract and the "Research quality gates" in this step's prompt.

## 7. Implementation approach

The work is ordered so that the pure helper lands first (and can be tested in isolation), then the engine module is built on top of it, and finally the tests exercise the composed behavior. This ordering matches the dependency graph: `weight-update.ts` depends on nothing beyond step 09's `inventoryIncrement` and `inventorySet`, while `engine.ts` depends on everything up to and including step 12. Writing the helper first means the engine can be written against a green test suite for the most failure-prone numeric operation, which is the only place in step 13 where a subtle off-by-epsilon bug could silently corrupt every later metric.

First, **write `lib/sim/engine/weight-update.ts`**. The function signature is `updateWeight(inventory: Inventory, language: Language, referent: Referent, lexeme: TokenLexeme, delta: number, mode: 'additive' | 'l1-normalized'): Inventory`. The function is implemented as a switch on `mode`. In the `'additive'` branch, call step 09's `inventoryIncrement(inventory, language, referent, lexeme, delta, 0)` and return the result directly — the floor argument of `0` matches the spec's "floored at 0" rule for Δ⁻. In the `'l1-normalized'` branch, first call `inventoryIncrement` exactly as in the additive branch to get an intermediate inventory, then extract the `(language, referent)` sub-map via `intermediate.get(language)?.get(referent)`, compute the sum of its values, and if the sum is strictly positive, walk each `(lex, weight)` pair in the sub-map and call `inventorySet(currentInventory, language, referent, lex, weight / sum)`, threading the returned `currentInventory` through the loop so each `inventorySet` composes on the previous one's output. Return the final inventory. If the sub-map is missing (the very first increment on an empty `(language, referent)` cell is the only way this can happen), the intermediate inventory is already correct by construction and the function returns it unchanged. If the sum is zero (only possible if every weight was already zero and `delta <= 0`), leave the entries at zero and return the intermediate inventory untouched — this is the degenerate case where normalization is undefined, and the convention is "do nothing" rather than "divide by zero." Document both edge cases in a JSDoc comment on the function so a future reader does not have to re-derive them.

Second, **sketch the `SimulationState`, `InteractionEvent`, `TickResult`, `SchedulerMode`, and `PartnerStrategy` types** at the top of `lib/sim/engine.ts`. These are type declarations only, no runtime code, and they establish the public surface before the implementation body is written. The field order in `InteractionEvent` follows the step brief verbatim: `{ tick, worldId, speakerId, hearerId, language, referent, token, success }`. The `worldId` field is added beyond the step brief because metrics consumers (step 15-17) need to partition events by world, and deriving the world from an `AgentId` alone requires a lookup the metrics code shouldn't pay. `TickResult` carries `state` (the updated `SimulationState`), `interactions` (the per-tick `InteractionEvent[]`), and `tickNumber` (a convenience duplicate of `state.tickNumber` that avoids one level of indirection in test assertions). The `SimulationState` type explicitly duplicates `tickNumber` at the top level rather than storing it inside `config` or deriving it from `interactions.length`, because the tick counter is a mutable state variable and `config` is immutable.

Third, **write `selectPartner`**. The function is `selectPartner(speaker: AgentState, world: World, rng: RNG, strategy: PartnerStrategy): AgentState | null`. The body is a `switch (strategy)` with a single `case 'uniform':` arm that calls `world.topology.pickNeighbor(speaker.position, rng)`, checks for `null` (isolated node) and returns `null` in that case, then resolves the non-null position index via `findAgentByPosition(world, positionIndex)`. If `findAgentByPosition` returns `undefined` (which should never happen given step 11's placement guarantees, but the types say it's possible), throw a descriptive error: `new Error(\`selectPartner: topology returned position ${positionIndex} but no agent lives there (speaker=${speaker.id})\`)` — this is defensive, and the test in section 9 does not exercise it, but the error message will immediately point a future debugger at the bootstrap / topology contract violation rather than at a generic `undefined is not an object`. After the switch, include an exhaustiveness check `const _exhaustive: never = strategy; throw new Error(\`Unknown strategy: ${_exhaustive}\`);` so that step 14's addition of `'preferential'` forces a compile error here until step 14 implements it. Add a comment at the switch: `// step 14 inserts 'preferential' here; see docs/plan/14-preferential-attachment.md for the biased strategy`.

Fourth, **write the scheduler sub-helper**. The scheduler's job is: given a world, return the list of speaker agents for the current tick in the order they should be activated. Extract this into a private `getActivationOrder(world: World, rng: RNG, mode: SchedulerMode): AgentState[]` helper (inside `engine.ts`, not exported). The three branches: `'sequential'` returns `world.agents.slice()` (a shallow copy in agent-id order — step 11 guarantees ids sort lexicographically, and `world.agents` is already constructed in that order at bootstrap time); `'random'` returns `rng.shuffle(world.agents)` (shuffle returns a fresh array, does not mutate the input); `'priority'` returns `rng.shuffle(world.agents)` with a comment `// priority mode is a placeholder for future (e.g. activation-rate weighting); default to random for v1 per step brief`. The exhaustiveness check at the end guards against config drift if a fourth mode is added later. Note that the sequential branch's shallow copy is important: later code in the tick body may mutate the returned array (e.g. to mark agents as already activated in this tick), and mutating `world.agents` directly would corrupt the caller's state.

Fifth, **write the tick body**. The `tick(state, rng)` function begins by destructuring `state` into `{ world1, world2, tickNumber, config }`. It then constructs a single `LanguagePolicy` via `createPolicy(config.policy)` — this happens once per tick, not once per interaction, because the policy factory is idempotent given the same config and the per-tick cost is a single map construction. It initializes `const events: InteractionEvent[] = []` as the collector for the per-tick `InteractionEvent` array. Then it processes the two worlds in order (world1 first, then world2 — this ordering is part of the determinism contract, mirroring step 11's draw-order discipline). For each world, it calls `getActivationOrder(world, rng, config.schedulerMode)` to get the speaker list, then iterates that list. For each speaker, it initializes `let retries = 0` and enters a `while (retries <= config.retryLimit)` loop — note the `<=` so a `retryLimit` of 1 allows one retry (total of two attempts). Inside the loop, it performs the six sub-steps below in order. Critically, the loop **terminates on success**, on partner `null`, on referent `null`, on token `null`, or when `retries > config.retryLimit`; it never loops infinitely. The loop termination is enforced by a `break` after every branch that resolves an interaction (success, failure after which retries increment, or unrecoverable unreachable partner), and a top-of-loop bounds check that exits if the retry budget is exhausted. A unit test in section 9 specifically exercises the retry-exhaustion path to catch any regression.

The six sub-steps inside the activation loop, in order:

(a) **Partner selection via the seam.** Call `const hearer = selectPartner(speaker, world, rng, 'uniform')`. If `hearer === null`, break out of the while loop (the speaker has no one to talk to; move on to the next speaker — this is not a "retry" case because a null from `pickNeighbor` means the topology is structurally disconnected at this position, not that the interaction failed). No `InteractionEvent` is emitted in this case; it simply does not count as an interaction.

(b) **Language selection via step 12's policy.** Call `const language = policy({ speaker, hearer, rng })`. The policy is deterministic per its inputs (step 12's contract), so this advances the RNG by however many draws the policy's biased coin flip consumes.

(c) **Referent selection.** Look up `speaker.inventory.get(language)` — if it is `undefined` or has zero entries, the speaker has no referents in the chosen language and the interaction cannot proceed. This is the "skip the interaction" branch the step brief calls out: break out of the while loop without emitting an event and without incrementing `retries`. (Rationale: the language policy committed the speaker to a language the speaker does not actually know words in; this is a configuration bug at the boundary rather than a runtime failure, and retrying would just re-invoke the same policy on the same pair with the same RNG-advanced state, likely producing the same result. A test in section 9 confirms the branch does not crash, and a comment documents that a future step can relax this into a policy-aware pre-filter.) Otherwise, call `const referent = rng.pick(Array.from(referentMap.keys()))` to choose a referent uniformly at random. Note: `Array.from(Map.keys())` materializes the keys into an array each call. At scale this is paid per interaction, but with a typical referent count of 2-5 the allocation is negligible; a documented comment notes this as a micro-optimization candidate for step 14's profiling.

(d) **Token utterance.** Look up `referentMap.get(referent)` to get the `Map<TokenLexeme, Weight>` for this referent. Guard against `undefined` (degenerate — the referent key exists at the language level but has no sub-map; shouldn't happen given step 11's bootstrap, but the types permit it) by breaking out of the loop without emitting an event. Convert the sub-map into parallel `lexemes: TokenLexeme[]` and `weights: number[]` arrays. Guard against `total === 0` (all weights are zero — can only happen in the `l1-normalized` mode under degenerate conditions; in additive mode the bootstrap seeds positive weights). If the total is zero, break out of the loop as a "no utterance possible" case. Otherwise, call `const token = rng.pickWeighted(lexemes, weights)` to get the chosen lexeme.

(e) **Hearer guess.** Look up `hearer.inventory.get(language)?.get(referent)?.get(token)` — if the result is a positive weight, the hearer has this `(language, lex)` mapped to the same referent: it is a **success**. If the result is `undefined`, check whether the hearer has this token under a **different** referent in the same language: walk `hearer.inventory.get(language)` and, for each `(otherReferent, tokenMap)` pair, check whether `tokenMap.get(token)` is defined. If yes for any other referent, the interaction is a **failure** (same-lexeme-different-referent, which is the canonical Naming Game "misunderstanding" case). If no (the lexeme is unknown to the hearer in this language entirely), that is also a **failure**. Emit an `InteractionEvent` with the appropriate `success: boolean`. Note that step 13 does **not** auto-add the unknown lexeme to the hearer's inventory — the minimal Naming Game variant described in `docs/spec.md` §3.3 has the hearer reinforce only on success; the "hearer adds on failure" variant is a separate model and is not in v1. A comment documents this faithfully.

(f) **Weight update.** If the interaction succeeded, call `updateWeight(speaker.inventory, language, referent, token, config.deltaPositive, config.weightUpdateMode)` and assign the returned inventory back to `speaker.inventory`; then do the same for the hearer with `config.deltaPositive`. If the interaction failed and `config.deltaNegative > 0`, call `updateWeight(speaker.inventory, language, referent, token, -config.deltaNegative, config.weightUpdateMode)` (note the sign flip — `deltaNegative` is stored as a positive number in the config and is negated here, floored at zero by `inventoryIncrement`'s internal floor). **Per the spec's minimal Naming Game default, `deltaNegative` is 0 out of the box, so the failure branch is a no-op in the default configuration.** Both speaker and hearer updates run against the same mode (additive or normalized). After the weight update, push an `InteractionRecord` onto each agent's `interactionMemory` (the bounded FIFO): if the current memory length equals `config.interactionMemorySize`, drop the oldest entry before pushing the new one. This is implemented as `agent.interactionMemory = [...agent.interactionMemory.slice(-(config.interactionMemorySize - 1)), newRecord]` — a fresh array on every push, preserving the immutability discipline. Step 14 will consume this memory to compute the preferential-attachment bias.

After sub-step (f), if the interaction succeeded, break out of the while loop — the speaker has had their activation and moves on to the next speaker. If the interaction failed, increment `retries` and continue the while loop (the speaker gets another chance to find a partner, up to `retryLimit` total attempts). If the top-of-loop bounds check fires (`retries > retryLimit`), break without error — retry exhaustion is an expected outcome and does not signal a bug. A test in section 9 explicitly exercises the retry-exhaustion path.

**Immutability discipline across the tick loop.** Because `inventoryIncrement` and `inventorySet` return new inventories (and `updateWeight` composes on top of them), the tick body must thread the returned `agent.inventory` through every subsequent read in the same tick. Concretely: when the weight update runs, it reassigns `speaker.inventory = updateWeight(...)` and `hearer.inventory = updateWeight(...)`. This is a mutation on the `AgentState` record itself (the `inventory` field is reassigned) but the `Inventory` value inside it is immutable. Two questions arise: (1) should `AgentState.inventory` be `readonly`? The answer is "no, not in v1" — step 09 declared it as a plain (non-readonly) field precisely so the tick loop can reassign it. (2) should the tick return a deep-cloned `SimulationState` rather than mutating agents in place? The answer is "the tick mutates `AgentState` fields in place (inventory, interactionMemory) but returns the same `SimulationState` object reference with `tickNumber + 1`." This is the **mutate-in-place-on-AgentState, immutable-within-Inventory** hybrid rule, documented at the top of `engine.ts`. It is the simplest design that satisfies the determinism contract (same seed → same result) and the step-20 worker contract (the worker owns the state object and mutates it in place; the main thread never holds a reference to it during a tick). An alternative fully-immutable design that produces a new `SimulationState` on every tick was considered and rejected in section 4 path-not-taken 7: the allocation cost for 10^4 ticks × 10^3 agents would dominate, and the benefit (snapshot recovery) is already satisfied by the step-16 snapshot tables that copy state explicitly at sampling intervals.

Sixth, **advance the tick counter** at the very end of the tick: `state.tickNumber = tickNumber + 1`. Return `{ state, interactions: events, tickNumber: state.tickNumber }`. This is the point at which the tick transitions from "tick N's effects" to "tick N+1 ready to run."

Seventh, **update `lib/sim/index.ts`** to re-export the new public surface (types and functions enumerated in section 5). Follow the alphabetical ordering convention steps 09-12 established.

Eighth, **write the test files** per section 9. Every test pins an explicit seed, constructs a minimal `ExperimentConfig` via Zod, calls `bootstrapExperiment(config, seed)`, then invokes `tick` with the resulting state. No shared mutable fixtures across tests. Every test file runs in the default Vitest `node` environment.

Ninth, **verify the acceptance criteria**. `npm test -- lib/sim/engine` passes. `npm run typecheck` passes. `npm run lint` passes. The small node-script smoke check from section 10 runs 200 ticks of a 50-agent-per-world simulation and prints a plausible success rate (e.g. > 0.7 for a pure-L1 configuration).

Tenth, **commit** with the exact subject line from section 12.

## 9. Unit tests

Two test files. Every test pins an explicit seed, constructs its own fixtures inline, and runs in the default Vitest `node` environment.

**`lib/sim/engine/weight-update.test.ts`:**

1. **Additive mode adds delta on existing weights.** Build a minimal inventory with `{L1: {red-ref: {'red': 1.0}}}`. Call `updateWeight(inv, L1, red-ref, 'red', 2.0, 'additive')`. Assert the returned inventory has weight `3.0` at that cell and that the input inventory is unchanged (immutable-return assertion). Assert `inv !== result` (reference inequality at the top level, since the write path returns a new outer Map).
2. **Additive mode floors at zero.** Starting from `{L1: {red-ref: {'red': 1.0}}}`, call `updateWeight(inv, L1, red-ref, 'red', -5.0, 'additive')`. Assert the returned weight is `0` (not `-4`). This is the `deltaNegative > current weight` case and the floor is load-bearing for the spec's "floored at 0" rule.
3. **Additive mode creates missing entries.** Starting from `emptyInventory()`, call `updateWeight(inv, L1, red-ref, 'red', 1.0, 'additive')`. Assert the returned inventory has weight `1.0` at that cell.
4. **L1-normalized mode renormalizes after increment.** Build `{L1: {red-ref: {'red': 1.0, 'rouge': 1.0}}}`. Call `updateWeight(inv, L1, red-ref, 'red', 2.0, 'l1-normalized')`. After the increment, the raw weights are `{red: 3, rouge: 1}` with total `4`; after normalization they become `{red: 0.75, rouge: 0.25}`. Assert the returned inventory has exactly those values. Assert further that `Σ weights per (L1, red-ref) ≈ 1.0 ± 1e-9` (the tolerance is for IEEE 754 floating-point arithmetic; the assertion uses `Math.abs(sum - 1) < 1e-9`).
5. **L1-normalized mode with a single-entry sub-map.** Build `{L1: {red-ref: {'red': 0.5}}}`. Call `updateWeight(inv, L1, red-ref, 'red', 1.0, 'l1-normalized')`. The raw post-increment weight is `1.5`, total `1.5`, normalized to `1.0`. Assert the returned weight is `1.0` (the degenerate single-entry case normalizes to 1).
6. **L1-normalized mode preserves other (language, referent) sub-maps.** Build `{L1: {red-ref: {'red': 1.0}, yellow-ref: {'yellow': 1.0}}}`. Call `updateWeight(inv, L1, red-ref, 'red', 1.0, 'l1-normalized')`. Assert the `red-ref` sub-map is renormalized (single entry becomes `1.0`, trivially) and the `yellow-ref` sub-map is **untouched** (`{yellow: 1.0}`). This is the cross-cell isolation property — normalization must not leak across referents.
7. **L1-normalized mode handles zero-total edge case.** Build `{L1: {red-ref: {'red': 0.0}}}` (a degenerate case). Call `updateWeight(inv, L1, red-ref, 'red', 0.0, 'l1-normalized')`. The total is `0`; the function must not divide by zero. Assert it returns an inventory with the `red` weight still `0` and does not throw.
8. **`updateWeight` is immutable.** For any combination of mode and delta, assert that the input inventory's reference identity is unchanged and that `JSON.stringify(inputInventory)` before and after the call is identical.

**`lib/sim/engine.test.ts`:**

9. **Determinism: same seed, same config, 100 ticks → identical final states.** Construct two `SimulationState` objects via `bootstrapExperiment(config, 42)`, run 100 ticks on each (constructing a fresh `RNG` from seed 42 for each run), and assert the final `world1.agents` arrays are deeply equal (via `JSON.stringify` comparison, which handles nested Map ordering by extracting the serialized shape). Mirror for `world2`. This is the single most important assertion in step 13. A failure here is an immediate stop-the-line bug for the whole research instrument.
10. **Determinism: same seed, 100 ticks, identical `InteractionEvent[]` lists.** Similar to test 9, but compare the accumulated `interactions` arrays across the two runs rather than the final state. Assert the lengths match and every event at index `i` deep-equals. This catches a class of bugs where the final state happens to converge but the intermediate events diverge — indicating a hidden source of non-determinism inside the tick body.
11. **Success rate climbs over time for a simple configuration.** Construct a **pure-L1 world** (all W1-Mono agents with the same initial inventory `{L1: {red-ref: {'red': 1.0}, yellow-ref: {'yellow': 1.0}}}`) with 50 agents, a lattice topology, and a seed. Run 100 ticks. Compute `successRate = interactions.filter(e => e.success).length / interactions.length` over the last 20 ticks and the first 20 ticks. Assert `lastSuccessRate > firstSuccessRate` (monotone improvement at this resolution is not guaranteed tick-by-tick but is guaranteed in the bulk average for a pure-L1 uniform-vocabulary world). This is the Dall'Asta 2008 mean-field convergence prediction made empirical at tick scale. The assertion uses a loose inequality because exact numeric convergence is sensitive to the RNG implementation; the qualitative "last > first" bar is what matters.
12. **Pure-L1 world converges to 100% success rate quickly.** Stronger variant of test 11: with 30 W1-Mono agents sharing the same initial vocabulary and running for 200 ticks on a lattice, assert that the success rate in the last 20 ticks is `> 0.95`. This is the "quickly converges" sanity check — a regression here would indicate the weight update or the hearer-guess branch is structurally broken.
13. **Mixed world has non-zero failure rate initially.** Construct a world with a mix of W1-Mono, W1-Bi, W2-Native, and W2-Immigrant agents with disjoint initial vocabularies (L1 and L2 with no shared lexemes for the same referent). Run 50 ticks. Assert that the success rate in the first 10 ticks is `< 1.0` (at least one failure). This confirms the engine actually exercises the failure branch — a regression that made every interaction succeed would silently pass test 12 but fail test 13.
14. **Retry-limit exhaustion does not crash the tick.** Construct a degenerate config with `retryLimit = 3` and a 2-agent well-mixed world where one agent's inventory has no tokens in the language the policy would pick (forcing the "skip the interaction" branch). Run 5 ticks. Assert that `tick(...)` returns a `TickResult` without throwing and that `state.tickNumber` has advanced by 5. The test does not assert the number of interactions emitted (it could legitimately be zero) — only that the tick function terminates cleanly.
15. **`selectPartner('uniform')` seam accepts the strategy argument without changing behavior.** Call `selectPartner(speaker, world, rng, 'uniform')` directly in the test and assert the returned agent is non-null for a connected topology, and that a second call with a freshly seeded RNG produces the same agent (determinism at the seam level). This test asserts the seam signature is stable — step 14 will extend the signature with a `'preferential'` variant, and this test guards against a regression that accidentally removes the `strategy` parameter or changes its type.
16. **Scheduler modes produce different activation orders for the same state.** Construct a state and call `tick` once with `config.schedulerMode === 'sequential'` and once with `config.schedulerMode === 'random'` (fresh RNGs seeded identically for each). Assert that the resulting `interactions` arrays, **if non-empty for both**, are not identical (the interaction ordering differs because the speaker activation order differs). If the populations happen to produce identical events by coincidence (unlikely but theoretically possible for a pathological topology), the test can also assert that internal call ordering differs via a spy on `selectPartner`, but the simpler interaction-array comparison is sufficient for the common case. This is the "three scheduler modes" acceptance criterion from the step brief.
17. **Normalized weight update keeps Σ weights per (agent, lang, ref) ≈ 1.0 across a full tick.** Construct a world with `config.weightUpdateMode === 'l1-normalized'`, bootstrap, and run 10 ticks. For every agent in `world1.agents` and every `(language, referent)` pair in that agent's inventory, assert that the sum of weights in the `(language, referent)` sub-map is within `± 1e-9` of `1.0`. This is the global invariant the normalized mode promises.
18. **`interactionMemory` is bounded.** Construct a config with `config.interactionMemorySize = 5` and run 30 ticks. Assert that every agent's `interactionMemory.length <= 5` after the run. Assert further that the most recent entry in an agent's memory has `tick === lastInteractionTick` (i.e. FIFO ordering is preserved and the newest entry is at the end).
19. **Events emitted match the tick counter.** Run 3 ticks. Assert that every event's `tick` field matches the tick counter at the time it was emitted: events from tick 0 have `tick === 0`, tick 1 has `tick === 1`, etc. This catches a regression where the tick counter is advanced before events are emitted (or after) and a comment in the code describes the chosen convention explicitly.
20. **Isolated-node topology produces no interactions for the isolated speaker.** Construct a 2-node network topology where the two nodes are disconnected (no edge between them). Bootstrap a world with `agentCount = 2` on this topology (one agent at each node). Run 5 ticks with `config.schedulerMode = 'sequential'`. Assert that no `InteractionEvent` is emitted for either speaker. This exercises the `selectPartner` null branch and confirms the tick loop handles it gracefully.

All 20 tests are deterministic, synchronous, and run under Vitest's default `node` environment. None touch `Date.now()`, wall-clock, file system, network, or `Math.random`. Combined runtime expected under 2 seconds on a development laptop.

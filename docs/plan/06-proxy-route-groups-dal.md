## 1. Goal

Land the three auth-layering pieces that every later route in the app depends on — **the Next.js 16 `proxy.ts` for cheap cookie-presence redirects**, **the `(auth)` / `(public)` route groups with their layouts**, and **the Data Access Layer at `lib/auth/dal.ts`** wrapping `verifySession` in React's `cache()` — as one step because they are co-designed and must compile together. The proxy is a performance optimization (avoids rendering a Server Component just to redirect an anonymous visitor) but is **not** the security boundary; the DAL is. Every Server Component in `app/(auth)/` and every Server Action imported by them will call `verifySession()` directly, and React's per-request `cache()` memoizes the single DB lookup across the whole render tree. The `(public)` route group exists from day one even though it is empty, because carving it out post-hoc would force the matcher regex and the public allowlist to be rewritten later. The step ends with the step-00 placeholder page moved into `app/(auth)/page.tsx` as the first authenticated route, with all three pieces exercised by unit tests against an in-memory SQLite DB, and with `next build` producing a clean bundle so step 07 can drop the login UI and logout action into a working shell.

## 2. Prerequisites

- **Step 04 — session schema and service.** `lib/auth/sessions.ts` must already export `validateSession(token)`, `SESSION_COOKIE_NAME` (the constant `'msksim_session'`), and `getSessionTokenFromCookie()`. The DAL in this step calls exactly these symbols and hardcodes no cookie name of its own.
- Transitively: step 02 (drizzle client singleton with `PRAGMA foreign_keys = ON`), step 03 (`users` table so the optional `getCurrentUser` helper can join), and step 00 (the temporary `app/page.tsx` placeholder and the Vitest wiring) must be in place.
- No new npm dependencies. All three pieces use modules already present: `next/server`, `next/navigation`, `next/headers`, `react` (for `cache()`), and the internal `@/lib/auth/sessions` and `@/lib/db/client` modules.

## 4. Research notes

- **`node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/proxy.md § Exports and § Execution order`** — authoritative v16 reference for the renamed convention. Three facts drive the implementation: (a) _"The file must export a single function, either as a default export or named `proxy`."_ We use the named `proxy` export so the function signature is discoverable by name. (b) The matcher supports negative-lookahead regex literals and the example `'/((?!api|_next/static|_next/image|.*\\.png$).*)'` is the exact shape we adapt. (c) The § Execution order "Good to know" note is the single most load-bearing paragraph for this whole step: _"[Server Functions] are handled as POST requests to the route where they are used, so a Proxy matcher that excludes a path will also skip Server Function calls on that path. A matcher change or a refactor that moves a Server Function to a different route can silently remove Proxy coverage. Always verify authentication and authorization inside each Server Function rather than relying on Proxy alone."_ This sentence is why the DAL is a hard requirement even after the proxy lands.
- **`node_modules/next/dist/docs/01-app/02-guides/authentication.md § Optimistic checks with Proxy (Optional)` and § Creating a Data Access Layer (DAL)`** — the end-to-end pattern this step implements, almost verbatim. The Optimistic-checks section explicitly says _"since Proxy runs on every route, including prefetched routes, it's important to only read the session from the cookie (optimistic checks), and avoid database checks to prevent performance issues"_ — that is the rule that turns our proxy into a cookie-presence check with zero DB access. The DAL section gives the canonical code template: `verifySession = cache(async () => { ... })` with `import 'server-only'`, a `cookies()` read, a validate call, and a `redirect('/login')` on the falsy branch. Our file mirrors this shape and substitutes `validateSession(token)` from step 04 for the guide's `decrypt(cookie)` call (we store raw opaque tokens by design; see step 04 research notes for the rejection of JWT).
- **`node_modules/next/dist/docs/01-app/02-guides/data-security.md § Data Access Layer` and § Authentication and authorization`** — the rationale that makes the DAL a hard requirement rather than a style preference. The § Authentication-and-authorization sub-section contains the exact paragraph quoted in `CLAUDE.md` "Next.js 16 deltas": _"A page-level authentication check does not extend to the Server Actions defined within it. Always re-verify inside the action."_ It also contains the audit checklist entry _"`proxy.ts` and `route.ts`: Have a lot of power. Spend extra time auditing these using traditional techniques."_ and the § Data Access Layer sub-section explicitly recommends `import 'server-only'` plus React's `cache()` and gives the `getCurrentUser = cache(async () => { ... })` template we use for the optional `getCurrentUser` helper.
- **`node_modules/next/dist/docs/01-app/02-guides/upgrading/version-16.md § middleware to proxy`** — confirms the rename is a hard break in v16 (_"The `edge` runtime is NOT supported in `proxy`. The `proxy` runtime is `nodejs`, and it cannot be configured. … The named export `middleware` is also deprecated. Rename your function to `proxy`."_). Our file is written against v16 from scratch, so there is no codemod step, but the deprecation block is cited in a source comment at the top of `proxy.ts` so a future reader searching for "middleware" lands on the right file.
- **`node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/route-groups.md § Convention and § Caveats`** — the authoritative reference for `(auth)` / `(public)` folders. Two facts matter: (a) _"A route group can be created by wrapping a folder's name in parenthesis: `(folderName)`. This convention indicates the folder is for organizational purposes and should not be included in the route's URL path."_ so `app/(auth)/page.tsx` resolves to `/`, not `/auth/`. (b) _"Routes in different groups should not resolve to the same URL path."_ — this is why the step-00 placeholder page at `app/page.tsx` must be deleted before `app/(auth)/page.tsx` is added; leaving both would produce two handlers for `/` and Next would error at build time.
- https://react.dev/reference/react/cache (WebFetched) — React's `cache()` docs pin five facts the DAL depends on: (a) import path is `import { cache } from 'react'` (not `react/cache`); (b) memoization is per-render, not global — each new server request gets a fresh cache, which is exactly the per-request dedup behavior we want; (c) argument matching is `Object.is` identity-based, which is why `verifySession()` takes no arguments (the cookie is read inside via `cookies()` so there is nothing to key on); (d) _Server Components only_ — safe under the `import 'server-only'` guard; (e) **cached errors are re-thrown from every call site with the same arguments**, which is subtle: if `verifySession()` throws a `NEXT_REDIRECT` on its first call, every subsequent call in the same render pass re-throws that same redirect. That is the intended behavior and is why placing the sole `verifySession()` call in `(auth)/layout.tsx` does not leave leaf components exposed if they also call it — the cached redirect propagates.
- https://cheatsheetseries.owasp.org/cheatsheets/Authorization_Cheat_Sheet.html (WebFetched) — OWASP's authorization cheat sheet reinforces the "don't rely on a single edge check" principle: _"an attacker only needs to find one way in … authorization logic must be distributed — not concentrated at one chokepoint. Data access layers particularly need independent verification to prevent horizontal privilege escalation through direct object reference manipulation."_ The Next 16 data-security doc and OWASP converge on the same rule from different vocabularies — that convergence is the justification for paying the per-request DB lookup on every authenticated route render rather than trusting the proxy alone.
- **Path not taken — put the session validate call in `proxy.ts` itself (single-source redirect logic).** Tempting because it would mean `(auth)/layout.tsx` could skip its own `verifySession()` call. Rejected for three load-bearing reasons. (1) The proxy runs on **every** matched request including client-side-transition RSC prefetches; a full DB lookup (even a primary-key sqlite read) on every prefetch is a hot-path cost with no security win. The Next doc's § Optimistic-checks-with-Proxy section explicitly calls this out. (2) The proxy cannot import `lib/auth/sessions.ts` safely — that module imports `lib/db/client.ts`, which pulls in `better-sqlite3` native bindings. Running those through the proxy's module graph is an invitation for Turbopack to try to bundle the native binary into the edge-adjacent code path and fail opaquely. The DAL's `import 'server-only'` guard exists exactly to prevent this leak, and keeping the session validate call inside the DAL keeps the proxy's module graph trivial. (3) The Next 16 doc's § Execution order "Good to know" paragraph about Server Functions bypassing proxy coverage means even a flawless proxy validate does not protect Server Actions — they have to re-verify anyway. If they have to re-verify, moving the validate into the DAL centralizes the check and the proxy becomes a pure UX optimization.
- **Path not taken — a single `app/layout.tsx` that calls `verifySession()` and no `(auth)` route group at all.** Simpler on paper, but the Next 16 authentication doc warns against this under § Layouts-and-auth-checks: _"Due to Partial Rendering, be cautious when doing checks in Layouts as these don't re-render on navigation, meaning the user session won't be checked on every route change."_ With a single root layout, an expired session mid-session would not trigger a redirect on client-side navigation between two authenticated routes. The group-level `(auth)/layout.tsx` fixes this because crossing from `(public)` back into `(auth)` (or between a page and an error boundary at the group root) re-renders the group layout. Combined with the per-Server-Component DAL calls, this gives two independent re-check points.
- **Path not taken — NextAuth / Auth.js or another third-party auth library.** The Next 16 doc lists ten auth-library integrations under § Resources. All are rejected here for the same reasons step 04 rejects `next-auth`, `lucia-auth`, `better-auth`, and `iron-session`: this app has two users, local credentials, a drizzle schema we already own, and a revocation requirement (step 04 § Path-not-taken-JWT). A library adds opinions we don't want (OAuth-provider assumptions, adapter layers hiding the schema, encrypted stateless cookies that re-introduce the revocation problem) for no measurable win over ~60 lines of proxy + DAL glue. The Next 16 data-security audit checklist explicitly treats rolling-your-own DAL as a first-class pattern, not a fallback.

Total citations: 9 (5 local Next doc sections cited by name — `proxy.md §§ Exports/Execution-order`, `authentication.md §§ Proxy/DAL`, `data-security.md §§ DAL/Auth-and-authz`, `version-16.md § middleware-to-proxy`, `route-groups.md §§ Convention/Caveats`; 2 external URLs WebFetched — React `cache()` docs and OWASP Authorization cheat sheet; 3 paths-not-taken — validate-in-proxy, single-root-layout, third-party-auth-lib). Exceeds the ≥ 5 total floor and each sub-floor (≥ 3 local, ≥ 2 external, ≥ 1 path-not-taken).

## 7. Implementation approach

Do the four pieces in the order listed below. This order is not cosmetic: the DAL is the contract the layouts depend on, the layouts are the structure the proxy is an optimization for, and the page move is the only change that could break `next build` if sequenced wrong.

**Slice one — the DAL (`lib/auth/dal.ts`).** Write this first because it is the single file every other piece in this step imports from. Start the file with `import 'server-only';` on line 1 (CLAUDE.md "Database access patterns" and `data-security.md § Preventing client-side execution of server-only code`). Import `cache` from `react` — **not** `react/cache`, which is a different ESM entry point and will fail to resolve in v16. Import `redirect` from `next/navigation` (the App Router navigation module; `next/router` is the obsolete Pages Router API). Import `getSessionTokenFromCookie` and `validateSession` from `@/lib/auth/sessions`. Define and export `type Session = { userId: string; expiresAt: Date }`. Define `verifySession` as a module-level `export const verifySession = cache(async (): Promise<Session> => { ... })`. Inside the function: (1) call `const token = await getSessionTokenFromCookie()`; (2) if `token == null`, call `redirect('/login')` — `redirect` throws `NEXT_REDIRECT` which the Next framework catches above the component tree, so the TypeScript compiler sees `redirect` return `never` and flow-narrowing makes the rest of the body reachable only on the happy path; (3) call `const session = await validateSession(token)`; (4) if `session == null`, call `redirect('/login')`; (5) return `{ userId: session.userId, expiresAt: session.expiresAt }`. Do **not** add any fallback logic for the `redirect` call — the throw is the return path, and wrapping it in `try/catch` would swallow the `NEXT_REDIRECT` and break the framework contract. Add a JSDoc comment above `verifySession` explaining the `cache()` wrapping: "Wrapped in React's `cache()` so twenty components calling `verifySession()` in the same render tree resolve to one DB lookup. Cache is per-request; each new server request starts fresh. Identity-matching on zero arguments collapses every call site to the same cache key by construction." Also note that the cache re-throws errors from every call site with the same key — so if the first call throws `NEXT_REDIRECT`, every subsequent call in the same render pass re-throws it, which is the intended propagation behavior (see the React `cache()` docs).

Then add the optional `getCurrentUser` helper, also wrapped in `cache()`. Its body calls `await verifySession()` (which is itself cached, so this does not double-dip on the cookie read), then does a single `db.select({ id: users.id, username: users.username }).from(users).where(eq(users.id, session.userId)).limit(1)` query, and returns `{ userId: row.id, username: row.username, expiresAt: session.expiresAt }` or calls `redirect('/login')` if the row is missing (which would indicate a deleted-user race — the session cascade from step 04 should make this unreachable, but belt-and-braces it anyway). This helper is what step 07's header component will call to render the logged-in username without needing to import `verifySession` directly.

**Slice two — the route group layouts.** Create `app/(auth)/layout.tsx` and `app/(public)/layout.tsx`. Both are Server Components by default (no `'use client'` directive). `app/(auth)/layout.tsx` imports `verifySession` from `@/lib/auth/dal`, `await`s it at the top of the component body (`const session = await verifySession();`), binds `session` to a `_session` variable or uses a `void session;` expression so the unused-variable lint rule does not fire (step 07 will consume `session` properly when it builds the header). Returns `<>{children}</>` — the actual header/nav UI belongs to step 07. `app/(public)/layout.tsx` simply returns `<>{children}</>` with a JSDoc comment documenting its pre-emptive role for the `/reports/*` carveout. Neither file declares `<html>` or `<body>` — the step-00 root `app/layout.tsx` still owns those. A JSDoc line in `(auth)/layout.tsx` should call this out explicitly because a future reader might be confused by a layout that has no HTML scaffolding.

**Slice three — the proxy (`proxy.ts`).** Create at the repository root (not under `app/`, not under `src/` — there is no `src/` directory in this project). Write the comment block at the top: _"Next.js 16 file convention. This was called `middleware.ts` in Next ≤ 15. Exported function is `proxy`. Runs on the Node.js runtime; setting `runtime` in this file throws at build time. See CLAUDE.md 'Next.js 16 deltas from training data' and node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/proxy.md."_ Import `NextResponse` and `type { NextRequest }` from `next/server` and `SESSION_COOKIE_NAME` from `@/lib/auth/sessions`. Define a module-level `const PUBLIC_PATHS = ['/login', '/reports'] as const` and a helper `function isPublicPath(pathname: string): boolean` that returns true if the pathname equals any entry exactly or starts with any entry followed by `/` (so `/login` matches `/login`, `/login/callback`, etc., and `/reports` matches `/reports/abc/123`). Export `async function proxy(request: NextRequest)`: (1) read `const { pathname } = request.nextUrl`; (2) if `isPublicPath(pathname)` or pathname starts with `/api/auth/` (the future `/api/auth/logout` endpoint, pre-allowlisted), return `NextResponse.next()`; (3) read `const hasSessionCookie = request.cookies.has(SESSION_COOKIE_NAME)` (presence check only — the proxy does not call `validateSession`, does not import the sessions module beyond the cookie-name constant, does not touch the DB); (4) if `hasSessionCookie`, return `NextResponse.next()`; (5) otherwise build the redirect: `const loginUrl = request.nextUrl.clone(); loginUrl.pathname = '/login'; loginUrl.searchParams.set('next', pathname + request.nextUrl.search);` and return `NextResponse.redirect(loginUrl)`. Note that `NextResponse.redirect` defaults to a 307 Temporary Redirect, which is what we want (preserves the original HTTP method, though in practice proxy is hit on GET for page requests).

Export the matcher `config` as the second named export:

```
export const config = {
  matcher: [
    // Run on everything EXCEPT:
    //   - Static Next internals: _next/static, _next/image
    //   - Public-folder assets by extension (favicon, svg, png, jpg, woff, woff2, ico)
    //   - The login page (allowlisted by name since the matcher runs before our handler)
    //   - The future /reports/* public routes (pre-allowlisted so zero-refactor v2 carveout)
    //   - API auth routes under /api/auth/*
    '/((?!_next/static|_next/image|favicon\\.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp|woff|woff2)|login|reports|api/auth).*)',
  ],
};
```

The matcher is a belt: it prevents the proxy handler from even running for static assets and the login page, which is a measurable perf win on every request. The `isPublicPath` braces inside the handler body are a suspender: they cover the case where the matcher's negative-lookahead regex lets something through that the handler should still allow (e.g., a `/reports` request without a trailing slash, which the regex may or may not anchor on depending on the exact `path-to-regexp` semantics). The doubled defense is also load-bearing because the Next 16 proxy doc § Negative matching has an explicit note that _"Even when `_next/data` is excluded in a negative matcher pattern, proxy will still be invoked for `_next/data` routes. This is intentional behavior to prevent accidental security issues."_ — so even a "perfect" matcher regex may still invoke the handler for certain paths, and the in-handler allowlist is the canonical place to make the final decision.

**Slice four — move the placeholder.** Copy the body of `app/page.tsx` (the minimal "welcome to msksim" heading + link from step 00) into `app/(auth)/page.tsx`. Delete `app/page.tsx`. Do not add any new content or styling — step 07 replaces this page with the real app shell. The file stays small and boring on purpose.

Finally run `npm run typecheck && npm run lint && npm run build && npm test -- lib/auth/dal` in sequence. The build step is the new one here: this is the first step in the pipeline that has a real Next surface (layouts, proxy, DAL) and the build is where Turbopack will catch any accidental client/server boundary violations (e.g., importing `lib/auth/dal.ts` into a client component, which would fail because `import 'server-only'` throws during client-bundle resolution). If the build fails with a native-binding error from `better-sqlite3`, the root cause is almost certainly a stray import from `lib/db/client.ts` into a module that Turbopack thinks is part of the client graph — check the import chain in the error, not the module being imported.

## 9. Unit tests

All live in `lib/auth/dal.test.ts`. Use the in-memory-sqlite pattern from `lib/auth/sessions.test.ts` (step 04): open `better-sqlite3(':memory:')`, run `PRAGMA foreign_keys = ON`, apply the drizzle schema, and build a fresh drizzle client per test. Stub `next/headers` with `vi.mock('next/headers', () => ({ cookies: async () => stub }))`, where `stub` is an in-file `Map`-backed fake with `get(name)` returning `{ value } | undefined`. Stub `next/navigation`'s `redirect` with `vi.mock('next/navigation', () => ({ redirect: vi.fn((path: string) => { throw new Error('NEXT_REDIRECT:' + path); }) }))` — the stub mimics the real `redirect`'s control-flow-terminating throw so the test can both assert that it was called and assert that control flow never reaches code after it.

1. **`verifySession` with a valid session returns `{ userId, expiresAt }`.** Insert a user row and a non-expired session row directly via the test's drizzle client. Set the cookie stub to return that session's token. Call `verifySession()` and assert the returned `userId` matches the seeded user's id and the returned `expiresAt` matches the row's `expires_at`. Assert the mocked `redirect` was **not** called.
2. **`verifySession` with no cookie calls `redirect('/login')`.** Leave the cookie stub empty. Call `verifySession()` inside a `try/catch`. Assert the catch path fires (because our `redirect` stub throws), assert the mocked `redirect` was called exactly once with `'/login'`, and assert the function never returned a value.
3. **`verifySession` with an expired session calls `redirect('/login')`.** Insert a user and a session row whose `expires_at` is one millisecond in the past (use `vi.setSystemTime` to make this deterministic). Set the cookie stub to return that token. Call `verifySession()` inside a `try/catch`. Assert `redirect('/login')` was called. Also assert that the expired row is still in the database after the call — `verifySession` (via `validateSession` from step 04) does not lazy-delete expired rows.
4. **`verifySession` with an unknown token calls `redirect('/login')`.** Set the cookie stub to return the string `'0'.repeat(64)` (a syntactically valid hex token that is not present in the sessions table). Call `verifySession()` inside a `try/catch`. Assert `redirect('/login')` was called. Extra: also test a malformed token (empty string, `'not-hex'`, and a 32-character string) and assert the same behavior in each case — `validateSession` from step 04 returns `null` for all of these, and our DAL flows cleanly into the redirect branch.

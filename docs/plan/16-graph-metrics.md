## 1. Goal

Implement the **graph-derived per-tick observables** from `docs/spec.md` ôÏ7.1 as a small, pure module that the step-20 simulation worker will call once per tick after the interaction engine (step 13) has produced its `InteractionEvent[]` for the tick. Five metrics land in this step:

1. **Largest-cluster size** ã the size of the largest connected component in the *token agreement graph*, where nodes are agents and an edge exists between any two agents that share at least one `(referent ã top-weighted token)` mapping. Computed **per world** (once for World 1, once for World 2). This is the canonical Naming Game observable that `docs/spec.md` ôÏ2 cites as the signature of topology-induced coarsening.
2. **Cluster count** ã the number of connected components in the token agreement graph **with size ãË 2**, per world. The `ãË 2` filter is deliberate: singletons (agents whose top-weighted tokens do not match anyone else's) are not a "cluster" in the sociolinguistic sense the research is probing. This is the observable `docs/spec.md` ôÏ7.1 enumerates and ôÏ2 argues is impossible under mean-field.
3. **Interaction-graph modularity** ã the Louvain modularity score on the **cumulative successful-interaction graph** (nodes = agents, undirected edges weighted by the frequency of successful interactions across all ticks so far). Computed via `graphology-communities-louvain`. This metric directly answers `docs/spec.md` RQ4 (emergent social cohesion) by measuring whether the interaction network fragments into communities.
4. **Assimilation index** ã among this tick's successful interactions between W2-Immigrants and W2-Natives, the fraction that occurred in L2. Rises toward 1 under assimilation, falls toward 0 under segregation. A **single scalar per tick** (not per world, because by construction it lives in World 2 only). This is the primary observable for `docs/spec.md` RQ1 and RQ5.
5. **Segregation index** ã the Louvain modularity of the **subgraph** of the cumulative interaction graph induced by W2-Immigrant nodes only. High values mean immigrants interact mostly among themselves (forming a distinct community within World 2); low values mean they are well-integrated with natives. This is the complement to the assimilation index and the second primary observable for RQ1.

All five functions are **pure** given the inputs: `(world1: World, world2: World, interactionGraph: Graph, tickInteractions: InteractionEvent[]) ã GraphMetricsSnapshot`. The caller ã step 20's worker integration ã is responsible for maintaining the cumulative `interactionGraph` across ticks (adding nodes on first appearance, incrementing edge weights on each successful interaction). Step 16 provides a helper in `lib/sim/metrics/interaction-graph.ts` that performs the per-tick merge so callers do not have to hand-roll the graphology calls, but step 16 **does not** own the lifetime of the graph object ã that lives on the worker state.

This step is parallel to step 15 (scalar metrics) and feeds step 17 (run-summary classifiers) and step 22 (metrics dashboard). It is the module where `docs/spec.md` ôÏ2's coarsening claim, ôÏ2's ghettoization argument, and the RQ1/RQ2/RQ4 observables all become concrete code.

## 3. Spec references

- `docs/spec.md` **ôÏ7.1 (Per-tick scalar metrics)** ã the authoritative list of what this step implements. The five load-bearing rows are:
  > **Largest-cluster size** ã Size of the largest connected component in the "token agreement graph" (edges where two agents share a top-weighted token). RQ1, RQ2.
  > **Cluster count** ã Number of connected components in the token-agreement graph with size ãË 2. RQ1, RQ2.
  > **Interaction-graph modularity** ã Louvain modularity score on the cumulative successful-interaction graph. High modularity = strong clustering. RQ2, RQ4.
  > **Assimilation index** ã Among successful interactions between W2-Immigrants and W2-Natives, the share that occurred in L2. Rises under assimilation, falls under segregation. RQ1, RQ5.
  > **Segregation index** ã Louvain modularity of the subgraph restricted to W2-Immigrants. Rises when immigrants interact mostly among themselves. RQ1, RQ5.

  Every type, test, and implementation choice in this step traces back to one of these five rows.
- `docs/spec.md` **ôÏ1.2 RQ1 ã Assimilation vs. segregation thresholds.** "Under what population ratios (monolingual:bilingual) and interaction probabilities do bilingual immigrants in World 2 assimilate into the host linguistic community versus segregate into an insular one?" The assimilation index and segregation index are the two observables that **define** RQ1 operationally. Without them, RQ1 is unmeasurable; with them, step 28's parameter sweep can sweep `monoToBiRatio` along a grid and plot the threshold at which the assimilation index collapses, which is the publication-grade figure the researchers want. Step 16 is therefore on the critical path for RQ1.
- `docs/spec.md` **ôÏ1.2 RQ2 ã Role of spatial topology.** "Does a 2D lattice constraint produce qualitatively different macroscopic outcomes than well-mixed populations with identical composition?" The largest-cluster size and cluster count are the primary observables for RQ2: `docs/spec.md` ôÏ2 states explicitly that well-mixed populations cannot sustain multiple coexisting clusters, so a cluster count that stays at 1 under well-mixed mode and climbs above 1 under lattice mode **is** the empirical falsification of the "lattice doesn't matter" null hypothesis. Step 16's token-agreement-graph construction is the mechanism by which that falsification becomes measurable.
- `docs/spec.md` **ôÏ1.2 RQ4 ã Emergent social cohesion.** "To what extent does successful communication predict or drive social bonding (measured as interaction-graph density and clustering)?" The interaction-graph modularity metric is the direct operationalization ã the question "are there cohesive subgroups?" maps exactly to "is Louvain Q non-trivially positive?" The cumulative-interaction-graph contract in this step is the reason step 23 (network view) can layer a visualisation on top of the same graph object without a separate build, because both step 16 and step 23 share the graphology Graph maintained by the worker.
- `docs/spec.md` **ôÏ2 (Does the Lattice Matter?).** The argument of ôÏ2 is that well-mixed populations converge homogeneously (single vocabulary, single cluster) while 2D lattices undergo topology-induced coarsening (multiple coexisting clusters during a long intermediate regime). Step 16's token-agreement graph and its `largestClusterSize`/`clusterCount` outputs are the instruments that quantify coarsening tick-by-tick. ôÏ2's time-to-consensus scaling `N^(1+2/d)` on a lattice versus `N^(3/2)` on mean-field becomes visible in the researcher's dashboard because `clusterCount` starts high on a lattice and slowly decays, while on well-mixed it drops to 1 almost immediately. Step 16 must produce that signature correctly or ôÏ2's entire case becomes unmeasurable.
- `docs/spec.md` **F4 (Spatial mode selector).** "The same population config can be run in all three modes without code changes; the topology is persisted with the run." Step 16's token-agreement graph is constructed **without reference to `world.topology.kind`** ã it only asks "which agents share a top-weighted token?", which is a function of inventories alone, not positions. This keeps F4's topology-agnostic invariant intact: running the same config under lattice, well-mixed, and network topologies all feed step 16 identically and produce metrics that are directly comparable across topologies. Step 16's code has **zero branches on `topology.kind`** and a grep-check in the review step (ôÏ7) enforces that.
- `docs/spec.md` **F7 (Live lattice view).** F7 lists "dominant token for a selected referent" as one of the per-cell projections. The "top-weighted token per referent" utility that step 16 builds for the token-agreement graph is the **same computation** F7's renderer needs in step 21; step 16 therefore exports a helper `topWeightedTokenByReferent(inventory, language?) ã Map<Referent, Token>` from `lib/sim/metrics/graph.ts` (or a sibling utility file) so step 21 can reuse it without re-implementing. This avoids the "two places compute 'dominant token' with subtly different tie-breaking" bug class.
- `docs/spec.md` **F8 (Live network view).** "A WebGL graph rendering showing the cumulative interaction network (nodes = agents, edges = past successful interactions weighted by frequency). Overlays Louvain community detection to highlight emergent clusters." Step 16's `interaction-graph.ts` helper is the canonical builder for this exact graph object, and the Louvain communities it computes for the modularity score are exactly the communities step 23's sigma.js renderer will color-code. Step 16 therefore produces two derivative outputs for step 23's downstream consumption: the `modularity` scalar (lands in the snapshot) and, as an optional second return, a `communityAssignments: Map<AgentId, number>` produced by `louvain.detailed()` ã step 23 will read this from the worker's cached state to color nodes. The main `computeGraphMetrics` return shape includes `modularity` as a scalar; a sibling helper `computeInteractionGraphCommunities(graph)` returns the full assignments and is exported for step 23's use.

**Paths not taken:**

7. **Leiden algorithm instead of Louvain.** Considered and rejected. The Leiden algorithm (Traag, Waltman, van Eck 2019, "From Louvain to Leiden: guaranteeing well-connected communities") fixes two known issues with Louvain documented on the Wikipedia article: the resolution limit (small communities get absorbed into larger ones) and the disconnected-community artifact (Louvain can leave communities that are technically disconnected). Leiden is strictly better on both counts and is the modern recommendation. **However**: (a) `graphology` has no Leiden plugin in its standard library ã the only community-detection implementation in the graphology ecosystem is `graphology-communities-louvain`, and adding Leiden would require pulling in an unrelated package (e.g., `igraph` bindings) that does not interoperate with the rest of the graphology toolchain that step 23 depends on for rendering; (b) the research question the `docs/spec.md` ôÏ7.1 row cites is "Louvain modularity," not "Leiden modularity," so the metric the spec names is specifically the Louvain flavor ã changing the algorithm would silently change what the researchers see in their dashboard, and the literature the spec references (Baronchelli et al. 2010's "In-depth analysis of the naming game dynamics in social networks," `docs/spec.md` ôÏ12.1) is itself Louvain-era; (c) the Louvain-vs-Leiden difference is a few percent on the Q value for small graphs and does not change the qualitative signal step 16 is measuring (RQ1's threshold location, RQ2's lattice-vs-well-mixed gap). The Leiden decision can be revisited in v2 if a graphology-compatible Leiden plugin lands; for v1 the spec's Louvain citation and the standard-library availability both point at Louvain, and step 16 honors both.
8. **Recomputing the cumulative interaction graph from the full `InteractionEvent[]` history every tick.** Considered and rejected. The naive approach ã "store all successful interactions in an array and rebuild the graphology graph from scratch each tick before calling Louvain" ã has time complexity `O(T û E)` where `T` is the current tick and `E` is the total successful-interaction count up to that tick, leading to quadratic growth in total wall-clock cost over a run. For the target range `N = 500` with `interactions_per_tick ã N/2 = 250` successful interactions per tick and runs of 10ãÇ ticks, that is 2.5 million events and a per-tick rebuild of `O(2.5e6)` at the end of the run. Instead, step 16 adopts the **caller-maintained incremental graph** design: the worker (step 20) holds a single `UndirectedGraph` object across ticks, calls `updateInteractionGraph(graph, tickInteractions)` once per tick to merge this tick's new edges (O(tickInteractions) per tick, total O(totalSuccessfulInteractions) across a run), and passes the same graph into `computeGraphMetrics`. This reduces the per-tick graph-maintenance cost to be linear in the tick's interaction count and keeps the cumulative graph's memory footprint linear in the total edge count rather than re-materializing it every tick. The incremental design also matches graphology's idiomatic usage pattern (`mergeEdge(source, target, attrs)` is a first-class graphology method designed for exactly this streaming-update use case), and it is the same pattern step 23's sigma.js renderer will consume for incremental visual updates. Step 16's implementation prose (ôÏ7) documents this contract explicitly so step 20's integration cannot accidentally duplicate the graph each tick.
9. **Incrementally maintaining the token agreement graph across ticks.** Considered and rejected for v1. The token agreement graph depends on each agent's current top-weighted token per referent, which can change on every weight update (step 13's öã¤ application); maintaining the graph incrementally would require tracking which agents' top tokens changed this tick and re-evaluating only the O(changes) affected edges. This is feasible but complex, and the `docs/spec.md` ôÏ7.1 row does not demand it. The O(Nôý) rebuild for N ãÊ 500 is `2.5 û 10ãç` agent-pair comparisons per tick, which is well under 10 ms on a modern browser even with a naive implementation. The step-16 prompt explicitly says "a separate graph, rebuilt each tick (or incrementally if performance requires ã v1 does it fresh)," so the v1 design honors that and documents the incremental design as a v2 optimization. If a future run with N = 10ãÇ shows the token-agreement rebuild dominates per-tick cost, the incremental design can be added without breaking the function signature.
10. **Returning the raw community-assignment map as part of `GraphMetricsSnapshot`.** Considered and rejected. The snapshot is persisted tick-by-tick into the drizzle `tick_metrics` table (step 08) and sent over the `postMessage` boundary on every tick (step 20); including a `Map<AgentId, communityId>` of length N = 500 on every tick would bloat the wire payload and the database row by two orders of magnitude for a metric that is only needed by the network view (step 23). Instead, step 16 exports a **separate** helper `computeInteractionGraphCommunities(graph: Graph): { assignments: Map<AgentId, number>; count: number; modularity: number }` that step 23 calls directly (on-demand, not per-tick) when the user clicks into the network view. The per-tick `computeGraphMetrics` function returns only the scalar `modularity` field and does not carry the assignments. This keeps the snapshot small, the wire fast, and the responsibilities clean.

- `lib/sim/metrics/graph.ts` ã the **public module**. Exports:
  - `function computeGraphMetrics(world1: World, world2: World, interactionGraph: Graph, tickInteractions: InteractionEvent[]): GraphMetricsSnapshot` ã the main entry point the step-20 worker calls once per tick, after step 15's `computeScalarMetrics` and after `updateInteractionGraph` (from the sibling `interaction-graph.ts` helper) has merged this tick's events into the cumulative graph.

  - `function largestClusterSize(agreementGraph: UndirectedGraph): number` ã returns the size of the largest connected component. Uses graphology's `connectedComponents` from `graphology-components` if that helper is already installed transitively, or a simple BFS/Union-Find implementation if not (see ôÏ7 implementation note). For N ãÊ 500 a hand-rolled BFS is sufficient and avoids a new dep.
  - `function clusterCount(agreementGraph: UndirectedGraph): number` ã returns the number of connected components whose size is ãË 2.
  - `function computeAssimilationIndex(tickInteractions: InteractionEvent[], l2Label: Language): number | null` ã the per-tick assimilation index, computed over the subset of `tickInteractions` where `(speakerClass, hearerClass)` is `(W2-Immigrant, W2-Native)` or `(W2-Native, W2-Immigrant)` **and** `success === true`. If that subset is empty, returns `null` ã see ôÏ7 for the rationale (the spec prompt says "NaN or null; document the choice"; step 16 chooses `null` because the downstream drizzle column is nullable and JSON's `null` is structuredClone-safe while `NaN` silently becomes `null` on JSON round-trip anyway, so being explicit is cleaner).
  - `function computeSegregationIndex(interactionGraph: Graph, world2: World, rng: RNG): number` ã extracts the subgraph induced by W2-Immigrant nodes and returns `louvain.detailed(subgraph).modularity`. Uses the step-09 seeded RNG for determinism.
  - `function computeInteractionGraphCommunities(interactionGraph: Graph, rng: RNG): { assignments: Map<AgentId, number>; count: number; modularity: number }` ã the on-demand, full-detail Louvain call. Called by step 23's network view, not by the per-tick hot path. Exposed here to centralize the Louvain-options boilerplate.
  - `function topWeightedTokenByReferent(agent: AgentState): Map<Referent, Token>` ã the shared utility that derives each agent's top-weighted token per referent from its inventory. Consumed by `buildTokenAgreementGraph` internally, and re-exported so step 21's lattice renderer (F7 "dominant token" projection) can reuse it. Tie-breaking: lexicographic ascending on the token string, applied after numeric-tie detection. Deterministic by construction (no RNG).

- `lib/sim/metrics/graph.test.ts` ã Vitest suite covering every public function enumerated above. Eight fixture-driven tests plus determinism assertions; full list in section 9.

- `lib/sim/metrics/interaction-graph.ts` ã the **incremental cumulative graph helper**. Exports:
  - `function createInteractionGraph(): UndirectedGraph` ã factory returning a fresh graphology `UndirectedGraph`. The worker (step 20) calls this once at simulation start and then passes the returned graph into `updateInteractionGraph` every tick.
  - `function updateInteractionGraph(graph: UndirectedGraph, tickInteractions: InteractionEvent[]): void` ã merges this tick's successful interactions into the cumulative graph. For each event where `success === true`, ensures both `speakerId` and `hearerId` exist as nodes (`graph.mergeNode(...)`, which is a no-op if the node is already present) and then increments the edge weight via `graph.mergeEdge(speakerId, hearerId, { weight: (w ?? 0) + 1 })` ã the graphology idiom for "add edge with weight 1, or increment existing edge's weight by 1." Ignores failed interactions (success === false) because the spec's definition of the cumulative interaction graph is "past **successful** interactions," per `docs/spec.md` ôÏ7.1's modularity row and F8's network-view definition.
  - `function interactionGraphNodeCount(graph: UndirectedGraph): number` ã thin wrapper returning `graph.order`. Exposed for step 22's dashboard to show the "number of active interactors" as a secondary display.
  - `function interactionGraphEdgeCount(graph: UndirectedGraph): number` ã thin wrapper returning `graph.size`. Same rationale.
  - These two wrappers exist so that consumers do not have to import `graphology` directly just to read graph statistics ã keeping the graphology dependency internal to `lib/sim/metrics/` wherever possible reduces the number of places step 20's worker bundle has to pull in graphology symbols.

- `lib/sim/metrics/interaction-graph.test.ts` ã Vitest suite covering the incremental update helper. Five tests: (a) empty-graph factory, (b) single-interaction creates two nodes and one edge with weight 1, (c) repeated same-pair interactions accumulate to weight N, (d) failed interactions are ignored, (e) multi-pair interactions produce the correct node/edge counts. Full list in section 9.

## 7. Implementation approach

This section is prose, per the plan-file contract. The order matters because later files depend on earlier ones and the test suites are strict about their fixtures.

**Step one is dependency management.** Begin by running `npm view graphology-communities-louvain version` to confirm the current stable release on npm. Install it with `npm install graphology-communities-louvain@<version>` and verify the resolved version now appears in `package.json` under `dependencies`. Do **not** install any separate `@types/...` package ã the WebFetch during research confirmed that the plugin bundles its TypeScript declarations, just like `graphology` itself does (per step 10's research). Run `npx tsc --noEmit` immediately after install to catch any resolution problems. Also verify that `import louvain from 'graphology-communities-louvain';` works in a scratch file (the default export is the `louvain` function, confirmed by the WebFetch). If TypeScript reports missing types, the fallback is to add `"graphology-communities-louvain"` to `tsconfig.json`'s `compilerOptions.types` or to import via the package's declared entry point ã but the WebFetched docs indicate bundled types, so this should not be necessary on a fresh install.

**Step two is extending the snapshot type.** Open `lib/sim/metrics/types.ts` (created in step 15) and append the `GraphMetricsSnapshot` interface as shown in section 6. Preserve the existing `ScalarMetricsSnapshot` interface verbatim ã step 16 does not modify it. Add a short comment block above the new interface explaining that `interactionGraphModularity` and `segregationIndex` are Louvain modularity scores (range `[-1, +1]`, with `> 0.3` indicating non-trivial community structure per the Wikipedia reference), and that `assimilationIndex` is `null` when the tick contains no W2-Imm ã W2-Native successful interactions (because the fraction is 0/0 and the spec prompt chose `null` over `NaN`).

**Step three is the interaction-graph helper.** Create `lib/sim/metrics/interaction-graph.ts`. Import `UndirectedGraph` from `graphology` (`import { UndirectedGraph } from 'graphology';`) and the `InteractionEvent` type from wherever step 13 exported it (likely `@/lib/sim/engine` or `@/lib/sim/types`). Implement `createInteractionGraph()` as `return new UndirectedGraph();` ã no options needed, because `UndirectedGraph` already implies undirected edges and the default behavior is a simple graph (at most one edge between any two nodes), which is what the cumulative-weight idiom requires. Implement `updateInteractionGraph(graph, tickInteractions)` as a loop over `tickInteractions`: skip any event with `success === false`, then for each successful event extract `speakerId` and `hearerId`, call `graph.mergeNode(speakerId)` and `graph.mergeNode(hearerId)` (these are idempotent ã no-ops on existing nodes), and then call `graph.updateEdge(speakerId, hearerId, (attrs) => ({ weight: (attrs?.weight ?? 0) + 1 }))` or equivalently `graph.mergeEdge(speakerId, hearerId, { weight: 1 })` followed by a manual increment. The **simpler, idiomatic graphology pattern** the plugin documentation endorses is `graph.updateEdge(source, target, (attrs) => ({ ...attrs, weight: (attrs.weight ?? 0) + 1 }))`, which creates the edge on first call (attrs is an empty object) and merges on subsequent calls. Use that form. Edge case: if `speakerId === hearerId` (self-interaction), graphology's default `UndirectedGraph` disallows self-loops and throws ã the interaction engine (step 13) already filters self-interactions out in its `selectPartner` step, so this case should not arise, but defensive code wraps the `updateEdge` call in a `speakerId !== hearerId` guard as belt-and-suspenders. The two thin wrappers `interactionGraphNodeCount(graph) = graph.order` and `interactionGraphEdgeCount(graph) = graph.size` are one-liners.

**Step four is the token-agreement graph.** Create `lib/sim/metrics/graph.ts`. Start with the utility `topWeightedTokenByReferent(agent: AgentState): Map<Referent, Token>` because every other function depends on it. For each `(language, referentMap)` entry in `agent.inventory`, for each `(referent, tokenMap)` entry in `referentMap`, iterate the tokens and find the one with the highest weight. **Tie-breaking rule**: if two tokens share the maximum weight, pick the lexicographically smallest token string (ascending). This deterministic tie-breaker is what ensures the token-agreement graph is identical across runs with the same state, which is what the determinism tests in ôÏ9 rely on. The function returns a `Map<Referent, Token>` where the key is the referent and the value is the winning token. **Cross-language aggregation**: if an agent has the same referent in both L1 and L2, `topWeightedTokenByReferent` needs to decide whether to treat them as separate "top tokens per (referent, language)" or as a single "top token per referent across all languages." The spec row in ôÏ7.1 says "edges where two agents share a top-weighted token" ã singular "a top-weighted token," which strongly implies per-referent-across-languages aggregation (otherwise every bilingual pair would share at least one edge trivially because their L1 top tokens would match even when their L2 top tokens differ). Step 16 therefore aggregates across languages: for each referent, pick the single highest-weighted token across all languages the agent knows. If the highest L1 weight and the highest L2 weight tie, the tie-breaker falls back to the lexicographic rule. Document this decision inline in the function doc comment so future contributors understand the choice.

Next, implement `buildTokenAgreementGraph(world: World): UndirectedGraph`. Construct a fresh `new UndirectedGraph()`, add every agent as a node (`graph.addNode(agent.id)`), and then perform the O(Nôý) pairwise scan: for every ordered pair `(a, b)` with `a.id < b.id`, compute `topWeightedTokenByReferent(a)` and `topWeightedTokenByReferent(b)`, and check whether there exists any referent `r` such that `aTopTokens.get(r) === bTopTokens.get(r)`. If yes, add an edge `graph.addEdge(a.id, b.id)`. **Performance note**: `topWeightedTokenByReferent` is called `2 û N` times per graph build even though it is called `O(Nôý)` times in the naive loop ã precompute `topWeightedTokenByReferent(agent)` for every agent into an array once, then do the O(Nôý) pair comparisons over those precomputed maps. For N = 500, this is 500 computations of `topWeighted...` plus 125,000 pair comparisons, which is trivial. Return the graph.

Then implement `largestClusterSize(agreementGraph: UndirectedGraph): number`. Use graphology's `connectedComponents` from the `graphology-components` standard-library package if it is already installed transitively by `graphology-communities-louvain` ã verify with `npm ls graphology-components` after the install in step one. If it is not installed, **do not add it as a new dependency**; instead, implement a minimal hand-rolled BFS: initialize a `Set<string>` of visited nodes, iterate over `agreementGraph.nodes()`, and for each unvisited node run a BFS that collects all reachable nodes into a component, record `component.length`, mark them visited, and track the maximum. Return the maximum (or 0 if the graph has no nodes ã defensive, should not happen for a world with agents). The BFS uses graphology's `forEachNeighbor(node, callback)` iterator to walk edges without materializing a neighbor array.

`clusterCount(agreementGraph)` reuses the same component-finding pass ã in fact, the cleaner implementation is a single private helper `findComponents(graph): number[]` that returns an array of component sizes, and then `largestClusterSize` is `Math.max(...sizes, 0)` and `clusterCount` is `sizes.filter(s => s >= 2).length`. Implement it as a single pass and call it twice per world per tick (once inside each of the two public functions) **or** refactor `computeGraphMetrics` to call `findComponents` once and derive both metrics from the single pass ã the second is strictly more efficient and is the chosen design. Export `findComponents` as a non-public helper (module-scoped, not exported from the index) so the tests can exercise it directly.

**Step five is the assimilation index.** Implement `computeAssimilationIndex(tickInteractions, l2Label)`. Filter `tickInteractions` to the subset where `success === true` **and** `(speakerClass === "W2-Immigrant" && hearerClass === "W2-Native") || (speakerClass === "W2-Native" && hearerClass === "W2-Immigrant")`. If the filtered subset is empty, return `null` immediately. Otherwise count the events where `language === l2Label` and divide by the filtered subset's length. Return the ratio as a number in `[0, 1]`. The `l2Label` parameter lets callers pass an arbitrary L2 label (per `docs/spec.md` ôÏ3.5's opaque-symbol rule); the step-20 worker will read `l2Label` from the experiment config before calling this function. Document in the doc comment that the index is symmetric in speaker/hearer role ã an Immigrant speaking L2 to a Native and a Native speaking L2 to an Immigrant both count toward the numerator, because the spec's wording "among successful interactions between W2-Immigrants and W2-Natives" is role-agnostic.

**Step six is the segregation index.** Implement `computeSegregationIndex(interactionGraph, world2, rng)`. The function must (a) build the subgraph of `interactionGraph` induced by the W2-Immigrant agent IDs, (b) handle the degenerate case where that subgraph has < 2 nodes or < 1 edge, and (c) call Louvain and return the modularity. For (a), graphology exposes `subgraph(graph, nodeSet)` in the `graphology-operators/subgraph` package (a separate npm package that may or may not be installed transitively). **Avoid adding a new dependency**: instead, construct the induced subgraph manually. Iterate `world2.agents` to collect the set of W2-Immigrant IDs; create a fresh `new UndirectedGraph()`; add each immigrant ID as a node (stringifying if the graphology graph uses string keys); iterate the cumulative `interactionGraph`'s edges via `graph.forEachEdge((edge, attrs, source, target) => { if (immigrantIds.has(source) && immigrantIds.has(target)) subgraph.addEdge(source, target, attrs); })`; return the subgraph. For (b), if `subgraph.order < 2` (fewer than two immigrants) or `subgraph.size < 1` (no inter-immigrant interactions yet), return `0` ã Louvain on a trivial graph is undefined, and `0` is the semantically correct "no segregation detected" value. Document this in the doc comment. For (c), call `louvain.detailed(subgraph, { getEdgeWeight: 'weight', rng: rng.nextFloat })` and return `result.modularity`. The `rng.nextFloat` passed as `rng` here is the step-09 seeded RNG's method ã its signature `() => number` in `[0, 1)` matches exactly what `graphology-communities-louvain` expects per its documentation.

**Step seven is the main entry point.** Implement `computeGraphMetrics(world1, world2, interactionGraph, tickInteractions)`. The function: (1) reads the L2 label from one of the worlds (the spec guarantees all worlds in an experiment use the same language labels; step 16 reads `world1.l2Label` or falls back to the step-09 default `"L2"`). If neither world carries the label directly on `World`, the function takes an additional `config: { l2Label: Language; rng: RNG; tick: number }` parameter ã adjust the signature accordingly and document that the worker (step 20) passes this in. **Preferred signature**: `computeGraphMetrics(world1, world2, interactionGraph, tickInteractions, ctx: { tick: number; l2Label: Language; rng: RNG }): GraphMetricsSnapshot`. The `ctx` object collects the ambient parameters that do not belong on the world types. (2) Build the two token-agreement graphs via `buildTokenAgreementGraph(world1)` and `buildTokenAgreementGraph(world2)`. (3) For each, call `findComponents(...)` once and derive `largestClusterSize` and `clusterCount` from the single pass. (4) Call `louvain.detailed(interactionGraph, { getEdgeWeight: 'weight', rng: ctx.rng.nextFloat })` once to get the `interactionGraphModularity` field. Handle the degenerate case where `interactionGraph.order < 2` by returning `0` before the Louvain call (same rationale as the segregation index). (5) Call `computeAssimilationIndex(tickInteractions, ctx.l2Label)` ã returns `number | null`. (6) Call `computeSegregationIndex(interactionGraph, world2, ctx.rng)` ã returns `number`. (7) Assemble the `GraphMetricsSnapshot` and return it. The function touches no I/O, no persistence, no wall clock ã pure data in, pure data out, deterministic under a fixed RNG.

**Step eight is the `computeInteractionGraphCommunities` helper.** Implement the on-demand full-detail Louvain call for step 23's network view: `function computeInteractionGraphCommunities(interactionGraph, rng)` calls `louvain.detailed(interactionGraph, { getEdgeWeight: 'weight', rng: rng.nextFloat })` and returns `{ assignments: new Map(Object.entries(result.communities)), count: result.count, modularity: result.modularity }`. The `Object.entries` conversion is because graphology's Louvain plugin returns the `communities` field as a plain object `{ nodeKey: communityId }`, while the rest of `lib/sim/` uses `Map` for keyed collections. This helper is exported from `graph.ts` alongside the main entry point and is **not** called from `computeGraphMetrics` ã it exists solely for step 23's on-click inspection and for tests that want to assert community structure beyond just the modularity scalar.

**Step nine is the tests.** Create `lib/sim/metrics/graph.test.ts` and `lib/sim/metrics/interaction-graph.test.ts` and wire them to exercise every test case listed in section 9. Each test that uses RNG pins a seed explicitly using the step-09 RNG factory and asserts determinism by running the computation twice with fresh RNG instances of the same seed. The Louvain modularity tests use **range assertions** rather than exact-value assertions, matching the Wikipedia reference's guidance that Q values vary by a few percent across implementations and initialization paths. The known-answer cases (6-node fully connected for `|Q| < 0.15`, two triangles linked by one edge for `0.30 ãÊ Q ãÊ 0.40`) are the literature-grounded anchors.

**Step ten is the topology-agnostic invariant review.** Before committing, grep the new files for any reference to `world.topology.kind`, `world.topology instanceof Lattice`, or any string literal `"lattice" | "well-mixed" | "network"` ã there should be zero hits anywhere in `lib/sim/metrics/graph.ts`, `lib/sim/metrics/interaction-graph.ts`, or either test file. This is how F4's topology-agnostic invariant stays honest: step 16's metrics must work identically on lattice, well-mixed, and network topologies, because that is the whole point of running the lattice-vs-well-mixed comparison for RQ2. Any future refactor that introduces `if (world.topology.kind === ...)` inside `lib/sim/metrics/` is a contract violation and must be rejected in review. Record this invariant as a short comment at the top of `lib/sim/metrics/graph.ts` alongside the pure-function contract.

**Step eleven is the full acceptance run.** Run `npm test -- lib/sim/metrics/graph` to execute both new test files; they must all pass. Run `npm run typecheck` (or `npx tsc --noEmit`) to confirm the new types compose cleanly with the existing `lib/sim/metrics/types.ts` exports and the step-13 `InteractionEvent` import. Run `npm test` to confirm no regressions in any other step's tests (steps 10, 11, 12, 13, 14, 15). Only then commit.

## 8. Library choices

- **`graphology-communities-louvain`** ã the canonical Louvain community-detection plugin for graphology, pinned to the latest stable release on npm at execution time (verified via `npm view graphology-communities-louvain version` during the step run; at the time of writing this plan file, that is **`2.0.2`**, but the agent must re-verify). Install with `npm install graphology-communities-louvain@<version>`; no separate `@types/...` package is needed because TypeScript declarations ship inside the main package. The plugin declares `graphology` as a peer dependency, which is already satisfied by the step-10 install. The plugin's default export is the `louvain(graph, options?)` function, and the load-bearing named export is `louvain.detailed(graph, options?)`, which returns `{ communities, count, modularity, dendrogram, deltaComputations, moves, nodesVisited }` ã step 16 reads `.modularity` for both the interaction-graph modularity scalar and the segregation index, and `.communities` + `.count` for step 23's network view (via the `computeInteractionGraphCommunities` helper).
- **`graphology`** ã already installed by step 10; step 16 does not reinstall. Step 16 uses `UndirectedGraph`, `Graph`, `mergeNode`, `updateEdge`, `forEachEdge`, `forEachNeighbor`, `order`, `size`, `nodes()`, `edges()`, and `addEdge` ã all standard graphology API surface.
- **No new test library** ã Vitest is already installed (step 00) and the default `node` environment handles pure-function tests. No `happy-dom`, no mocking libraries.
- **No `graphology-components` or `graphology-operators`** ã these are sibling plugins in the graphology standard library that would provide `connectedComponents(graph)` and `subgraph(graph, nodes)` respectively, but step 16 hand-rolls both (BFS for components, edge-filter loop for the induced subgraph) to keep the dependency footprint minimal. Adding them is a legitimate v2 refactor if the hand-rolled implementations show performance problems at N = 10ãÇ, but for v1's N ãÊ 500 target a few dozen lines of straightforward TypeScript is cheaper than a new npm package and a new `package.json` line to review.

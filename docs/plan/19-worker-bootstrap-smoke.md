## 1. Goal

De-risk the entire worker integration track **in isolation**, before any simulation code crosses the Comlink wire. This step stands up the smallest possible end-to-end Web Worker built with the Turbopack-native `new Worker(new URL('./hello.worker.ts', import.meta.url), { type: 'module' })` pattern, wraps it in a Comlink proxy, calls three trivial methods from a Client Component, and verifies the round trip under `next build && next start` via the chrome-devtools MCP harness. The point is **not** to ship functionality — the smoke page is deleted in step 20. The point is to catch Turbopack worker bundling idiosyncrasies **now**, with a tiny diff and a minimal reproduction, rather than during step 20 when ~10 simulation modules will be wired through the same plumbing and a failure would be opaque.

Step 19 is the single most important de-risking commitment in the entire pipeline. `docs/spec.md` §8 and §12.4 cite Park.is' Next 15 + webpack-worker-loader blog post as the canonical pattern; **that citation is obsolete for Next 16**. Next 16 uses Turbopack by default for both `next dev` and `next build`, and a custom `webpack` config in `next.config.ts` **fails the build outright** (`node_modules/next/dist/docs/01-app/02-guides/upgrading/version-16.md` line 142 is explicit about this: _"the build will fail to prevent misconfiguration issues"_). The only supported worker construction under Turbopack is the web-standard `new Worker(new URL(...), { type: 'module' })` idiom, which is the same pattern MDN, webpack 5, Vite, and Parcel have converged on. This step confirms that convergence holds on the installed `next@16.2.2` toolchain and that the `lib/sim/` modules (reused in the worker as a canary) bundle cleanly through Turbopack's unified graph into a Web Worker chunk.

Secondary goals, all load-bearing: (a) prove that an ESM simulation module (`lib/sim/rng` from step 09) imported from a worker module resolves correctly; (b) prove that a CJS `node_modules` dep (`graphology` from step 10) also resolves into a worker module, or document the attempt and its result if the CJS path is hard; (c) verify that the seeded RNG produces bit-identical outputs when invoked twice inside the worker with the same seed — a pre-test of the determinism invariant that step 20 will turn into a hard test against the full simulation; (d) verify that React 19 strict mode's double-invoked `useEffect` does not leak workers when the effect's cleanup callback is implemented idempotently, which is the exact pitfall `CLAUDE.md` "Worker lifecycle" already names and which step 20 will inherit.

Local Next.js 16 docs (authoritative per `CLAUDE.md` "Next.js 16 deltas from training data" and `AGENTS.md`):

The doc does **not** give a full code example of the `new Worker(new URL('./x.worker.ts', import.meta.url), { type: 'module' })` form, but the MDN external reference (below) does, and the two fit together cleanly — Turbopack implements the same bundler contract MDN names as "recommended by major bundlers." This step's worker construction uses that exact form.

- **`node_modules/next/dist/docs/01-app/02-guides/upgrading/version-16.md` § Turbopack by default (lines 114-166).** _"Starting with Next.js 16, Turbopack is stable and used by default with `next dev` and `next build`. - **`node_modules/next/dist/docs/01-app/02-guides/lazy-loading.md` § Magic Comments (lines 175-215).** Independent confirmation of the `new Worker()`-as-bundler-expression contract: _"Next.js supports magic comments to control how dynamic imports are handled by the bundler. These comments work with dynamic `import()`, `require()`, `require.resolve()`, and `new Worker()` expressions."_ The § Good to know note clarifies that _"Magic comments do not work with static `import` statements... They only work with dynamic expressions"_ — which is why the worker's URL argument is constructed via `new URL(..., import.meta.url)` at call time rather than imported as a static string literal. The magic-comment machinery (`turbopackIgnore`, `turbopackOptional`, `webpackIgnore`) is **not** used in this step; we want Turbopack to bundle the worker normally and emit a separate chunk. The doc confirms that default Turbopack handling of `new Worker(new URL(...))` is to bundle the referenced module as a worker chunk and rewrite the URL to the built artifact — the same behavior webpack 5, Vite, and Parcel converged on.

- **`node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/proxy.md` §§ Exports and Execution order (cited in step 06 §4).** Re-read here only to confirm one fact: the proxy allowlist change in this step (`/_dev/*` for the duration of step 19, removed in step 20) does not require any new proxy export or config option — only a single-line edit to the in-handler `PUBLIC_PATHS` array (or equivalent helper) that step 06 shipped. - **`node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md` §§ When to use Server and Client Components.** Confirms that `'use client'` boundaries are required for any component that uses `useEffect`, `useState`, event handlers, or the `Worker` global. The smoke page at `app/(public)/_dev/worker-smoke/page.tsx` is therefore a Client Component (`'use client'` directive on line 1). External references (WebFetched during research):

- `https://github.com/GoogleChromeLabs/comlink` (WebFetched). Confirmed the canonical Comlink pattern is the two-line shape `Comlink.expose(obj)` inside the worker and `Comlink.wrap(worker)` on the main thread. Installation: `npm install --save comlink`. TypeScript pattern: _"When exposing a value of type `T`, the corresponding `wrap()` call returns `Comlink.Remote<T>`, providing type safety for remote method calls."_ Key quoted invariant: _"If you are using the proxy, put `await` in front of it"_ — every method call on a wrapped remote returns a `Promise<T>`, so the smoke page's `useEffect` awaits each call sequentially. Bundle size: ~1.1 kB brotli. Browser support: Chrome 56+, Firefox 52+, Safari 10.1+ — well within the Next 16 browser baseline (`version-16.md` line 112: Chrome 111+, Edge 111+, Firefox 111+, Safari 16.4+). Current stable version on npm: **`comlink@4.4.2`** (verified via `npm view comlink version` — the agent re-runs this at step execution time and pins whatever `latest` resolves to; the 4.x line has been stable for years and a patch bump within 4.x is expected, a major bump is not). The Comlink README does not document a specific `new Worker(new URL(...))` example because Comlink is worker-construction-agnostic — it accepts any `Worker` instance as the argument to `wrap()`, so swapping from webpack-loader to Turbopack-native is transparent to Comlink itself. This is why the Park.is blog post's obsolescence is about the worker-construction half of the integration, not about Comlink.

- `https://developer.mozilla.org/en-US/docs/Web/API/Worker/Worker` (WebFetched). Authoritative reference for the `new Worker(new URL(...))` idiom. Quoted canonical syntax: _"`const myWorker = new Worker(new URL("worker.js", import.meta.url));`"_ and with module type: _"`const myWorker = new Worker(new URL("worker.js", import.meta.url), { type: "module" });`"_ The MDN "Bundler considerations" sub-section states: _"Bundlers such as webpack, Vite, and Parcel, recommend passing URLs that are relative to `import.meta.url` to the `Worker()` constructor."_ This confirms the pattern is the **unified industry standard**, not a Next-specific or Turbopack-specific quirk. The two benefits MDN cites are: (1) _"Makes the path relative to the current script instead of the HTML page"_ and (2) _"Allows bundlers to safely perform optimizations like renaming, since the bundler can make assumptions about what files it controls."_ Both are exactly why Turbopack's § Magic Comments section treats `new Worker()` as a recognized expression form. The MDN page does not mention Turbopack by name (Turbopack is newer than the MDN reference text), but the webpack 5 pattern MDN documents is the one Turbopack implements by design — the bundler contract is the same.

Paths not taken:

- **Webpack worker-loader via `next.config.ts` `webpack()` option.** This is the pattern the Park.is blog post cited in `docs/spec.md` §8/§12.4 describes, and it is the default answer in most Next 14/15 tutorials. **Rejected — it would fail `next build` immediately on Next 16.** `node_modules/next/dist/docs/01-app/02-guides/upgrading/version-16.md` line 142: _"If your project has a custom `webpack` configuration and you run `next build` (which now uses Turbopack by default), the build will fail to prevent misconfiguration issues."_ There is no webpack-loader escape hatch that does not involve either (a) switching the whole build to `next build --webpack` (gives up every Turbopack performance and bundling benefit for one feature, and the CLAUDE.md "Stack and versions" commitment explicitly names Turbopack as the default), or (b) adding the `--turbopack` flag to restore the Next 15 opt-in behavior (but the webpack config is then ignored, making the worker-loader route a no-op anyway). There is no fallback that keeps the Park.is pattern working under Next 16 without a major architectural detour, and the Turbopack-native `new Worker(new URL(...))` pattern fully replaces its functionality with less code. Rejection rationale recorded here and cross-referenced in the `CLAUDE.md` "Next.js 16 deltas" entry that already names this obsolescence.

- **`next/dynamic` with `ssr: false` to wrap a component that constructs the worker at module top-level.** Considered as a shortcut to avoid the `useEffect` idempotent-cleanup dance. Rejected for two reasons. First, constructing the worker at module-scope (outside any effect) still executes on the first client render and **still** gets double-invoked under React 19 strict mode's `<StrictMode>` wrapping — strict mode re-mounts components, not just effects, so a module-scope side effect inside a component body runs twice as well. The only difference is that a module-scope side effect has nowhere to put a cleanup callback, so the second worker instance leaks for the lifetime of the page. Second, `next/dynamic` with `ssr: false` adds a client-only chunk and a loading state that this step does not need — the smoke page is already a Client Component with `'use client'`, so SSR is already disabled for the worker-hosting component. Using `useEffect` with an idempotent cleanup is both simpler and better-matched to the React 19 semantics documented in `CLAUDE.md` "Worker lifecycle". Step 20 will follow the same pattern for the real simulation worker.

- **Skipping the CJS-dependency canary (not importing `graphology` from inside the worker).** Considered because worker-graph CJS interop is a known sharp edge in some bundlers. Rejected because the whole point of step 19 is to surface that sharp edge **now** rather than during step 20 when ~10 modules are being wired at once. The worker imports `graphology` via `import Graph from 'graphology'` (same form step 10's `lib/sim/topology/network.ts` uses), calls `new Graph()` once, and reads the resulting `graph.order` (which is `0` for a fresh graph). If Turbopack cannot resolve `graphology` from the worker module graph, the build fails at this step and the pipeline stops — which is exactly the right outcome because step 20 would also have failed and the debugging surface is smaller here. If the build succeeds but the import throws at runtime (e.g., because graphology's CJS entrypoint probes for `module`/`exports` in a way the worker scope does not provide), the MCP script's console triage catches the thrown error and fails the step loudly. If the import is impossible in practice (say, because graphology's shipped artifact has a `process.env` reference that the worker scope rejects), the implementing agent must **document the attempt and its result** in a comment at the top of `workers/hello.worker.ts` and remove the import — but this must be treated as a finding, not a silent workaround. The ESM canary (`lib/sim/rng`) stays regardless because it is pure TypeScript with no runtime dep quirks and is guaranteed to work.

- **Placing the smoke page under `/public/_dev/worker-smoke/page.tsx` (making `public` a real URL segment rather than a route group).** Considered as an alternative to adding `/_dev/*` to the proxy allowlist. Rejected because it would require either (a) creating a new non-route-group directory that clashes with the existing `public/` static-asset folder at the repository root, which is a Next.js reserved name and cannot be used as a URL segment, or (b) placing the page at `app/public-dev/worker-smoke/page.tsx` with a `public-dev` prefix, which pollutes the URL namespace with a name that means nothing to the user and has to be renamed/removed in step 20 anyway. Extending the proxy allowlist with a temporary `/_dev/*` entry is a one-line change that step 20's plan file already knows how to reverse, leaves the route-group convention (`(public)`) untouched, and means the smoke page URL is the clean and meaningful `/_dev/worker-smoke`. The `_dev` prefix is a deliberate convention borrowed from Rails' `app/dev` and Django's `development/` conventions: a URL segment starting with `_` signals "developer-only, not for production end users," which is exactly the semantic step 19 wants.

## 5. Files to create

- **`workers/hello.worker.ts`** — the worker module itself. A flat TypeScript file, no `'use client'` directive and no `import 'server-only'` directive (it is neither a client nor a server module — it is a worker module, a third environment). Imports: `import * as Comlink from 'comlink'`, `import { createRNG } from '@/lib/sim/rng'` (ESM-only simulation canary from step 09), and `import Graph from 'graphology'` (CJS-shaped node_modules canary from step 10; if the import fails during `next build`, document the failure and remove the import — see §4 path-not-taken). Defines and exports the `HelloWorkerApi` type: `{ ping(): string; add(a: number, b: number): number; echo<T>(value: T): T; determinismCheck(seed: number, count: number): boolean }`. Defines `const api: HelloWorkerApi = { ping: () => 'pong', add: (a, b) => a + b, echo: <T,>(v: T) => v, determinismCheck: (seed, count) => { /* see §7 */ } }`. Calls `Comlink.expose(api)` at module top level. A header comment block cites `CLAUDE.md` "Worker lifecycle", `node_modules/next/dist/docs/01-app/03-api-reference/08-turbopack.md` § Magic Comments (line 129), and the MDN `Worker()` constructor reference, and notes that the file will be deleted in step 20.

- **`app/(public)/_dev/worker-smoke/page.tsx`** — the temporary smoke page. Client Component with `'use client'` on line 1. Imports: `import { useEffect, useState } from 'react'` and `import * as Comlink from 'comlink'` and `import type { HelloWorkerApi } from '@/workers/hello.worker'` (a **type-only** import — `import type`, not a runtime import — so the worker module is not pulled into the page's client bundle graph; the worker module is instantiated only via `new Worker(new URL(...))` at runtime, which is the bundler contract Turbopack understands). Component function body uses `useState` to track four strings (`status`, `pingResult`, `addResult`, `echoResult`) and one boolean (`determinismResult`). `useEffect(() => { ... }, [])` runs the worker bootstrap: constructs the worker with `const worker = new Worker(new URL('../../../../workers/hello.worker.ts', import.meta.url), { type: 'module' })`, wraps it with `const api = Comlink.wrap<HelloWorkerApi>(worker)`, then in an async IIFE awaits the four method calls sequentially and `setState`s each result, setting `status = 'ready'` last. The effect returns a cleanup callback that calls `worker.terminate()` — **this is the idempotent cleanup** that protects against React 19 strict mode's double-invocation. The JSX renders a small DOM tree with elements bearing the five stable `data-testid` attributes listed in §10. A top-of-file comment notes the file will be deleted in step 20. **Subtlety on the URL path**: `import.meta.url` resolves relative to the current module, so from `app/(public)/_dev/worker-smoke/page.tsx` the relative path to `workers/hello.worker.ts` is `../../../../workers/hello.worker.ts` (four levels up: out of `worker-smoke/`, out of `_dev/`, out of `(public)/`, out of `app/`, then into `workers/`). **Verify this path manually** with `ls` before committing — an off-by-one in the relative path is the most common failure mode for new-to-Turbopack worker setups, and the error message Turbopack produces ("Module not found") is not immediately obvious as a URL-resolution bug. If the relative path calculation is fragile, a cleaner alternative is to use the `@/` alias inside the `new URL()` call — but `new URL('@/workers/hello.worker.ts', import.meta.url)` does **not** work because the alias is a TypeScript-path-mapping concern, not a module-resolver concern, and `new URL()` is a runtime browser API that does not know about tsconfig paths. **Use the relative path and verify it by running `next build` first.**

- **(Optional) a thin client-side wrapper component `app/(public)/_dev/worker-smoke/WorkerSmoke.tsx`.** Only if the inline component in `page.tsx` becomes unwieldy — which it should not at this scale. The plan's default is to keep everything in `page.tsx` for maximum locality; if the implementing agent finds the JSX + the effect logic hard to read together, split them, but this is a judgment call and not a hard requirement. If split, the wrapper is a plain Client Component exported from `WorkerSmoke.tsx` and `page.tsx` becomes a two-line re-export.

## 7. Implementation approach

This section is prose. The order matters because each slice depends on the previous one being correct, and the failure modes compound — a broken worker bundle that only manifests at runtime is harder to debug than a broken build that fails immediately. The sequencing is designed to move failures as far left as possible.

**Slice one — extend the proxy allowlist.** This is the smallest and highest-leverage change, and it has to land first because without it the MCP script cannot reach the smoke page. Open `proxy.ts` at the repository root. Locate the `PUBLIC_PATHS` array (from step 06 §7 slice three). Add a single new entry `'/_dev'` with an adjacent `// TEMPORARY:` comment explaining that step 20 removes it. Do not touch anything else in `proxy.ts`; in particular, do not touch the `config.matcher` regex. Verify with `npm run typecheck` that the file still compiles, then run `npm run build` to confirm `next build` still produces a clean build and the routes summary lists the `(public)` group. There is no unit test for the proxy allowlist (step 06 §9 declines to write one, and step 19 does not revisit that decision); the MCP script's Phase F network triage is the integration test.

**Slice two — install Comlink.** Run `npm view comlink version` at the command line to capture the current stable release (expected `4.4.2` or a later 4.x patch). Run `npm install comlink@<resolved-version>` to add it. Verify `package.json` now lists `comlink` under `dependencies` with a `^4.x` range. Verify `node_modules/comlink/package.json` exists and its `"types"` or `"typings"` field points at a `.d.ts` file — this is how Comlink's TypeScript support works without a `@types/comlink` package. Run `npx tsc --noEmit` as a sanity check; it should exit cleanly because no code imports Comlink yet. If the install emits any peer-dependency warnings, record them in the step log but do not act on them — Comlink 4.x has no peer dependencies.

**Slice three — write the worker module (`workers/hello.worker.ts`).** Create the directory `workers/` at the repository root (it does not exist yet — step 19 is the first step to populate it; `CLAUDE.md` "Directory layout" already lists `workers/` as reserved for "Web Worker entrypoints (steps 19-20)"). Write the top-of-file comment block citing `CLAUDE.md` "Worker lifecycle", the Turbopack doc line and the MDN `Worker()` reference, and noting the file will be deleted in step 20. Import `* as Comlink from 'comlink'`, `createRNG` from `@/lib/sim/rng` (the step-09 ESM canary), and `Graph` from `graphology` (the step-10 CJS canary — use `import Graph from 'graphology'` following the same shape `lib/sim/topology/network.ts` uses). The type-only import is load-bearing: if you write `import { HelloWorkerApi } from ...` (without `type`), Turbopack will try to pull the worker module into the client bundle graph, which defeats the purpose of code-splitting a worker. The `type`-only form is erased at compile time and leaves only the `new Worker(new URL(...))` call as the runtime reference to the worker module, which is exactly what Turbopack's bundler contract expects.

Inside the page function body, declare five `useState` hooks: `const [status, setStatus] = useState<'idle' | 'loading' | 'ready' | 'error'>('idle')`, `const [pingResult, setPingResult] = useState<string>('')`, `const [addResult, setAddResult] = useState<string>('')`, `const [echoResult, setEchoResult] = useState<string>('')`, `const [determinismResult, setDeterminismResult] = useState<string>('')`. Then the `useEffect`:

```
useEffect(() => {
  setStatus('loading');
  const worker = new Worker(
    new URL('../../../../workers/hello.worker.ts', import.meta.url),
    { type: 'module' }
  );
  const api = Comlink.wrap<HelloWorkerApi>(worker);
  let cancelled = false;
  (async () => {
    try {
      const ping = await api.ping();
      const add = await api.add(2, 3);
      const echo = await api.echo({ hello: 'world' });
      const det = await api.determinismCheck(42, 100);
      if (cancelled) return;
      setPingResult(ping);
      setAddResult(String(add));
      setEchoResult(JSON.stringify(echo));
      setDeterminismResult(String(det));
      setStatus('ready');
    } catch (err) {
      if (cancelled) return;
      setStatus('error');
      console.error('worker smoke error:', err);
    }
  })();
  return () => {
    cancelled = true;
    worker.terminate();
  };
}, []);
```

The `cancelled` flag + cleanup-terminate pattern is the idempotent-effect shape `CLAUDE.md` "Worker lifecycle" names. React 19 strict mode double-invokes effects in development, so the effect body runs twice; the first invocation's cleanup fires before the second invocation starts, meaning the first worker is terminated and cleaned up deterministically. The `cancelled` flag prevents state updates from a terminated worker's in-flight promises from landing after the component has begun its second mount. Without this flag, the second effect run would see stale `setState` calls from the first run and either throw a "setState on unmounted component" warning (React 18) or a "setState on a stale render" warning (React 19). The empty `[]` dependency array means the effect runs once per mount; the cleanup runs once per unmount. In strict-mode dev, that's two mounts → two effect runs → two worker instances, but the first is terminated in its own cleanup before the second starts, so only one is ever live.

## 11. CLAUDE.md updates

Append ≤ 25 lines to the "Worker lifecycle" section (40-line cap per the existing header comment). The section was pre-populated with a six-bullet outline; step 19 fills in the first layer of concrete guidance. Do not edit any other section. Do not exceed 25 appended lines.

- The Turbopack-native worker construction pattern is `new Worker(new URL('./name.worker.ts', import.meta.url), { type: 'module' })`. The URL argument **must** be a `new URL(..., import.meta.url)` expression, not a string literal; Turbopack recognizes the expression form as a bundler directive (per `node_modules/next/dist/docs/01-app/03-api-reference/08-turbopack.md` § Magic Comments) and emits a separate worker chunk. The `type` qualifier is load-bearing — a runtime import would pull the worker module into the main bundle graph and defeat the code-split.

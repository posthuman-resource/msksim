## 1. Goal

Ship the **F10 interactive controls** layer of the live playground: a client-side `ControlsPanel` component that wires the Playground page's existing simulation shell (from steps 21-23) to the full set of researcher-facing controls defined in `docs/spec.md` §4.2 / F10. Concretely, this step delivers (a) transport controls — **Play**, **Pause**, **Step**, **Reset** — that drive the shell's tick loop deterministically; (b) a **variable tick-rate selector** with presets 1×, 10×, 100×, 1000× that governs how many `api.step(count)` invocations happen per `requestAnimationFrame` tick; (c) an **RNG seed input** with a "Reseed and reset" button that validates non-negative integers (including the explicit `0` case required by F10's acceptance criteria) and re-bootstraps the simulation via `api.init(config, seed)`; (d) **debounced sliders** for the five "hot" runtime parameters from §3.3 and F10 (monolingual:bilingual ratio, interaction probability, Δ⁺, Δ⁻, preferential-attachment strength/temperature), using a 300ms `setTimeout`-based debounce because — per the React docs WebFetched in section 4 — `useDeferredValue` is explicitly **not** suitable for debouncing side effects like worker `postMessage` calls (it only defers rendering); (e) a **Reset UX** that distinguishes parameters applied live on the next tick (Δ⁺, Δ⁻, interaction probability) from parameters that require a rebootstrap (ratio, preferential-attachment strength, seed) — the latter surface an inline warning before the change lands, per the step-specific context's "Reset UX" bullet; (f) a **reproducibility banner** showing the current seed and a short SHA-256 hash of the canonicalized config (computed via `crypto.subtle.digest` in the `config-hash.ts` helper, and reused later by step 30 for export filenames per CLAUDE.md "Export conventions"); and (g) a worker API extension — a new `updateConfig(partial)` method on `SimulationWorkerApi` that merges hot parameters into the worker's in-memory `ExperimentConfig` so the engine picks them up on the next tick without losing the time-series history. The step also wires the controls into the `SimulationShell` component created in step 21 so that Play/Pause drive the existing render loop, Reset clears the metrics dashboard's history (step 22) and the cumulative interaction graph underlying the network view (step 23), and debounced slider updates flow through the new `updateConfig` path. Scope boundary is strict: this step **does not** add new metrics, does not persist runs to SQLite (that is step 26), does not expose a config editor form (that is step 25), and does not introduce batch queueing (step 27). The deliverable is a set of interactive controls that make the single-run live playground feel like a research instrument the two researchers can drive by feel — Play to watch, Pause to inspect, Step to single-trace, Reset to try again, sliders to vary the knobs the paper's hypotheses depend on.

## 3. Spec references

- `docs/spec.md` **§4.2 Live playground mode → F10. Interactive controls.** This is the literal feature step 24 implements. The spec text: *"Start, pause, step, reset, variable tick-rate (1× through 1000×), RNG seed input, and sliders for the 'hot' runtime parameters (monolingual:bilingual ratio, interaction probability, Δ⁺, Δ⁻, preferential-attachment strength)."* Every control listed in §4.2 appears in the `ControlsPanel` component. The acceptance clause — *"The simulation is reproducible given the same seed and config; sliders are debounced so they do not thrash the engine; a seed of 0 is explicitly supported."* — drives three concrete requirements step 24 honors: (1) the reproducibility banner (seed + config hash) in the UI; (2) the 300ms debounce on sliders, verified by MCP step 7 in section 10; (3) the seed input's validator accepts `0` as a distinct valid value, not treated as "missing" — this is a known foot-gun in naive input validation where `!seed` treats `0` as falsy. Section 7 documents the validator's explicit `Number.isInteger(value) && value >= 0` check that distinguishes `0` (valid) from `-1`, `NaN`, `null`, `undefined` (invalid).

- `docs/spec.md` **§3.3 Interaction rules → "hot" parameters that vary per run.** Lists the five parameters F10 exposes as sliders: Δ⁺ (weight increment on success), Δ⁻ (optional weight decrement on failure), interaction probability (governs whether an activated agent attempts any interaction this tick), monolingual:bilingual ratio (governs world bootstrapping — this is **not** a live-update parameter because it changes initial conditions, so step 24 flags it as reset-required in the Reset UX), and preferential-attachment strength (the temperature of the softmax over cosine similarity from F6 / step 14; live-updatable because it only affects partner selection going forward). Section 7's slider list maps each parameter to its applied-live or requires-reset classification and documents the rationale for each. The classification is **load-bearing** because misclassifying a live parameter as reset-required would wipe the metric history unnecessarily, and misclassifying a reset-required parameter as live would silently desynchronize the simulation from the researcher's mental model (e.g., changing the ratio mid-run would not actually change the agent population, but would change the stored config used for persistence and export, producing a run that claims to have been run at a ratio it was never run at — a reproducibility landmine).

- `docs/spec.md` **§8 Architecture Sketch → "Key architectural commitments".** The third bullet: *"Simulation code is pure and deterministic given seed + config. No Math.random(); the engine takes a seedable RNG. No wall-clock-dependent logic."* Step 24 is the first step that lets a researcher vary the seed interactively, and the reproducibility commitment means the seed + config must fully determine the run. The reproducibility banner in section 7 makes this explicit to the user: the current seed is displayed as a number, the current config is displayed as the first 8 hex characters of its SHA-256 hash, and together they form the identifier a researcher can copy into a lab notebook or a shared message to reproduce the run exactly. The fourth bullet: *"Rendering is decoupled from simulation: the worker emits metrics and snapshots at the sampling interval; the main thread chooses when to redraw."* Step 24's Play loop calls `api.step(tickRate)` at 60Hz (once per `requestAnimationFrame`) and consumes whatever `TickReport` the worker returns — the main thread never tries to push work into the worker or mutate its state from outside the Comlink API surface.

- `docs/spec.md` **§4.2 Live playground mode → F7 (live lattice view), F8 (live network view), F9 (live metrics dashboard).** Step 24 cuts across all three views: Play starts the shared tick loop that feeds all three; Pause stops it; Step advances them all by one tick in lockstep; Reset clears all three. The cross-cutting nature means step 24 is the first step that touches the `SimulationShell` state surface in a non-trivial way — steps 21-23 each added their own consumer of the shell's state, but step 24 is the first to **mutate** that state from outside the shell (via the controls panel). Section 7's state-lift decision lives here: the tick loop timer, the `isRunning` flag, the `tickRate` value, the seed, and the current config all live in the `SimulationShell`'s top-level state. The `ControlsPanel` receives props for reading current values and callback props for requesting state changes. This keeps the shell authoritative and the controls stateless (modulo the debounced slider drafts that only exist while the user drags). CLAUDE.md's "UI verification harness" section demands that MCP scripts drive real user interactions; sections 10's script exercises each control end-to-end.

- `docs/spec.md` **§8 Reproducibility architectural commitment (reiterated).** The spec devotes an entire bullet to this: every run must be reproducible from its `(seed, config)` pair. Step 24 operationalizes this by (a) making both fields user-visible in the reproducibility banner, (b) hashing the config so researchers can compare two runs at a glance without diffing the full JSON, and (c) using the config hash as the filename stem for exports per CLAUDE.md "Export conventions" (`msksim-<config-hash>-seed-<seed>-<kind>.csv`). The `configHash` helper lives in `app/(auth)/playground/config-hash.ts` with a unit test that asserts a known-answer SHA-256 hash for a canonical fixture config — this is the step's only new unit test, documented in section 9. The commitment is load-bearing because steps 26 (run persistence), 29 (run comparison), and 30 (export) all reuse the config hash as a stable identifier; a bug in the hashing logic (e.g., non-canonical key ordering producing different hashes for equivalent configs) would propagate into every downstream surface. Section 9's known-answer test guards against this regression.

## 4. Research notes

**Local Next.js 16 docs (authoritative per `CLAUDE.md` "Next.js 16 deltas from training data" and `AGENTS.md`):**

1. `node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md` §"Using Client Components" and §"Reducing JS bundle size" — confirms that `'use client'` at the top of a module is the canonical way to declare a Client Component boundary, that once a file is marked `'use client'` **all its imports and child components are part of the client bundle** (so importing `ControlsPanel` into a Server Component page is fine — the `'use client'` boundary is in the controls file itself), and that the best practice is to scope `'use client'` to the **smallest interactive leaf** rather than the whole page. Step 24 follows the best practice: `app/(auth)/playground/page.tsx` remains a Server Component (it just calls `verifySession()` from `lib/auth/dal.ts` per CLAUDE.md "Authentication patterns", then renders `<SimulationShell />`), the `SimulationShell` from step 21 is already a Client Component (it owns the worker lifecycle in `useEffect`), and the new `ControlsPanel` is a separate Client Component rendered as a child of `SimulationShell`. The `ControlsPanel` file has `'use client'` at the top; `page.tsx` does **not**. The doc also confirms that React hooks (`useState`, `useEffect`, `useRef`) only work inside Client Components, which is why the debounce state and the slider draft state must live inside `ControlsPanel`, not in the page.

2. `node_modules/next/dist/docs/01-app/02-guides/forms.md` — documents Next 16's recommended form patterns. Load-bearing for the seed input: the seed is a **number** field with the constraint that `0` is valid but negative numbers and non-integers are not. The doc shows that HTML5 `<input type="number" min="0" step="1" />` is the declarative approach, but client-side validation on the button click is still required because `type="number"` does not reject `-5` at submit time (browsers accept and normalize out-of-range values inconsistently). Step 24's seed input uses `type="number"` for the native spinner affordance plus an imperative `Number.isInteger(value) && value >= 0` guard inside the "Reseed and reset" button handler. If the guard fails, the button displays an inline error message and does not call `api.init()`. The doc also confirms that form submissions inside Client Components do **not** need to be Server Actions — plain `onClick` handlers are fine for UI-only state transitions like this one, where no server-side mutation is intended.

3. `node_modules/next/dist/docs/01-app/03-api-reference/08-turbopack.md` §"Language features" and §"Module resolution" — confirms that Turbopack's tsconfig path aliases (`@/` pointing to the repo root) resolve inside Client Components the same way they resolve inside Server Components and workers. Load-bearing because `ControlsPanel` imports types from `@/lib/sim/worker-client` (the re-exported `SimulationWorkerApi`, `TickReport`, and the new `ExperimentConfigPartial` type step 24 adds) and imports the `ExperimentConfig` Zod schema from `@/lib/schema/experiment` (step 01). The doc reminds that type-only imports (`import type { ... }`) are erased at build time, so importing the worker's type surface into the controls component does not pull the worker's runtime code into the main-thread bundle — this is the same invariant step 20 relied on in `lib/sim/worker-client.ts`, now reused at a second call site.

**External references (WebFetched):**

4. **React 19 `useDeferredValue` documentation** — `https://react.dev/reference/react/useDeferredValue` (WebFetched during research). Load-bearing facts confirmed by the fetch:
   - `useDeferredValue(value, initialValue?)` returns a value that **lags behind** the current value, with the delay chosen adaptively by React's scheduler based on device speed. There is no fixed millisecond delay — it is **not** a debounce in the setTimeout sense.
   - **`useDeferredValue` does NOT prevent side effects.** The React docs are explicit: *"`useDeferredValue` does not by itself prevent extra network requests."* It only defers **rendering**, not effects, event handlers, or async side effects. This is the decisive finding for step 24: the goal of slider debouncing is to avoid calling `api.updateConfig()` on every keystroke — that call is a `postMessage` side effect, not a render. `useDeferredValue` would not help. A traditional `setTimeout`-based debounce is required.
   - `useTransition` also does not help for this specific use case: it marks state updates as low-priority but does not delay them, and side effects inside the transition still run immediately.
   - The correct pattern for debouncing an async side effect on a controlled input is: keep the input's value in a `useState` (the **draft** value, updated on every keystroke/drag), and schedule a `setTimeout` inside a `useEffect` that clears the previous timer and fires the side effect after the delay. The `useEffect` cleanup function clears the timer on unmount and on every re-render, so the side effect only runs when the draft value has been stable for the full delay. Section 7 documents this pattern in detail and section 9's optional test covers it.
   - The React docs' recommendation for debouncing actual async side effects is: *"use separate techniques (debounce/throttle libraries, useEffect with delays, or useTransition with async transitions)."* Step 24 uses the middle option (useEffect with delays) because the first option would add a dependency (lodash.debounce) for a trivial piece of logic, and the third option requires the async work to be inside a transition callback which would then also need to be `Comlink.proxy`-wrapped — needless complexity for a 300ms timer.

5. **MDN `SubtleCrypto.digest` documentation** — `https://developer.mozilla.org/en-US/docs/Web/API/SubtleCrypto/digest` (WebFetched during research). Load-bearing facts:
   - `crypto.subtle.digest(algorithm, data)` takes an algorithm string (`"SHA-256"` for step 24) and a `BufferSource` (`ArrayBuffer`, `TypedArray`, or `DataView`) and returns `Promise<ArrayBuffer>`. The `ArrayBuffer` must be converted to hex via `Array.from(new Uint8Array(buffer), b => b.toString(16).padStart(2, '0')).join('')` because the newer `Uint8Array.prototype.toHex()` method is not yet shipping in all target browsers as of the April 2026 stability date. Section 7 uses the `padStart` approach for maximum compatibility.
   - The API is available in **Main thread, Web Workers, and Node.js**. Step 24 calls it only from the main thread (inside `ControlsPanel`), but the same helper works unchanged if step 20's worker ever needs to self-hash its config — a future-proofing property worth noting here so step 26 (run persistence) can reuse the helper without refactoring.
   - **Secure context required**: `crypto.subtle` is only available in HTTPS contexts (or `localhost` exceptions). Next's `next start` listens on `http://localhost:$PORT`, which qualifies for the localhost exception, so MCP verification passes without HTTPS. Production deployments must be behind HTTPS for this to work — documented as a known gotcha in section 11's CLAUDE.md append.
   - **SHA-256 always produces 256-bit (32-byte) output**, regardless of input size. Step 24 truncates to the first 8 hex characters (4 bytes, ~32 bits) for display in the reproducibility banner — enough to distinguish configs in a researcher's head, short enough to fit in a banner. The full 64-character hex is still available via the helper's return value for export filenames per CLAUDE.md "Export conventions".
   - The **canonicalization problem**: two configs that are semantically identical but have keys in different orders produce **different** SHA-256 hashes because JSON serialization is order-sensitive. Step 24's `config-hash.ts` helper must sort object keys before serializing. Section 7 documents the canonical-stringify algorithm (recursive key sort), and section 9's known-answer test verifies that `{ a: 1, b: 2 }` and `{ b: 2, a: 1 }` produce the same hash.

**Paths not taken:**

6. **External debounce library (lodash.debounce, use-debounce, etc.).** Rejected. `lodash.debounce` would add ~2KB to the client bundle and an extra dependency for a feature that is ~15 lines of idiomatic React (`useState` + `useEffect` + `setTimeout` + cleanup). `use-debounce` is a React-specific wrapper that offers a `useDebouncedCallback` hook, but its API surface is thin enough that replicating it in-tree is trivial, and adding a third-party hook library would couple step 24 to its maintenance timeline. The React docs (WebFetched in item 4) explicitly list "useEffect with delays" as a valid pattern. Per CLAUDE.md "Library choices" discipline, every dependency must earn its place; a 15-line utility does not earn it. This is the "≥ 1 path not taken" required by the research quality gates. **Recorded here as an explicit rejection** so a future agent does not revisit the design without reading this rationale first.

7. **Using `useDeferredValue` for slider debouncing.** Rejected after WebFetching the React docs (item 4). `useDeferredValue` is designed for rendering pauses, not side-effect debouncing — it would not stop the worker from being bombarded with `updateConfig` calls on every drag frame. The React team documents this limitation explicitly, and the correct pattern for the step 24 use case is `useEffect` + `setTimeout`. Listed as a path not taken because the step-specific context's "Required reading" item 5 suggested WebFetching the `useDeferredValue` docs, and the honest research finding is that it does not apply here; documenting that finding prevents a future agent from re-reading the same docs, reaching the opposite conclusion, and rewriting the debounce using a hook that will not actually debounce the side effect.

8. **Applying all parameter changes as full re-inits rather than live `updateConfig` calls.** Considered and rejected for the parameters that do not need a rebootstrap. Re-initializing the simulation from scratch every time the Δ⁺ slider moves would (a) destroy the metric history in the dashboard, (b) destroy the cumulative interaction graph in the network view, (c) reset the tick counter to 0, and (d) make it impossible for the researcher to observe the effect of the parameter change on the ongoing run. The whole point of live slider updates is to let the researcher see "same population, different Δ⁺, what happens from here?" — re-init defeats that. For Δ⁺, Δ⁻, interaction probability, and preferential-attachment temperature, `updateConfig` is the correct primitive because the engine re-reads these values on every tick from `state.config` (per step 13's engine design). For monolingual:bilingual ratio and seed, re-init is the only option because the ratio determines the initial population distribution and the seed determines the initial RNG state — neither can be changed in-flight without violating determinism. Section 7 documents the live-vs-reset classification for each parameter and section 3's §3.3 reference justifies each classification against the spec.

9. **Storing the tick loop timer inside `ControlsPanel` rather than `SimulationShell`.** Rejected. The shell from step 21 already owns the tick loop (it is the component that hosts the worker and drives the Canvas renderer), and lifting the timer into the controls panel would create a circular dependency: the controls would need a ref to the shell to pull data out for display, and the shell would need a callback into the controls to know when to stop. The clean division is: shell owns the timer and the worker; controls own only the draft slider values and the debounce timeouts; shell exposes callbacks (`onPlay`, `onPause`, `onStep`, `onReset`, `onSeedChange`, `onConfigUpdate`) that controls invoke, and state values (`tick`, `isRunning`, `seed`, `config`) that controls read via props. This matches step 21's intent (the shell is authoritative) and keeps the controls panel stateless aside from its own draft values.

10. **Adding an `AbortSignal` to long-running `run` calls so slider changes can cancel them mid-run.** Deferred to a future step (v1 does not need it). Step 24's Play loop calls `api.step(tickRate)` repeatedly inside a `requestAnimationFrame` callback, not `api.run(largeN)` — so there is no long-running call to cancel; each frame's call to `step` returns in well under 16ms for the N ≤ 500 agent budget. The `requestAnimationFrame` loop can be torn down instantly by clearing the rAF handle on Pause. If a future step switches to a batched `api.run(1000, onProgress)` pattern for higher throughput, an `AbortSignal` escape hatch can be added then; for v1's tick-by-tick model, `clearAnimationFrame` is the cancellation primitive.

11. **Using React's `useSyncExternalStore` to read tick count and metric values from the worker.** Considered and rejected. `useSyncExternalStore` is the right tool for subscribing a component to a store-like external source (e.g., Redux, Zustand, a custom event emitter). Step 24's data flow is simpler: the shell polls the worker once per frame via `api.step()` which returns a `TickReport`, then calls `setState` with the new tick number. React's normal state-update propagation handles everything; the controls panel reads `tick` via props and re-renders naturally. Introducing `useSyncExternalStore` would require wrapping the Comlink proxy in a custom store abstraction that the shell and the controls both subscribe to, duplicating the work React already does for free. The simpler design is sufficient and follows the shell-owns-the-loop architecture from path 9.

12. **Persisting slider values to `localStorage` so the playground remembers them across page reloads.** Out of scope for step 24 — persistence of experiment configurations is step 25's job (the config editor form backed by the `configs` SQLite table). Step 24 does not write to `localStorage` at all; a page reload resets all sliders to their config defaults. Documented here so a future agent does not scope-creep persistence into this step.

**Total research items: 3 local Next docs + 2 external WebFetched references + 7 paths not taken = 12 items**, comfortably clearing the quality gates (≥ 3 local Next docs, ≥ 2 external URLs WebFetched, ≥ 1 path not taken, total ≥ 5).

## 5. Files to create

- `app/(auth)/playground/controls-panel.tsx` — the **controls Client Component**. Starts with `'use client'` on line 1 (per the Next 16 client-component docs in research item 1). Imports: `useState`, `useEffect`, `useRef`, `useCallback` from `'react'`; `type { SimulationWorkerApi, TickReport }` from `'@/lib/sim/worker-client'`; `type { ExperimentConfig }` from `'@/lib/schema/experiment'`; `computeConfigHash` from `'./config-hash'`. Declares a component `ControlsPanel` with props `{ tick: number; isRunning: boolean; seed: number; config: ExperimentConfig; configHashShort: string; onPlay: () => void; onPause: () => void; onStep: () => void; onReset: () => void; onSeedChange: (newSeed: number) => void; onConfigUpdate: (partial: Partial<ExperimentConfig>) => void; onRebootstrap: (partial: Partial<ExperimentConfig>) => void; tickRate: 1 | 10 | 100 | 1000; onTickRateChange: (rate: 1 | 10 | 100 | 1000) => void; }`. The component renders four subsections: (1) **Transport row** — Play/Pause toggle button (shows "Pause" when `isRunning`, "Play" otherwise), Step button (disabled when `isRunning`), Reset button. Each button has a `data-testid` attribute: `play-pause-button`, `step-button`, `reset-button`. The tick counter displays next to the buttons with `data-testid="tick-counter"`. (2) **Tick-rate row** — a segmented control or `<select>` element with four options (1, 10, 100, 1000); `data-testid="tick-rate-select"`; the current value is read from the `tickRate` prop. (3) **Seed row** — `<input type="number" min="0" step="1" />` with `data-testid="seed-input"`, a "Reseed and reset" button with `data-testid="reseed-button"`, and the reproducibility banner showing `seed=<seed>` and `config=<configHashShort>` with `data-testid="config-hash-display"`. Validation: on "Reseed and reset" click, the handler reads the input's raw value, parses it with `Number()`, checks `Number.isInteger(parsed) && parsed >= 0`, and either calls `onSeedChange(parsed)` or displays an inline error message. The error message explicitly mentions that `0` is valid. (4) **Sliders** — one row per hot parameter. Each row contains a `<label>`, a `<input type="range">` with appropriate min/max/step attributes, and a numeric readout with a `data-testid` like `delta-plus-value`, `delta-minus-value`, `interaction-probability-value`, `ratio-value`, `prefattach-temp-value`. The slider's `onChange` handler updates a local draft state (`deltaPlusDraft`, etc.) on every event — keeping the input responsive — and a `useEffect` with a dependency on the draft schedules a 300ms `setTimeout` that calls either `onConfigUpdate` (for live parameters) or `onRebootstrap` (for reset-required parameters). The effect's cleanup function clears the timer, guaranteeing that rapid drags only fire the side effect once after the drag stabilizes. The reset-required sliders (ratio, preferential-attachment temperature) show a small inline warning text ("Changing this will reset the run") next to their label to set researcher expectations. File size: ~280 lines including the debounce effect, inline styles (Tailwind classes), and doc comments explaining the live-vs-reset classification.

- `app/(auth)/playground/config-hash.ts` — **pure helper for config canonicalization and SHA-256 hashing**. No `'use client'` directive because the file contains only pure functions and type exports — it is safe to import from any bundle context. Header comment: *"Client-safe pure helper. Canonicalizes an ExperimentConfig to a sorted-key JSON string, then hashes it with crypto.subtle.digest. The SHA-256 hex output is used for (a) the reproducibility banner in the playground controls, (b) export filenames per CLAUDE.md 'Export conventions'. The canonicalization is necessary because JSON.stringify is key-order-sensitive and two semantically identical configs with different key orders would hash differently."* Imports: `type { ExperimentConfig }` from `'@/lib/schema/experiment'`. Exports: (a) `canonicalStringify(value: unknown): string` — a recursive function that sorts object keys before serializing; `Array.isArray` is preserved in original order (arrays are ordered data); `null`, `number`, `string`, `boolean` are serialized via `JSON.stringify`; objects are serialized by sorting their own-enumerable keys with `.sort()` then serializing each value recursively. Edge cases: `undefined` values and function values are skipped (JSON-compatible only); `NaN` and `Infinity` are rejected with a thrown `Error` because SHA-256 of a NaN-containing config would be non-deterministic under JSON coercion. (b) `async computeConfigHash(config: ExperimentConfig): Promise<string>` — calls `canonicalStringify(config)` to produce the canonical JSON, encodes it with `new TextEncoder().encode(...)` to a `Uint8Array`, passes it to `crypto.subtle.digest('SHA-256', ...)`, converts the resulting `ArrayBuffer` to hex via `Array.from(new Uint8Array(buffer), b => b.toString(16).padStart(2, '0')).join('')`, and returns the full 64-character hex string. (c) `async computeConfigHashShort(config: ExperimentConfig): Promise<string>` — convenience wrapper that returns the first 8 characters of the full hash, suitable for display in the reproducibility banner. File size: ~80 lines including doc comments and edge-case handling.

- `app/(auth)/playground/config-hash.test.ts` — **Vitest unit test for the hash helper**. Runs under the default `node` environment; Node ≥ 20.9 exposes `globalThis.crypto.subtle` natively (since Node 20), so no jsdom or happy-dom is needed. The test file imports `canonicalStringify`, `computeConfigHash`, and `computeConfigHashShort` from `./config-hash`. It contains four `describe` blocks: (1) **`canonicalStringify` determinism** — asserts that `{ a: 1, b: 2 }` and `{ b: 2, a: 1 }` produce identical strings, that nested objects are sorted at every level, that arrays preserve their order, and that rejecting `NaN`/`Infinity` throws with a clear error message. (2) **`computeConfigHash` known-answer test** — constructs a canonical fixture config (e.g., a minimal `ExperimentConfig` with hardcoded field values that never change: `{ seed: 42, world1: { ... }, world2: { ... }, ... }`, plucked from the step-01 Zod schema's defaults so it is both valid and stable), computes the hash, and asserts equality with a **pre-computed expected hex string** baked into the test. The expected hex is computed once by the implementing claude (e.g., via `node -e "crypto.subtle.digest('SHA-256', new TextEncoder().encode('<canonical-json>')).then(b => console.log(Array.from(new Uint8Array(b)).map(x => x.toString(16).padStart(2, '0')).join('')))"`) and committed into the test. Any future regression in the canonicalization logic (e.g., a developer "optimizes" `canonicalStringify` and breaks key ordering) fails this test loudly. (3) **`computeConfigHash` canonicalization invariance** — constructs the same fixture config with its top-level keys in a different order and asserts that `computeConfigHash` returns the same hex. This is the "two semantically identical configs hash identically" invariant. (4) **`computeConfigHashShort` length** — asserts that the short form is exactly 8 characters and is a prefix of the full form. File size: ~120 lines including the fixture config and the doc-commented expected-hash literals.

The function's signature is `export function canonicalStringify(value: unknown): string`. The body: (1) handle `null` by returning `"null"`. (2) handle the primitive types (`"boolean"`, `"number"`, `"string"`) via `JSON.stringify(value)` — but for `"number"`, first check `Number.isFinite(value)` and throw a descriptive error if the check fails, because `NaN` and `Infinity` serialize as `null` under `JSON.stringify` which would silently lose information. **Architectural note on state ownership.** The controls panel is deliberately **stateless aside from its own draft values**. It does not own `tick`, `isRunning`, `seed`, or `config` — it reads those via props from `SimulationShell`. The shell is the single authoritative source of run state. This separation enables two future refactors cleanly: (a) step 25 can swap `SimulationShell`'s hardcoded config for a config loaded from the SQLite `configs` table without touching `ControlsPanel`; (b) step 26 can add a "save this run" action at the shell level without the controls panel needing to know about persistence. The only state the controls panel truly owns is (i) the five slider draft values (kept locally so dragging is instant) and (ii) the seed input draft (kept locally so typing is instant). Both drafts exist only between the start and end of a user interaction gesture; on a committed change they are propagated up via the handler props, and on a re-render from the shell with new config values they re-sync via the `useEffect(..., [config])` pattern documented in the React docs research item 4.

**Architectural note on debounce determinism.** The 300ms delay was chosen empirically from similar research-instrument UIs; it is long enough that a fast drag does not fire multiple updates, short enough that a deliberate slider move feels immediately applied. The exact value is not load-bearing — the MCP script in section 10 waits 500ms after the drag to observe the debounced update, giving a 200ms margin. If future user feedback indicates 300ms feels sluggish, the value can be lowered without ceremony; the debounce is local to the controls panel and does not affect any other component or downstream step.

**Architectural note on the tick-rate semantics.** "1× through 1000×" in F10 is ambiguous: is it "tick rate in ticks per second" or "tick multiplier per display frame"? Step 24 interprets it as the latter, because the former would require the shell to adapt its frame rate based on the slider (complex, and limited by browser animation-frame cadence to ~60Hz). The "multiplier per frame" interpretation means: at 1×, the simulation runs one engine tick per display frame (≈60 engine ticks per second on a 60Hz display); at 1000×, it runs 1000 engine ticks per display frame (≈60000 engine ticks per second, limited by how fast `api.step(1000)` returns — for N ≤ 500 agents this is comfortably under 16ms per frame per the step-20 performance budget). This interpretation is consistent with §4.2's "1× through 1000×" and is simpler to implement. If a researcher wants a fixed wall-clock tick rate (e.g., "one tick per second for a live presentation"), they can select 1× and mentally pace the display; if they want raw throughput, they select 1000×. Section 10's MCP script verifies the 100× tick-rate produces a tick counter delta > 100 over 1 second, which holds under this interpretation at any display refresh rate ≥ 1Hz.

**Architectural note on the five-slider set.** The hot-parameter set is exactly the five listed in F10 and in §3.3: Δ⁺ (positive weight update), Δ⁻ (negative weight update), interaction probability, monolingual:bilingual ratio, preferential-attachment strength. No other parameters are exposed as sliders in v1. In particular: the retry limit (from §3.3), the warm-up period for preferential attachment (from F6), and the per-language-policy bias parameters (from F5) are **not** sliders because they change infrequently and are better edited via the config form in step 25. Step 24's controls panel is the "knobs I tweak while the simulation runs" surface; step 25's config editor is the "full parameterization I set up before starting" surface. The division is deliberate and documented here so step 25's implementing claude does not duplicate sliders.

## 9. Unit tests

**Tests shipped in step 24:**

- `app/(auth)/playground/config-hash.test.ts` — the four describe blocks documented in section 5. This is the only new test file. It runs under Vitest's default `node` environment (Node ≥ 20.9 exposes `globalThis.crypto.subtle` natively, so no browser harness is needed). The known-answer SHA-256 test is the centerpiece: it constructs a canonical fixture config, computes its hash via `computeConfigHash`, and asserts equality with a pre-computed expected hex string committed into the test source. This guards against regressions in `canonicalStringify` (e.g., a developer "optimizes" key sorting and breaks determinism), in the `TextEncoder` encoding path (e.g., a future change to UTF-8 handling), and in the `crypto.subtle.digest` algorithm selection (e.g., a typo changes `'SHA-256'` to `'SHA-1'`). The test is fast (< 50ms) and deterministic — it never uses `Math.random`, never reads the clock, and never touches the filesystem or network.

**Tests intentionally not shipped in step 24:**

- **No dedicated test for the debounce `useEffect`**. The debounce logic is ~15 lines of standard React pattern and its correctness is observable end-to-end in the MCP script (step 7 of section 10 drags a slider, waits 500ms, and asserts the debounced value is visible in the readout). A unit test for the debounce would require rendering `ControlsPanel` in a happy-dom harness, simulating slider drags via `fireEvent.change`, and asserting the callback fires exactly once after the timeout — all of which is exercised more realistically by the MCP script against the real browser. Shipping a component-level test in addition would duplicate coverage at maintenance cost.

- **No dedicated test for the worker's `updateConfig` method**. Per step 20's section 9 rationale (which this step inherits), worker testing under Vitest has three viable paths (happy-dom + stubbed Worker, jsdom + stubbed Worker, Playwright) and each has material drawbacks at v1 scale. The correctness of `updateConfig` is observable in the MCP script (step 24's section 10 step 7 verifies the debounced slider update lands, which indirectly verifies `updateConfig` was called, executed, and updated the worker's `state.config`). Plus, the function body is 15 lines of null-guard + Zod parse + spread-merge — the Zod parse is already tested by step 01's schema tests. A dedicated test would re-test the Zod schema without adding independent coverage.

- **No dedicated test for `SimulationShell`'s new callbacks**. The shell's state surface is exercised end-to-end by the MCP script. A component-level test for the callbacks would require mounting the shell (which requires mocking the worker, which is the hard part — see step 20's section 9). The MCP script is the correct coverage level for this.

**Summary: one new unit test file, ~120 lines, five assertions total (one per describe block's key assertion, plus the length check in block 4).** This is the minimum coverage that gates the load-bearing invariant (config hash determinism) without ballooning the step's scope. All other correctness is verified end-to-end by the MCP script in section 10.

Exact appended block for "Known gotchas" (appended to the existing bullet list):

- `crypto.subtle.digest` requires a **secure context** — HTTPS or `localhost`. `next start` on `http://localhost:$PORT` qualifies for the localhost exception, so MCP verification works, but production deployments must be behind HTTPS or the config-hash helper and any export filename derivation will throw. ## 13. Rollback notes

If step 24 lands in a broken state (e.g., the MCP script passes but a subsequent human review finds a regression, or a follow-up step fails because of a step-24 contract bug), revert via:

```
git reset --hard <step-23-commit-sha>
```

where `<step-23-commit-sha>` is the SHA of the `step 23: network view` commit (the immediate predecessor). `run-plan.ts` will detect the missing `step 24:` marker on its next invocation and re-run the step from scratch.

Alternatively, for a surgical rollback that preserves step 24's config-hash helper and test (which are standalone and harmless) but undoes the UI and worker changes:

```
git revert <step-24-commit-sha>
```

This creates a new commit that inverts the step-24 diff while leaving the prior history intact. The helper files would be re-added in a follow-up commit if still desired.

**Preferred approach: `git reset --hard` to the step-23 SHA.** Reverts leave an asymmetric history (a step 24 commit followed by a revert commit) that confuses `run-plan.ts`'s progress detection (the marker is still present in `git log` even though the changes are undone). Hard reset is the clean option when the step is unambiguously broken and needs to be re-executed from scratch.

No database rollback is needed because step 24 does not touch the SQLite schema or any persisted data. No filesystem rollback outside `git` is needed because step 24 does not write outside the repository working tree. The screenshot at `docs/screenshots/step-24-controls.png` is tracked by git and is removed by the same reset/revert operation.

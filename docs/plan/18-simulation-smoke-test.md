## 1. Goal

Close Phase 2 of the build plan with a **full end-to-end compose-everything smoke test** of the pure-TypeScript simulation core that steps 09-17 produced. Concretely, this step delivers two deliverables: (a) a CLI script at `scripts/sim-smoke.ts` that a developer can run as `npx tsx scripts/sim-smoke.ts [tickCount]` to load the PDF canonical config, bootstrap both worlds with seed 42, run the interaction engine for 200 ticks (or a CLI-supplied count), collect per-tick scalar + graph metrics, compute the run summary, and print a compact text report to stdout; and (b) a colocated Vitest suite at `lib/sim/sim-smoke.test.ts` that runs a shorter 50-tick version of the same pipeline as an **automated post-step gate** in CI. Both files also serve as the first — and, before step 20's Comlink worker wrapping, the only — place in the codebase where the **entire simulation stack composes**: RNG (step 09), core types (step 09), topologies (step 10), agent bootstrapping (step 11), language policies (step 12), the interaction engine tick loop (step 13), preferential attachment (step 14, optional ablation toggle), scalar metrics (step 15), graph metrics (step 16), and run-summary classification (step 17) all interact under one roof, with no worker postMessage layer and no browser to confuse the picture. This is the **last pure-TypeScript step** before the Web Worker integration (steps 19-20) arrives and brings Comlink, `structuredClone`, and Turbopack's worker-graph bundling rules with it — so the step exists specifically to de-risk "did the core work?" before "did the worker wrapping work?" becomes an entangled question in step 20. The step is also the **first place** in the codebase that can answer the research-question-level sanity checks the `docs/spec.md` §7 metrics taxonomy is designed to produce: does the communication success rate rise above zero at tick 200 (i.e. are any interactions succeeding under the default config)? Does the assimilation index respond to a policy ablation (i.e. does swapping the PDF's default policy set for the `'always-l1'` alternative produce a materially different final classification)? Does the whole 200-tick pipeline produce bit-identical output across two invocations of the same seed (i.e. is the determinism promise from `CLAUDE.md` "Testing conventions" actually honored end-to-end, not just per-module)? Each of those questions is asked by exactly one assertion in the script or the test file, and each is load-bearing for the project's epistemic guarantee that `msksim` is a research instrument, not a demo. The step touches **no** worker code, **no** UI surface, **no** persistence, and **no** database — those arrive in later steps and are explicitly out of scope here per the scope boundary in the step-specific context.

**Local Next.js 16 docs (authoritative per `CLAUDE.md` "Next.js 16 deltas from training data" and `AGENTS.md`):**

The step-specific context phrases this as "Uses `test.concurrent: false` because the simulation is CPU-bound"; Vitest's default in-file serialization already satisfies this, and the test file adds a top-of-file comment pointing at this doc so a future reviewer does not mistakenly add `.concurrent` to speed things up.

Step 18's script and test file deliberately import the same `@/lib/sim/*` modules that step 20 will later import into `workers/simulation.worker.ts`. If step 18's script runs cleanly under `tsx` (which uses Node's native module resolution plus esbuild's transform pipeline), then those same modules will bundle cleanly under Turbopack's worker graph in step 20 — the two resolution strategies exercise the same module-purity invariant (no `fs`, no `path`, no `crypto`, no `next/*`, no `react`). Step 18 is thus the **last-line defense** against a module-purity regression before the worker integration arrives. A top-of-file comment in `scripts/sim-smoke.ts` documents this "if this script runs, step 20 will bundle" intent so future readers understand why a seemingly trivial "just run the sim in Node" script is load-bearing for the worker integration that follows it.
3. `node_modules/next/dist/docs/01-app/01-getting-started/05-server-and-client-components.md` §"Preventing environment poisoning" — reiterates the guidance that only modules that must not cross the client boundary carry `import 'server-only'`. **External references (WebFetched):**

5. **Dall'Asta, L., Baronchelli, A., Barrat, A., & Loreto, V. (2008). *In-depth analysis of the Naming Game dynamics: the homogeneous mixing case.*** <https://arxiv.org/abs/0803.0398> (WebFetched during planning). The full PDF body is not readable through WebFetch's HTML-to-markdown pipeline, but the abstract and metadata are. The confirmed framing: the paper "investigates further the Naming Game, a model able to account for the emergence of a shared vocabulary of form-meaning associations through social/cultural learning" and its central observable is the number of distinct words `Nw` across the population. The specific quantitative results relevant to step 18's sanity checks — in particular the **mean-field scaling `t_consensus ~ N^(3/2)`** and the **three-phase trajectory** (invention/borrowing phase, peak-Nw plateau, monotonic decay to consensus) — are documented in the spec's §12.1 citation of this paper rather than in the WebFetched HTML. Step 18 uses this citation in the informational sense: the 200-tick run on N ~ 100 agents per world is **well below** the mean-field consensus-time scale (`N^(3/2)` ~ 1000 ticks for N = 100) so the run is expected to terminate in the **climbing phase** rather than at consensus. The plausibility check "successRate > 0 at tick 200" is the direct operational form of this expectation: at 200 ticks of a 100-agent run, we are in the Nw climb-up, some interactions are succeeding, and the success rate is strictly above zero but well below 1. A test that asserted `successRate === 1` (full consensus) would be testing the wrong phase and would be unreliable. This is the single most important piece of theoretical grounding for the plausibility check to not be arbitrary.
6. **Baronchelli, A., Dall'Asta, L., Barrat, A., & Loreto, V. (2006). *Topology-induced coarsening in language games.* Physical Review E 73, 015102(R).** <https://pubmed.ncbi.nlm.nih.gov/16486202/> (WebFetched during planning). The PubMed abstract confirms the key result: on low-dimensional lattices (d ≤ 4), Naming Game consensus is reached through a **coarsening process** with consensus time scaling as `N^(1+2/d)`, which in d=2 gives `N²`. The paper also establishes that the lattice case requires **less memory per agent** than the mean-field case (`N` vs `N^(3/2)`) but **longer total consensus time**. The direct consequence for step 18: a 2D lattice run with ~100 agents per world is expected to take on the order of **N² = 10^4 ticks** to reach full consensus, which means a 200-tick run is **very** early in the coarsening process — too early for the "domains" to have grown visible, too early for Nw to have decayed significantly, but well into the regime where the success rate has begun climbing above zero. The success-rate-above-zero plausibility check is theoretically sound for exactly this reason: it tests "are we in the coarsening phase at all" rather than "have we reached consensus," which is the right question to ask of a 200-tick smoke test. The paper's second contribution — the boundary-diffusion mapping in 1D — is not directly exercised by step 18 (the default topology is 2D lattice, not 1D), but is cited for completeness because it establishes that the coarsening is an **ordered, analytically tractable** process rather than a chaotic relaxation. Step 18 does not try to reproduce the coarsening exponent; it only tests that a 200-tick run sits in the early-coarsening phase rather than in a degenerate all-failures or all-successes regime.

**Paths not taken:**

7. **Pure vitest-only, no CLI script.** Rejected. Step 18 could have shipped only `lib/sim/sim-smoke.test.ts` and relied on `npm test -- lib/sim/sim-smoke` as the sole invocation path. That would be cheaper to build and would cover the CI gate requirement, but it would miss two important use cases. First, **manual developer exploration**: researchers iterating on their hypotheses (per the project charter in `CLAUDE.md` "Purpose and audience") want to run a 500-tick or 1000-tick sim and read a text report without spelunking through Vitest's test-runner output. The CLI script is the entry point that makes this workflow feasible, and it costs very little because the test file can be a thin wrapper over the same internal function. Second, the implementing claude at the end of step 18 needs to **run the full 200-tick pipeline once** as a final verification gate, and doing so via `npx tsx scripts/sim-smoke.ts` is dramatically faster to interpret than scrolling a Vitest output buffer for a specific test's console output. The step-specific context explicitly calls out both use cases. The chosen design ships **both** files: the script is the developer-facing tool, the test is the CI-facing gate, and they share a common internal `runSmoke(config, seed, tickCount)` helper so the pipeline logic lives in one place.
8. **Running 10_000 ticks in the smoke test.** Rejected. A 10_000-tick run on N ~ 100 agents per world would execute ~10⁶ interactions and produce 10⁴ `ScalarMetricsSnapshot` + 10⁴ `GraphMetricsSnapshot` entries. That is a ~10-second run on a modern laptop (estimated from the O(1) tick cost of step 13 under the default config and the O(N²) matching-rate metric in step 15) which is fine for a CLI smoke but painful for a Vitest run that needs to complete in under a minute alongside dozens of other tests. The chosen tick counts are **200 for the CLI script** (long enough for the success rate to rise above zero and for the ablation to diverge) and **50 for the Vitest test** (short enough to complete in under 1 second, long enough for the determinism and plausibility invariants to hold at measurable signal). Both thresholds are tunable via CLI arg for the script and via constant-at-top-of-file for the test.
9. **Testing the convergence status (`converged` / `metastable` / `diverged` / `unresolved`).** Rejected as brittle. A 200-tick run on a 2D lattice with ~100 agents will almost certainly return `convergenceStatus === 'unresolved'` per Baronchelli 2006's coarsening-time analysis (consensus needs ~10⁴ ticks, we are running 200). Asserting `=== 'unresolved'` is technically correct but is testing the wrong thing: the assertion would pass even if the whole coarsening dynamics were broken, because a broken run would *also* be unresolved. Instead, step 18 asserts that the classification label is **one of the four valid strings** (an integrity check on the summary reducer output) and leaves the specific-label assertion to a future longer-run integration test that actually runs long enough to distinguish `converged` from `metastable`. The step-specific context explicitly lists "classification label is one of the four valid values" as one of the summary assertions for the test file, which is the form this plan file implements.
10. **Directly asserting the PDF's 3:2 monolingual:bilingual ratio is honored.** Rejected as redundant. Step 11's `bootstrapExperiment` test suite (its own `lib/sim/bootstrap.test.ts` per step 11's section 9) already asserts that a 100-agent world with the 3:2 default produces 60 monolingual and 40 bilingual agents. Re-asserting the same invariant in step 18 would duplicate coverage without adding information. Step 18's script still *uses* the 3:2 default (by calling `ExperimentConfig.parse({})`), so the ratio is exercised; only the redundant test assertion is cut.
11. **Testing both `always-l1` AND `always-l2` ablations.** Rejected as redundant. The step-specific context explicitly names `always-l1` as the ablation lever because it produces the sharpest behavioral change (immigrants speak L1 to natives who don't know L1 → failures → no L2 assimilation). `always-l2` would produce the symmetric result for a different mechanism, which is diagnostically interesting but is a second ablation test — outside the scope of a smoke test. Researchers who want the symmetric ablation can run it via the step-25 config editor once that lands; step 18 ships exactly the one ablation its contract names.

Total research items: **4 local Next doc citations** + **2 external URLs WebFetched** (Dall'Asta 2008, Baronchelli 2006) + **5 paths not taken** = **11 links**, comfortably above the ≥ 5 floor mandated by the plan-file contract and `CLAUDE.md` "Research quality gates" (≥ 3 local Next docs, ≥ 2 external URLs, ≥ 1 path not taken, total ≥ 5).

## 5. Files to create

- `scripts/sim-smoke.ts` — the CLI script. A single-file Node program runnable via `npx tsx scripts/sim-smoke.ts` (with optional trailing positional tick-count argument, e.g. `npx tsx scripts/sim-smoke.ts 50` for a 50-tick run). The file exports **one named function** `runSmoke(config: ExperimentConfig, seed: number, tickCount: number, opts?: { policy?: PolicyName }): SmokeResult` where `SmokeResult = { scalarTimeSeries: ScalarMetricsSnapshot[]; graphTimeSeries: GraphMetricsSnapshot[]; summary: RunSummary; wallClockMs: number; tickCount: number; seed: number }`, and invokes that function from a top-level `if (import.meta.url === ...)` or tsx-compatible main-guard block. The main block: (1) parses the optional tick-count CLI argument with a default of 200 and a sanity cap of 100_000; (2) loads the PDF canonical config via `ExperimentConfig.parse({})` (which fills every default including the 3:2 mono:bi ratio, two worlds, default vocabulary, default policies); (3) calls `runSmoke(config, 42, tickCount)`; (4) calls `runSmoke(config, 42, tickCount)` a **second time** and asserts bit-identical output on both scalar and graph time series (the determinism check); (5) calls `runSmoke` a third time with `opts: { policy: 'always-l1' }` (a shallow-merged override of the default policy config — see section 7 step 4 for the exact override mechanics); (6) asserts the absolute difference between the default and always-l1 final-tick assimilation indices exceeds 0.05 (the ablation check); (7) runs the plausibility checks (successRate > 0, Nw ≤ lexicon size, assimilation index in [0,1], modularity in [-0.5, 1.0]); (8) prints the text report; (9) exits 0 on success or 1 with a diff-formatted error message on any failure. The script has no side effects at import time — `runSmoke` is a pure function of its arguments, and the main-guard block only runs when the file is invoked directly (not when the test file imports `runSmoke`). The file begins with a top-of-file JSDoc comment explaining the script's role as the last pre-worker integration check, and ends with a trailing newline per the project's flat ESLint config. No `import 'server-only'` (deliberately — the file imports from `lib/sim/` which is cross-boundary-shared; see research note 3).

- `lib/sim/sim-smoke.test.ts` — the Vitest suite. A single test file in the default `node` environment (no `// @vitest-environment` docblock needed; the vitest config from step 00 defaults to `node`). The file imports `runSmoke` from `@/scripts/sim-smoke` via the `@/` alias from step 00 (or, if the alias does not extend to `scripts/`, imports via a relative path like `../../scripts/sim-smoke`; the exact path is determined at commit time by reading `tsconfig.json` and `vitest.config.ts`). It also imports `ExperimentConfig` from `@/lib/schema/experiment` to parse the default config in each test. The file contains six tests, enumerated in section 9 below: (1) full pipeline runs 50 ticks without throwing, (2) determinism invariant, (3) plausibility at the end of the run, (4) ablation against `always-l1`, (5) classification label validity, (6) tick count matches input. Every test uses a constant `TICK_COUNT = 50` and `SEED = 7` pinned at the top of the file. A leading comment block points at this plan file's section 9 and at `CLAUDE.md` "Testing conventions" so a future reader understands the serialization invariant and the determinism-first philosophy. No `Math.random()`, no `Date.now()`, no file I/O — every test reads only from inline fixtures and the deterministic `runSmoke` function.

Both files are tree-shakeable ESM with no Node built-ins beyond what `tsx` already injects for the script's main-guard block (`process.argv`, `process.exit`, `process.hrtime.bigint` — all three are Node-only APIs that are intentionally confined to the main-guard block and never touched from `runSmoke` or the test file, keeping the pure-function portion of the code importable from any context).

## 7. Implementation approach

The step proceeds in prose-ordered phases. Each phase leaves the repository in a testable intermediate state, and the final phase is the single-commit squash and push required by `CLAUDE.md` "Commit-message convention."

**Phase 1 — Survey the prior steps.** Before writing any code, the implementing claude reads the relevant parts of each prior step's landed code (not the plan files, the actual committed files) to confirm the exact export names: `lib/sim/rng.ts` (for `createRNG`), `lib/sim/bootstrap.ts` (for `bootstrapExperiment` and the `World` type), `lib/sim/engine.ts` (for `tick`, `SimulationState`, `InteractionEvent`, `TickResult`), `lib/sim/metrics/scalar.ts` (for `computeScalarMetrics`), `lib/sim/metrics/graph.ts` (for `computeGraphMetrics`, the cumulative interaction graph helper, and whatever merge function step 16 exposed), `lib/sim/metrics/summary.ts` (for `computeRunSummary`), `lib/sim/metrics/types.ts` (for the `ScalarMetricsSnapshot`, `GraphMetricsSnapshot`, `RunSummary`, `ConvergenceStatus`, and `RunClassification` type exports), and `lib/schema/experiment.ts` (for `ExperimentConfig` and its `parse` method). If any prior step's actual exports differ from this plan file's assumptions, **use the actual exports** — this plan file's assumptions are a guide, not a contract, and schema drift across nine steps is normal. Specifically, if step 15 or 16 did not expose a standalone per-tick helper but only a batched one, the smoke script adapts its tick loop to call whatever they did expose. If the per-tick graph metrics path requires maintaining a cumulative interaction graph across ticks (per step 16's design), the script maintains it explicitly and passes it into `computeGraphMetrics` on each tick. If step 16's `updateInteractionGraph` is the mutation helper and `computeGraphMetrics` is the pure reducer, the script calls them in the order step 16 documents.

**Phase 2 — Write the shared `runSmoke` function.** This is the heart of the script and the test. Its signature is `runSmoke(config: ExperimentConfig, seed: number, tickCount: number, opts?: { policy?: PolicyName }): SmokeResult` and its body proceeds as a straight-line sequence:

First, apply any policy override from `opts.policy`. If `opts.policy` is provided and differs from `config.policy.policyName` (or whatever step 01 named the field), construct a shallow-cloned config with the override applied — **do not mutate the input config**. The shallow clone is `{ ...config, policy: { ...config.policy, policyName: opts.policy } }`. This mirrors the "immutable inputs" discipline every step in phase 2 has committed to. If `opts.policy` is the same as the config's default (or `opts` is absent), use the config unchanged.

Second, record the wall-clock start via `process.hrtime.bigint()` if the function is being called from the CLI main-guard block. The test file does not care about wall-clock time and can pass a sentinel / ignore the field. To keep `runSmoke` pure at call sites that do not want wall-clock measurement, the time measurement happens inside `runSmoke` unconditionally, and the test simply ignores `result.wallClockMs` in its assertions — the field is always present but optional to read. `process.hrtime.bigint()` is the canonical monotonic clock in Node; it returns nanoseconds as a `bigint`, and the script converts to milliseconds via `Number(nsEnd - nsStart) / 1_000_000`. This choice is documented in a code comment because `Date.now()` would be a determinism violation per `docs/spec.md` §8 ("No wall-clock-dependent logic" for simulation code) — the measurement is for *reporting*, not for *simulation logic*, so it does not violate the rule, but a future reader should see the distinction called out explicitly.

Third, bootstrap the experiment: `const { world1, world2, rng } = bootstrapExperiment(config, seed)`. This returns two fully-populated worlds and the single shared RNG per step 11's single-RNG discipline. The RNG is threaded into every subsequent call that makes random draws — `tick`, `computeGraphMetrics`, and anywhere else that needs entropy.

Fourth, initialize the `SimulationState`: `let state: SimulationState = { world1, world2, tickNumber: 0, config }` (or whatever field names step 13 settled on). Initialize an empty cumulative interaction graph (step 16's `createInteractionGraph()` or equivalent — the exact initializer is whatever step 16 committed). Initialize two empty arrays: `const scalarTimeSeries: ScalarMetricsSnapshot[] = []` and `const graphTimeSeries: GraphMetricsSnapshot[] = []`.

Fifth, run the tick loop `for (let t = 0; t < tickCount; t++) { ... }`. Each iteration: (a) calls `tick(state, rng)` to get a `TickResult`, (b) updates the cumulative interaction graph by merging in the tick's emitted `InteractionEvent[]` via step 16's update helper, (c) computes this tick's scalar metrics via `computeScalarMetrics(state.world1, state.world2, tickResult.interactions)` (or whatever signature step 15 committed), (d) computes this tick's graph metrics via `computeGraphMetrics(state.world1, state.world2, cumulativeGraph, tickResult.interactions)` (or step 16's signature), (e) pushes both snapshots into their respective time-series arrays, (f) replaces `state` with `tickResult.state` so the next iteration sees the updated worlds. The loop makes no side-effect calls beyond array pushes and graph mutations; every call is deterministic given the seed.

Sixth, call `computeRunSummary(scalarTimeSeries, graphTimeSeries, config)` from step 17 to get the `RunSummary`.

Seventh, compute the wall-clock duration via the `process.hrtime.bigint()` delta from step two.

Eighth, return `{ scalarTimeSeries, graphTimeSeries, summary, wallClockMs, tickCount, seed }` as a frozen object (via `Object.freeze` on the outer shell; the inner arrays are left mutable only because freezing them would require a deep traversal and the caller is trusted to not mutate).

**Phase 3 — Write the CLI main block.** The main-guard uses `tsx`-compatible detection: `tsx` evaluates `import.meta.url` identically to Node's native ESM loader, so `if (import.meta.url === \`file://${process.argv[1]}\`)` (or the more robust `url.pathToFileURL(process.argv[1]).href` comparison from Node's `url` module) is the canonical idiom. Inside the main block:

1. Parse the optional tick-count argument: `const tickCount = process.argv[2] ? Number.parseInt(process.argv[2], 10) : 200`. Sanity-check: if `tickCount < 1 || tickCount > 100_000 || Number.isNaN(tickCount)`, print a usage message and exit 2. The 100_000 cap is defensive against a user accidentally typing `1000000` and blowing out the node heap.
2. Parse the default config: `const config = ExperimentConfig.parse({})`. This loads every default (3:2 mono:bi, two worlds, default vocabulary, default policies). If step 01's schema requires any non-default fields (unlikely given step 01's plan mentions a full-defaults `parse({})` call works), this step's claude fills them in inline with a comment explaining why each field is non-default.
3. Run the primary simulation: `const result = runSmoke(config, 42, tickCount)`. Seed 42 is documented in a comment as "the canonical reproducibility seed used throughout this codebase." Use a numeric literal, not an env var or `Date.now()`.
4. **Determinism check.** Run `const result2 = runSmoke(config, 42, tickCount)`. Compare `result.scalarTimeSeries` and `result.graphTimeSeries` to `result2.scalarTimeSeries` and `result2.graphTimeSeries` via a deep-equal function. Since both time series are arrays of plain JSON-serializable objects (per step 15 and step 16's `structuredClone`-safe design), the cheapest reliable deep-equal is `JSON.stringify(result.scalarTimeSeries) === JSON.stringify(result2.scalarTimeSeries)` followed by the same check on the graph series. If either comparison fails, print a diff-like error message that shows the first differing tick index and the differing field (computed by iterating both arrays and finding the first mismatch), then `process.exit(1)`. The "diff-like" format is three lines: `Determinism violation at tick ${firstMismatchIndex}`, `expected: ${JSON.stringify(result.scalarTimeSeries[firstMismatchIndex])}`, `actual:   ${JSON.stringify(result2.scalarTimeSeries[firstMismatchIndex])}`. This is the most informative failure message a CLI script can produce without pulling in a dedicated diff library, and it matches the step-specific context's requirement to "exit 1 with a diff-like error message."
5. **Policy ablation check.** Run `const ablation = runSmoke(config, 42, tickCount, { policy: 'always-l1' })`. Extract the final-tick assimilation index from each of `result` and `ablation`. Compute `|result.final_assim - ablation.final_assim|`. If the absolute difference is ≤ 0.05, print an error message explaining that the policy ablation did not produce a materially different outcome (which means either the policy system is broken or the ablation is running against too short a tick count) and exit 1. The 0.05 threshold is from the step-specific context and is documented in an inline comment with a link to this plan file's section.
6. **Plausibility checks.** Extract the final-tick metrics from `result`:
   - `successRate > 0`: read `result.scalarTimeSeries[tickCount - 1].overall.successRate.rate` (or `.world1.successRate.rate` if step 15 did not expose `overall`, in which case use the higher of the two worlds' rates — a successRate > 0 in either world is sufficient for the sanity check). Assert strictly greater than 0. If the check is being run on a degenerate config where no interactions happen at all, this assertion is expected to fail, and the failure is a real signal rather than noise.
   - `Nw_world1 <= |L1 lexicon|`: read `result.scalarTimeSeries[tickCount - 1].world1.distinctActiveTokens` (or whatever key step 15 committed — see section 7 step 1). Read `|L1 lexicon|` from the config's vocabulary seed: the cardinality of the L1 lexicon is the total count of distinct `TokenLexeme` values under `config.vocabularySeed.l1` across all referents, or however step 01 shaped that field. Assert `Nw <= |L1 lexicon|`. This check catches bugs where the metric accidentally double-counts tokens across referents or across worlds.
   - `0 <= assimilationIndex <= 1`: read `result.graphTimeSeries[tickCount - 1].assimilationIndex`. Handle the `null` case (step 16 specified `assimilationIndex: number | null`): if `null`, the check passes (null means "insufficient data," not "out of range"). Otherwise assert `0 <= x && x <= 1`.
   - `modularity ∈ [-0.5, 1.0]` (the Louvain range from step 16's research notes): read `result.graphTimeSeries[tickCount - 1].interactionGraphModularity`. Assert `-0.5 <= x && x <= 1.0`. This is the theoretical range of Louvain modularity per the Wikipedia reference step 16 cited; anything outside it indicates a numerical bug in the Louvain implementation or an edge case step 16 did not handle.
   Any failing plausibility check prints its own error line (`FAIL plausibility: successRate = 0 at tick ${tickCount}`, etc.) and sets an error flag; after all checks run, if any flag is set, exit 1. The script intentionally runs all checks even if an earlier one fails, so a single run surfaces all failures at once — this is more useful for debugging than bailing on the first failure.
7. **Print the compact text report.** Six-to-eight lines of plain text: (a) `sim-smoke: tick count = ${tickCount}, wall-clock = ${wallClockMs.toFixed(1)}ms`; (b) `  Nw: world1 = ${nwW1}, world2 = ${nwW2}` (from final-tick graph metrics); (c) `  success rate: world1 = ${rate1.toFixed(4)}, world2 = ${rate2.toFixed(4)}` (from final-tick scalar metrics); (d) `  assimilation index = ${assim ?? 'null'}, segregation index = ${seg.toFixed(4)}`; (e) `  classification = ${summary.classification}`; (f) `  time-to-consensus = ${summary.timeToConsensus ?? 'not reached'}`; (g) `  determinism: OK (bit-identical across two 200-tick runs)`; (h) `  ablation: default vs always-l1 Δassim = ${diff.toFixed(4)} (threshold 0.05) OK`. The text is aligned with two-space indentation for readability; no ANSI color codes (the script runs in CI and in plain terminals, and colors are extra noise). The report is what a developer sees when they run `npx tsx scripts/sim-smoke.ts`, and it is what the implementing claude at the end of step 18 eyeballs to confirm the pipeline works.
8. **Exit 0** if all checks passed, **exit 1** if any check failed. Never exit with a negative code.

## 9. Unit tests

`lib/sim/sim-smoke.test.ts` is a single Vitest file in the default `node` environment, containing **six named tests** organized as flat `test('...', () => { ... })` blocks (no nested `describe`). Every test uses `const TICK_COUNT = 50` and `const SEED = 7` pinned at the top of the file, and every test calls `runSmoke(config, SEED, TICK_COUNT [, opts])` where `config = ExperimentConfig.parse({})`. No shared `beforeEach` state; each test builds its own `config` inline so failures are self-contained. All tests use the default `node` environment and run sequentially per Vitest's default in-file ordering (the step-specific context's `test.concurrent: false` requirement is satisfied by not marking any test as `.concurrent`). Expected total runtime: under 5 seconds on a modern laptop for six 50-tick runs + two additional runs for determinism and ablation checks.

1. **Full pipeline runs 50 ticks without throwing.** Call `runSmoke(ExperimentConfig.parse({}), 7, 50)` and assert the returned `result.tickCount === 50`, `result.scalarTimeSeries.length === 50`, `result.graphTimeSeries.length === 50`, and `result.summary` is a non-null object with a `classification` string field. This is the basic "it composes" check that catches any import-time or type-mismatch regression across the nine prior steps. If this test fails, the most likely cause is an export-name drift in one of steps 09-17; the failure message guides the implementing claude to the specific module.

2. **Determinism invariant: two 50-tick runs produce bit-identical scalar + graph time series.** Call `runSmoke(config, 7, 50)` twice (storing the results in `result1` and `result2`) and assert `JSON.stringify(result1.scalarTimeSeries) === JSON.stringify(result2.scalarTimeSeries)` and `JSON.stringify(result1.graphTimeSeries) === JSON.stringify(result2.graphTimeSeries)`. Using `JSON.stringify` for the equality check is deliberate: it is (a) cheap, (b) produces a readable error message via Vitest's default diff, and (c) exactly equivalent to the deep-equal check the CLI script performs, so the two surfaces agree on what "bit-identical" means. If the test fails, Vitest's diff highlights the first differing character, which is usually close enough to the problem tick that the implementing claude can locate the non-determinism source quickly. This test is the **load-bearing invariant** for `docs/spec.md` §8's "reproducible given seed + config" architectural commitment and for `CLAUDE.md` "Testing conventions" determinism rule; a failure here is a release-blocking bug.

3. **Plausibility: success rate > 0 at the end of the run.** Call `runSmoke(config, 7, 50)`. Read the final-tick success rate from the scalar time series (the exact accessor path is whatever step 15 committed — typically `result.scalarTimeSeries[49].overall.successRate.rate` or the higher of the two per-world rates). Assert it is **strictly greater than 0**. This is the "are we in the coarsening phase at all" check motivated by Baronchelli 2006 and Dall'Asta 2008; at 50 ticks on a ~100-agent-per-world 2D lattice with L1+L2 seeded vocabularies, some interactions must have succeeded, and zero means either the engine is broken or the default config produces no interactions at all. The test also asserts the final Nw for world1 is **at most** the number of distinct L1 lexemes in the config's vocabulary seed (sanity on token counting) and that the final assimilation index, if not `null`, is in `[0, 1]`, and the final interaction-graph modularity is in `[-0.5, 1.0]` (the theoretical Louvain range). All four plausibility sub-assertions live in a single test because they share fixture generation and a single 50-tick run; separating them would triple the runtime for no added coverage.

4. **Ablation: always-l1 policy produces a materially different assimilation index than the default policy.** Call `runSmoke(config, 7, 50)` for the default run and `runSmoke(config, 7, 50, { policy: 'always-l1' })` for the ablation run. Read the final-tick assimilation index from each. Handle the `null` case explicitly: if either is `null`, treat the ablation check as inconclusive for this short-run seed and skip the assertion with a `console.warn` explaining why (the 50-tick run may be too short for the W2-Immigrant↔W2-Native interaction count to cross the denominator threshold step 16 uses for assimilation index). Otherwise assert `Math.abs(defaultFinalAssim - ablationFinalAssim) > 0.05`. The 0.05 threshold matches the CLI script's threshold and the step-specific context's requirement. The rationale is spelled out in a long comment in the test file: the `always-l1` policy forces immigrants to speak L1 to natives who don't know L1, which produces failures, which means no L2-mediated successful interactions, which means the assimilation index (share of successful immigrant↔native interactions that happened in L2) should be dramatically suppressed relative to the default. A failure here means either the policy system is not wired into the tick loop (the step-12/step-13 interface is broken) or the metric is not sensitive to the policy (step 16's assimilation index computation is wrong).

5. **Summary: classification label is one of the four valid values.** Call `runSmoke(config, 7, 50)` and assert `result.summary.classification` is a member of the set `['assimilated', 'segregated', 'mixed', 'inconclusive']`. The specific value is not asserted because a 50-tick run is very early in the coarsening dynamics and the classification will most likely be `'inconclusive'` or `'mixed'`, but either is correct at this tick count. This test is an **integrity check** on the step-17 summary reducer output: if the classification field is `undefined`, `null`, a number, or an unexpected string, the summary reducer has regressed. The four-value enumeration is the exact set step 17 committed in `RunClassification`.

6. **Tick count matches input.** Call `runSmoke(config, 7, 50)` and assert `result.tickCount === 50` (the configured tick count) and `result.seed === 7` (the configured seed). This is a triviality-check that catches a regression where `runSmoke` silently runs a different number of ticks (e.g., an off-by-one loop) or returns a wrong metadata field. The check is cheap and catches a specific class of bugs that would otherwise only be visible by reading through the time-series array length.

All six tests run under the default `node` environment, pin their seeds and tick counts, and have no I/O or wall-clock dependencies beyond the `process.hrtime.bigint` measurement baked into `runSmoke` (which is captured but not asserted against, since `wallClockMs` varies per machine).

## 13. Rollback notes

5. Re-run the pipeline from step 18 with an adjusted plan file — typically the adjustment is to the tick count, the plausibility thresholds, or the ablation mechanism, not to the overall structure.

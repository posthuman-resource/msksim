## 1. Goal

Deliver the **scalar (non-graph) per-tick observables** from `docs/spec.md` §7.1 as a small, pure-function module under `lib/sim/metrics/`. Step 15 covers exactly five metrics — communication success rate, mean token weight, token weight variance, number of distinct active tokens (`Nw`), and matching rate — and nothing else. The graph-derived metrics (largest-cluster size, cluster count, interaction-graph modularity, assimilation index, segregation index) are step 16's territory and the end-of-run classifiers (time-to-consensus, assimilation/segregation/mixed/inconclusive labeling) are step 17's. Splitting the §7.1 table across steps 15-17 is deliberate: each `claude -p` invocation stays focused on a tight, testable surface, and the step-17 run-summary classifier can consume the `ScalarMetricsSnapshot` type defined here without waiting for step 16 to land. The module's single public entrypoint, `computeScalarMetrics(world1, world2, tickInteractions)`, is a **pure function** of the current `World` pair plus the tick's `InteractionEvent[]` — no RNG, no wall clock, no mutation, no persistence. Writing the snapshot to the `tick_metrics` table is a later concern: step 20 wires the worker to emit these values per tick, and step 26 persists them via a Server Action. Step 15 is pure computation, no I/O. The values produced here feed RQ3 (token-weight dynamics) and RQ1/RQ4 (communication success and matching as proxies for cohesion), and step 22's Recharts dashboard and step 30's long-format CSV export both ultimately depend on this snapshot shape. Efficiency matters: for `N ≤ 500` per world at up to 10⁴ ticks per run with batch sweeps of hundreds of replicates, every metric in this step must be `O(N)` per tick — with one documented exception for matching rate, which is naively `O(N²)` but acceptable in the v1 interactive regime per `docs/spec.md` §11 question 2.

## 3. Spec references

- `docs/spec.md` **§7.1 Per-tick scalar metrics** — the authoritative table for all five metrics in this step. The exact rows step 15 implements are:
  1. **Communication success rate** — _"`successful_interactions / total_interactions` in the tick; broken down per world and per `(speaker.class, hearer.class)` pair."_ Supports RQ1, RQ4.
  2. **Mean token weight** — _"Mean of all non-zero token weights, broken down per world and per language."_ Supports RQ3. The "non-zero" qualifier is load-bearing: tokens with weight 0 are treated as inactive and excluded from the mean so that adding a new agent with sparse inventory does not artificially depress the average.
  3. **Token weight variance** — _"Variance across non-zero token weights, per world, per language. High variance indicates a split vocabulary."_ Supports RQ3. Same non-zero filter as mean. Sample variance (divisor `n−1`) matches the statistical-analysis-in-R workflow the `docs/spec.md` §9 CSV export target expects.
  4. **Number of distinct active tokens (Nw)** — _"Count of tokens with any weight > 0, per world. Canonical Naming Game observable."_ Supports RQ3. A "token" here is a `(language, lexeme)` pair — step 15 counts **distinct pairs across the whole population**, not the sum of per-agent counts. The "canonical Naming Game observable" phrasing ties this directly to Dall'Asta et al. 2008's mean-field analysis (see §4 below).
  5. **Matching rate** — _"Fraction of agent pairs whose top-weighted token for a given referent agrees, averaged over referents."_ Supports RQ1, RQ4.
- `docs/spec.md` **§7.1 rows 6-11** — explicitly **not** implemented here. Largest-cluster size, cluster count, interaction-graph modularity, assimilation index, segregation index, and time-to-consensus belong to steps 16 and 17. Step 15's `ScalarMetricsSnapshot` type has no fields for them.
- `docs/spec.md` **§3.3 Interaction rules** — defines the `InteractionEvent` schema step 13 produces. Success/failure is determined by step 5 of the interaction rules (_"the hearer looks up the token in its own inventory. If the hearer has the token associated with the same referent, the interaction is a success"_). Step 15 does not re-derive success — it reads the boolean directly from `event.success`.
- `docs/spec.md` **§11 question 2 "What is the right agent count?"** — _"default to N = 50-500 per world for interactive playground mode [...] and allow headless sweeps up to N = 10⁴ in workers."_ This is the load-bearing citation for the matching-rate cost discussion: the naive O(N²) formulation is fine at N ≤ 500 per world (≤ 125,000 pairwise comparisons per world per tick, trivially sub-millisecond in pure JS), but is **not** fine at N = 10⁴ per world (50 million comparisons per tick). Section 7 below documents the accepted trade-off and the extension point for a sampled approximation if a future headless sweep makes it necessary.
- `docs/spec.md` **§6 Research goals → software support matrix** — RQ3 _"token-weight dynamics"_ is primarily supported by _"mean token weight, weight variance, number of distinct active tokens Nw,"_ all of which live in this step. RQ1 _"assimilation vs. segregation thresholds"_ is partially supported by communication success rate (here) with the rest landing in steps 16-17.

**External references (WebFetched during planning):**

4. **Dall'Asta, L., Baronchelli, A., Barrat, A., & Loreto, V. (2008). "In-depth analysis of the Naming Game dynamics: the homogeneous mixing case."** _International Journal of Modern Physics C_ 19, 785. <https://arxiv.org/abs/0803.0398> — WebFetched. Abstract (fetched): _"The paper investigates the Naming Game model, which accounts for 'the emergence of a shared vocabulary of form-meaning associations through social/cultural learning.' The authors analyze the dynamics using numerical simulations and analytical arguments, reviewing existing results and presenting new understanding of this multi-agent system."_ The paper is the **canonical citation** for Nw as the primary Naming Game observable in the mean-field (well-mixed) regime. In Dall'Asta et al., Nw rises sharply from its initial value (≈ N, because every agent invents its own label) through a peak at Nw ≈ O(N^1.5) and then decays toward the absorbing state Nw = 1 as consensus is reached. The shape of the Nw(t) curve is the single most diagnostic time-series in the entire Naming Game literature — it distinguishes "still exploring" from "about to converge" from "metastable stuck," and step 22's metrics dashboard will plot it prominently. Step 15's job is to compute Nw at the current tick exactly as Dall'Asta defines it: count distinct `(language, lexeme)` pairs with strictly positive weight across the full agent population, **per world**. No aggregation across worlds; step 17's summary classifier decides whether/how to compare the two.
5. **Baronchelli, A. (2016). "A gentle introduction to the minimal Naming Game."** _Belgian Journal of Linguistics_ 30, 171-192. <https://arxiv.org/abs/1701.07419> — WebFetched. Abstract (fetched): _"Social conventions govern countless behaviors all of us engage in every day, from how we greet each other to the languages we speak. But how can shared conventions emerge spontaneously in the absence of a central coordinating authority? The Naming Game model shows that networks of locally interacting individuals can spontaneously self-organize to produce global coordination."_ Baronchelli 2016 is the gentle introduction the spec already cites in §9 and is the reference the team uses to ground the "minimal Naming Game rules" assumption encoded in steps 13-15. For this step specifically, the paper motivates **communication success rate** and **matching rate** as the two complementary success measures: success rate is _"what happened this tick"_ (instantaneous), matching rate is _"how much does the population currently agree"_ (latent). They move together during healthy convergence and diverge during metastable splits — a diagnostic pattern step 17's classifier will use later. The paper also covers the **minimal additive weight-update rule** that step 13 implements (`+Δ⁺` on success, no decrement on failure by default), which is why this step's metrics treat positive weight as the sole activeness criterion: in the minimal rule, a weight is either > 0 (token is known) or was never introduced (token is absent from the map), never exactly 0 mid-run unless the optional Δ⁻ floor clamped it.

**Paths not taken:**

6. **Streaming variance via Welford's online algorithm.** Welford (1962) gives a numerically stable one-pass recurrence for variance that avoids the catastrophic cancellation failure mode of the textbook `E[X²] − E[X]²` formula on values with small variance relative to their mean. **Considered and rejected** for v1 because: (a) the token weights in a Naming Game are bounded — with default `Δ⁺ = 0.1` and a typical run length of 10⁴ ticks, a single weight rarely exceeds `~100` and the variance lives comfortably inside IEEE 754 double precision without cancellation; (b) each per-tick variance computation walks at most N × (languages) × (referents) × (lexemes per referent) ≤ 500 × 2 × 2 × 4 ≈ 8,000 values per world — trivially small for any formulation; (c) Welford adds read-write complexity (running mean, running M2, running count) that the two-pass naive formulation does not need, and unit-testing it against hand-computed reference values is harder. The decision is revisitable: if a future step discovers numerical issues with long runs or normalized weights, swap the body of `tokenWeightVariance` to Welford; the public API does not change. The Welford code path is documented in a commented-out stub at the bottom of `scalar.ts` so the swap is a one-liner.
7. **Precomputing "top-weighted token per agent per referent" outside `computeScalarMetrics`.** The matching rate needs each agent's argmax token per referent. A memoized cache computed once at the top of `computeScalarMetrics` (a `Map<AgentId, Map<Referent, TokenLexeme>>`) is the canonical implementation. **Considered** caching that same map inside the agent itself (an extra `topToken` field updated incrementally by step 13's interaction engine) to avoid re-computing it on every tick. **Rejected** because: (a) it would require touching step 13's engine output shape, which is not a step-15 concern; (b) incremental updates are error-prone — forgetting to invalidate on a weight change produces silently wrong metrics; (c) the N × referents × lexemes recomputation cost per tick is O(N) and fast enough (≤ 500 × 2 × 4 = 4,000 lookups per world). The argmax map is computed fresh at the start of each `computeScalarMetrics` call and thrown away when the function returns. Step 16's graph metrics will also want the same map and can compute it themselves or accept an optional `prebuiltArgmax` param passed from the worker — step 16 decides at its own plan time.
8. **O(N²) matching rate vs. a sampled approximation.** The naive matching-rate formulation walks every unordered pair of agents per referent and counts how many pairs agree on their top-weighted token. Cost: `referents × N × (N − 1) / 2`. At N = 500 per world and 2 referents, that is `2 × 500 × 499 / 2 = 249,500` pairwise comparisons per world per tick — half a million total across both worlds, still ≤ 1 ms in modern V8 and well under the 16 ms per-tick budget at 60 fps. At N = 10⁴ per world the cost jumps to `2 × 10⁴ × (10⁴ − 1) / 2 ≈ 10⁸` comparisons per tick, which is ~100 ms and would dominate the tick budget. **Accepted trade-off for v1**: the interactive playground regime (N ≤ 500 per `docs/spec.md` §11 question 2) uses the naive pairwise formulation and documents the cost in a file-header comment. A sampled approximation (draw K = 1,000 random pairs per referent and divide the agreeing count by K) is described in a commented-out stub near the `matchingRate` function so a future headless-sweep step can drop it in without redesigning the signature. Rejected alternatives: (a) O(N) histogram-based matching rate (count how many agents have each top token, then compute the fraction of pairs that agree as `Σ c_i × (c_i − 1) / (N × (N − 1))` — this is mathematically equivalent to the pairwise count and would be the right implementation if someone demanded O(N); filed as a follow-up optimization, **not implemented in v1** to keep the step's code footprint minimal and its test suite easier to reason about); (b) Monte Carlo sampling as the default, **rejected** because it introduces randomness into a metric that the researchers will publish, which violates the determinism contract.

Total research items: **3 local Next doc citations + 2 external URLs WebFetched + 3 paths not taken = 8 links**, well above the ≥ 5 floor mandated by the plan-file contract and `CLAUDE.md` "Research quality gates."

## 7. Implementation approach

Start by reading `lib/sim/interaction.ts` and `lib/sim/world.ts` (and whatever step 13 named its `InteractionEvent` record) to pin down the exact field names. Do not guess from this plan file — the determinism and correctness of the metrics depends on matching the producer's shape byte-for-byte. Write the type file first, then the per-metric helpers, then the composite `computeScalarMetrics` entrypoint, then the tests. Each metric is small enough (5-20 lines) that the whole module is under 300 lines of production code.

First, author `lib/sim/metrics/types.ts`. Declare `PerLanguageScalarMetrics` as `{ meanTokenWeight: number; tokenWeightVariance: number }` — two fields, both possibly `NaN` when the language has no active tokens. Declare `PerWorldScalarMetrics` as `{ successRate: { successful: number; total: number; rate: number }; successRateByClassPair: SuccessRateByClassPair; distinctActiveTokens: number; matchingRate: number; perLanguage: Record<Language, PerLanguageScalarMetrics> }`. Declare the top-level `ScalarMetricsSnapshot` as `{ tick: number | null; world1: PerWorldScalarMetrics; world2: PerWorldScalarMetrics; overall: { successRate: { successful: number; total: number; rate: number }; successRateByClassPair: SuccessRateByClassPair } }`. The `tick` field is nullable because step 15 itself does not know what tick it is — the caller (step 20's worker) will stamp the tick number into the snapshot. Including the nullable field here rather than in a wrapping object keeps the serialized shape flat and cleanly one-row-per-tick, which is what step 30's long-format CSV export wants. Declare `SuccessRateByClassPair` as `Record<ClassPairKey, { successful: number; total: number; rate: number }>` where `ClassPairKey = ${AgentClass}__${AgentClass}`. Declare `ClassPairKey` as an exhaustive template-literal type so that a TypeScript compile error surfaces if a future agent class is added without updating the metric code. All exported types are `readonly` where possible — the snapshot object is produced by `computeScalarMetrics` and must not be mutated by consumers.

Second, author `lib/sim/metrics/scalar.ts`. Start with a top-of-file doc block summarizing: (a) the five metrics computed, (b) the "pure function, no I/O, no RNG" contract, (c) the `O(N² × R)` cost of matching rate and its acceptance under `N ≤ 500`, (d) the scope boundary excluding graph metrics (step 16) and run summaries (step 17). Then import `World`, `AgentState`, `AgentClass`, `Language`, `Referent`, `TokenLexeme`, `AgentId`, `Weight`, `Inventory`, and `InteractionEvent` from their existing home paths, plus the types from `./types`. No runtime dependencies beyond the type imports.

Third, implement `computeCommunicationSuccessRate(tickInteractions, worldId)`. The function scans the events once. If `worldId === 'overall'`, it counts every event; otherwise it filters by `event.worldId === worldId`. For each retained event, increment `total` unconditionally and increment `successful` if `event.success === true`. Return `{ successful, total, rate: total === 0 ? NaN : successful / total }`. Document the NaN-for-empty-denominator choice in a JSDoc comment — it is a deliberate design decision (see §9 test 3). One pass over the events, O(events) cost.

Fourth, implement `computeSuccessRateByClassPair(tickInteractions, worldId?)`. Initialize the result object with all 16 `(AgentClass, AgentClass)` keys pre-populated to `{ successful: 0, total: 0, rate: NaN }` — the pre-population is load-bearing because downstream consumers expect every key to be present even when there are no events for that cell this tick. Iterate the (optionally world-filtered) events once, computing the key as `${event.speakerClass}__${event.hearerClass}`, and increment the counts. After the loop, walk the 16 keys one more time and compute `rate = total === 0 ? NaN : successful / total` in place. O(events + 16) per call.

Fifth, implement `computeMeanTokenWeight(world, language)`. Iterate every agent, then every `(referent, tokenLexeme, weight)` triple under the given language in that agent's inventory, collecting every **strictly positive** weight into a running `sum` and `count`. Return `count === 0 ? NaN : sum / count`. O(N × R × L_max) per call where `L_max` is the max lexemes per referent — in practice a small constant.

Sixth, implement `computeTokenWeightVariance(world, language)`. Use the **two-pass naive formulation** for clarity: first pass computes the mean (delegating to the same iteration pattern as `computeMeanTokenWeight`; optionally extract a shared helper that returns `{ count, sum, values }` so both functions walk the inventory once when called from `computeScalarMetrics`), second pass computes `Σ (x − mean)²` over the collected values, and returns `count < 2 ? NaN : Σ / (count − 1)`. Document the sample variance choice (divisor `n − 1`) in a comment linking to the R `var()` default that the long-format CSV export targets. The two passes are fine at N ≤ 500; see §4 path-not-taken 6 for the Welford rationale.

Seventh, implement `computeDistinctActiveTokens(world)`. Create a `Set<string>` (the composite key `${language}:${lexeme}`, matching the `tokenKey` helper from step 09). Iterate every agent, every language in the agent's inventory, every referent in the language, every lexeme in the referent; for each lexeme with weight > 0, add the composite key to the set. Return `set.size`. Note the deliberate cross-referent deduplication: a token `(L1, "yellow")` that is positively weighted for referent A **and** referent B counts as **one** distinct token, not two — Dall'Asta et al.'s Nw counts labels, not label-meaning pairs. This is why the set uses `${language}:${lexeme}` and not `${language}:${referent}:${lexeme}` as the key. Document this choice with an explicit one-line comment citing the Dall'Asta 2008 paper. O(N × L × R × lexemes) = O(N) for bounded constants.

Eighth, implement the `topTokenPerAgentPerReferent(world)` helper. For each agent, for each referent in `world.referents`, scan all languages in the agent's inventory for the lexeme with the maximum strictly-positive weight, ties broken by deterministic lexicographic comparison on `${language}:${lexeme}`. Store the result as a `Map<AgentId, Map<Referent, TokenLexeme | null>>` where `null` means "this agent has no positive-weight token for this referent." Document the tie-break rule explicitly — it is the single most common source of metric drift across refactors. The cross-language argmax is deliberate: the matching rate asks "do these two agents agree on the surface form they would utter for this referent," and a bilingual agent whose top L1 and L2 tokens have equal weight resolves the tie toward the lexicographically smaller `${language}:${lexeme}` composite. Researchers can ablate this by swapping to a per-language matching rate in a later step if needed.

Ninth, implement `computeMatchingRate(world, argmaxCache?)`. The function accepts an optional precomputed argmax cache; if not provided, it calls `topTokenPerAgentPerReferent(world)` internally. For each referent in `world.referents`, iterate every unordered pair of agents `(i, j)` with `i < j`, look up both argmax tokens from the cache, and count a match **iff** both values are non-null and equal. Let `agreed_r` be the count of matching pairs and `total_r` be `N × (N − 1) / 2` — the total unordered pair count, _not_ the count where both agents have a non-null top token. Compute `perReferentRate_r = total_r === 0 ? NaN : agreed_r / total_r`. The matching rate is the mean of `perReferentRate_r` over all referents, with `NaN` rates filtered out of the mean (if all referents are `NaN`, the overall rate is `NaN`). Document the "denominator is all pairs, not just pairs with non-null top tokens" choice in a comment — it is deliberate because a null top token means "this agent has nothing to say about this referent," and treating that as a non-match (rather than excluding the pair) matches the spec's "agents whose top-weighted token for a given referent agrees" phrasing literally. Document the O(N² × R) cost at the top of the function. Include the sampled-approximation Monte Carlo formulation as a commented-out stub immediately below the naive implementation so a future optimization step can drop it in.

Tenth, assemble `computeScalarMetrics(world1, world2, tickInteractions)`. The function precomputes the two argmax caches at the top (one per world), then builds each per-world sub-object by calling the per-metric helpers with the cache threaded through for matching rate. It builds the `perLanguage` sub-object by iterating `world.languages` and calling `computeMeanTokenWeight` and `computeTokenWeightVariance` for each language. It builds the `overall.successRate` and `overall.successRateByClassPair` by calling the success-rate helpers with `worldId: 'overall'` and no world filter respectively. It returns the flat snapshot object with `tick: null` — the caller stamps the tick number afterward. The whole function is ~40 lines of orchestration, no branching logic beyond what the helpers already do. Document the argmax-cache reuse in a one-line comment ("matching rate and any future graph metric that wants argmax share this single precomputation per world").

Eleventh, author the test file `scalar.test.ts`. Group tests into five `describe` blocks — one per metric family — plus a final `describe("computeScalarMetrics", ...)` for the assembled entrypoint. Each `describe` opens with a tiny fixture helper (`makeTinyWorld(...)` or similar) that builds a `World` with a short list of agents and inline inventory literals. **Do not use `bootstrapExperiment`** for most tests — hand-constructed worlds give the exact inventory control that known-answer tests need, and avoid coupling this step's tests to step 11's bootstrap behavior. One or two integration-style tests at the end _may_ use `bootstrapExperiment` as a sanity check that the real bootstrap output flows through `computeScalarMetrics` without crashing. Enumerated tests in §9 below.

Twelfth, run the verification checklist: `npm test -- lib/sim/metrics/scalar` exits 0 with all enumerated tests passing, `npm run typecheck` exits 0, `npm run lint` exits 0 on the new files. Commit with the exact subject line from §12.

**Efficiency note (for future readers of this plan).** The module is intentionally not pre-optimized beyond the argmax-cache reuse. The O(N²) matching rate is the only super-linear metric and is cheap at N ≤ 500. If step 27 or step 28 later observes the metric pipeline dominating a batch sweep's wall-clock budget, the documented drop-in replacement is (a) histogram-based matching rate for O(N) per referent, and (b) Welford's algorithm for one-pass variance. Both are described in §4 paths not taken and have commented-out stubs in `scalar.ts`. Premature optimization would make this step larger and harder to review for no observable benefit in the v1 interactive regime.

**ScalarMetricsSnapshot key layout (exact).** Consumers in steps 17, 20, 22, and 30 will index into this object by key, so the layout is locked here as part of the step contract:

- `tick`: `number | null`
- `world1.successRate`: `{ successful, total, rate }` (world-scoped)
- `world1.successRateByClassPair`: `Record<ClassPairKey, { successful, total, rate }>`
- `world1.distinctActiveTokens`: `number` (Nw for world 1)
- `world1.matchingRate`: `number`
- `world1.perLanguage`: `Record<Language, { meanTokenWeight: number; tokenWeightVariance: number }>`
- `world2.successRate`, `world2.successRateByClassPair`, `world2.distinctActiveTokens`, `world2.matchingRate`, `world2.perLanguage` — symmetric to `world1`.
- `overall.successRate`: `{ successful, total, rate }` across both worlds combined.
- `overall.successRateByClassPair`: `Record<ClassPairKey, { successful, total, rate }>` across both worlds combined.

`overall` intentionally omits `distinctActiveTokens`, `matchingRate`, and `perLanguage` — those metrics are per-world by spec definition (§7.1) and merging them across worlds would be meaningless (the two worlds do not share agents or a topology). Step 17's summary classifier can compute cross-world comparisons from the two per-world values as needed.

## 9. Unit tests

`lib/sim/metrics/scalar.test.ts` is a single Vitest file with **10 named tests** organized under `describe` blocks. Every test uses hand-constructed fixtures with explicit inventory literals — no `Math.random`, no `Date.now`, no floating-point tolerance beyond what the metric's known answer requires (`toBeCloseTo(x, 10)` where appropriate). All tests run in the default `node` environment.

1. **Communication success rate: known-answer integer ratio.** Build a `tickInteractions` array of 10 events — 7 with `success: true`, 3 with `success: false`, all in `world1` — and assert `computeCommunicationSuccessRate(events, 'world1')` returns `{ successful: 7, total: 10, rate: 0.7 }`. Also assert `computeCommunicationSuccessRate(events, 'world2')` returns `{ successful: 0, total: 0, rate: NaN }` (no world2 events).
2. **Communication success rate: overall aggregates both worlds.** Build a `tickInteractions` array with 4 world1 events (3 successful) and 6 world2 events (2 successful). Assert `computeCommunicationSuccessRate(events, 'overall')` returns `{ successful: 5, total: 10, rate: 0.5 }`. This test pins the semantics of the `'overall'` value so step 17's classifier can rely on it.
3. **Empty interaction list yields NaN success rate, not 0.** `computeCommunicationSuccessRate([], 'overall')` returns `{ successful: 0, total: 0, rate: NaN }`. The decision is documented in the test name itself: NaN is the canonical "undefined" marker in JS numeric pipelines and Recharts/d3 skip it, whereas a literal `0` would plot as a misleading "100% failure" data point on the live dashboard. Step 22's dashboard treats NaN as a gap in the time series. This test pins the convention; any future refactor that swaps to `0` must update this test and the corresponding classifier logic in step 17 together.
4. **Success rate by class pair: 16 keys always present.** Build a tick with 3 events: one `W1-Mono` → `W1-Mono` success, one `W1-Bi` → `W1-Mono` failure, one `W2-Immigrant` → `W2-Native` success. Assert the returned record has **all 16** keys (e.g. `W1-Mono__W1-Mono`, `W1-Mono__W1-Bi`, ..., `W2-Immigrant__W2-Immigrant`), that the three observed cells have correct counts and rates, and that the other 13 cells have `{ successful: 0, total: 0, rate: NaN }`. This guards against the subtle bug where the record only contains observed pairs and downstream consumers crash when they expect a fixed shape.
5. **Mean token weight: known-answer 3-agent hand computation.** Build a world with 3 agents, each with a single `(L1, yellow-referent, 'yellow')` entry with weights `[1.0, 2.0, 3.0]` respectively. Assert `computeMeanTokenWeight(world, L1)` returns exactly `2.0`. Assert `computeMeanTokenWeight(world, L2)` returns `NaN` (no L2 entries exist). Document that the NaN-for-empty convention matches test 3.
6. **Token weight variance: known-answer 3-agent hand computation.** Same fixture as test 5 (weights `[1.0, 2.0, 3.0]`). The sample variance with divisor `n − 1 = 2` is `((1 − 2)² + (2 − 2)² + (3 − 2)²) / 2 = (1 + 0 + 1) / 2 = 1.0`. Assert `computeTokenWeightVariance(world, L1)` returns `1.0` via `toBeCloseTo(1.0, 10)`. Assert `computeTokenWeightVariance(world, L2)` returns `NaN`. **Also assert** the single-agent case: a world with one L1 weight returns `NaN` from the variance function (fewer than 2 observations, by the `n < 2` guard).
7. **Nw: 10 agents all with identical `(L1, yellow)` + `(L1, red)` inventories have Nw = 2.** Build a 10-agent World 1 where every agent has only L1 with `{yellow-referent: {yellow: 1.0}, red-referent: {red: 1.0}}`. Assert `computeDistinctActiveTokens(world) === 2`. This is the "full consensus" known-answer case and is also the base fixture for test 10.
8. **Nw increments by 1 when a new token is added to one agent.** Take the test-7 fixture and mutate _one_ agent (via a fresh `inventorySet` call so the test stays deterministic and does not depend on in-place mutation) to add `(L1, goldenrod, 0.5)` under the yellow-referent. Assert `computeDistinctActiveTokens(newWorld) === 3`. This test pins the "one positive weight anywhere in the population counts" semantics and also exercises the cross-agent aggregation — a bug that only counted the first agent would still return 2 here and would fail this test.
9. **Matching rate: 10 agents all with identical inventories have matching rate = 1.0 and success rate of prior interactions can be 1.0 for the same fixture.** Use the test-7 fixture (10 agents, identical L1 inventories with yellow and red). Build a synthetic `tickInteractions` array of 5 speaker→hearer pairs, all successful (the hearer always has the speaker's uttered token at the same referent because the inventories are identical). Assert `computeMatchingRate(world)` returns `1.0` (every pair agrees on both referents' top token). Assert `computeCommunicationSuccessRate(events, 'world1').rate === 1.0`. This is the **"known-answer test" from the task brief**: the "full consensus" baseline that every downstream metric test can spot-check against.
10. **Matching rate: two disjoint sub-populations of size 5 each have matching rate = 0.5 and Nw = 4.** Build a 10-agent world where agents 0-4 have `{L1: {yellow-referent: {yellow: 1.0}, red-referent: {red: 1.0}}}` and agents 5-9 have `{L2: {yellow-referent: {jaune: 1.0}, red-referent: {rouge: 1.0}}}`. Assert `computeDistinctActiveTokens(world) === 4` (two distinct tokens per referent × nothing shared across the subpopulations, but note that `(L1, yellow)` and `(L2, jaune)` are different composite keys — the expected count is `yellow + red + jaune + rouge = 4`). For matching rate: there are `10 × 9 / 2 = 45` unordered pairs. Pairs **within** the first subpopulation (`5 × 4 / 2 = 10`) agree on both referents. Pairs **within** the second subpopulation (also 10) agree on both referents. Pairs **across** the two subpopulations (`5 × 5 = 25`) disagree on both referents (the argmax of agent 0's yellow-referent is `(L1, yellow)` and the argmax of agent 5's yellow-referent is `(L2, jaune)` — different composite keys). So the per-referent matching rate is `(10 + 10) / 45 = 20/45 ≈ 0.4444...`, identical for both referents, and the overall matching rate is the mean of the two, also ≈ 0.4444. **Update the task brief's expected value**: the task brief says "half know yellow/red, half know jaune/rouge → matching rate = 0.5," but the correct mathematical answer for this exact split is 20/45 ≈ 0.4444. The implementation must match the mathematical answer, not the brief's informal approximation. The test pins `toBeCloseTo(20 / 45, 10)` for the matching rate and `4` for Nw. Leave a comment in the test file explaining the 20/45 derivation so a future reader is not confused by the brief's looser phrasing.
11. **`computeScalarMetrics` integration: per-world and overall success rates are computed correctly (overall ≠ world1, overall ≠ world2).** Build two tiny worlds — world1 with 5 agents and world2 with 3 agents — and a `tickInteractions` array with 4 world1 events (2 successful) and 2 world2 events (2 successful). Call `computeScalarMetrics(world1, world2, events)` and assert: `result.world1.successRate.rate === 0.5`, `result.world2.successRate.rate === 1.0`, `result.overall.successRate.rate === 4/6`. This test pins that `overall` is a genuine combined computation, not just a copy of `world1` or `world2`, and is the task brief's explicit "Per-world and combined values are computed correctly" requirement.
12. **`computeScalarMetrics` returns the expected key shape.** Using any small fixture, call `computeScalarMetrics(w1, w2, [])` and assert the returned object has exactly the keys `tick`, `world1`, `world2`, `overall` at the top level, that `world1` has exactly `successRate`, `successRateByClassPair`, `distinctActiveTokens`, `matchingRate`, `perLanguage` (symmetric for `world2`), and that `overall` has exactly `successRate` and `successRateByClassPair`. Use `Object.keys(...).sort()` for stable assertion. This test is the contract-lock for downstream consumers in steps 17, 20, 22, and 30. If a future step renames a field, this test fails loudly and forces the consumer-step plan files to update in lockstep.

All 12 tests are deterministic, take no RNG input, and run under Vitest's default `node` environment. Test count exceeds the task brief's "≥ 8 tests" floor.

## 1. Goal

Move the pure simulation core produced by steps 09-18 (seeded RNG, core types, topologies, world bootstrap, language-selection policies, interaction engine, preferential attachment, scalar metrics, graph metrics, run summary) into a dedicated **Web Worker** at `workers/simulation.worker.ts` and expose a typed Comlink RPC surface that the main thread can drive without ever touching the simulation state directly. The worker owns the seeded RNG, the two-world `SimulationState`, the cumulative interaction graph, and the per-tick metrics reducers; the main thread owns only a `Remote<SimulationWorkerApi>` proxy plus a deterministic `terminate()` function. This step also removes the temporary files step 19 introduced while de-risking the Turbopack worker-bundling pattern (`workers/hello.worker.ts`, `app/(public)/_dev/worker-smoke/page.tsx`, and the `/_dev/*` allowlist entry in `proxy.ts`), on the grounds that step 19 was a plumbing validation and its smoke surface should not linger into v1. The outcome is: (a) a single worker module that imports everything under `@/lib/sim/*` and exposes `init`, `step`, `run`, `getMetrics`, `getSnapshot`, `reset` as promise-returning methods; (b) a thin main-thread client wrapper at `lib/sim/worker-client.ts` that constructs the worker with the Turbopack-native `new Worker(new URL(...))` pattern, wraps it with Comlink, guards against SSR `window is undefined`, and exposes a `createSimulationWorker()` factory that returns `{ api, terminate }`; (c) the step-19 temporary files deleted so the repository no longer ships a `/_dev/` route; (d) a determinism invariant asserted by calling `run(50, ...)` twice with the same seed and comparing the metric time series bit-for-bit (executed inside a Vitest node-side test that exercises the simulation modules directly, not through Comlink, per the strategy justified in section 9). This step is **plumbing only** — it adds zero UI, it persists nothing, and it ships no visualization. The first UI consumer of the worker is step 21 (lattice canvas renderer), which is where any behavioral bug in this step will manifest. The scope boundary is strict: step 20 does not render, does not persist, does not add experiment-config forms, and does not touch `proxy.ts` beyond the single-line deletion of the `/_dev/*` allowlist entry.

## 3. Spec references

- `docs/spec.md` **§8 Architecture Sketch** — this step is the literal implementation of the "Simulation runs in Web Workers, never on the main thread" architectural commitment. The ASCII diagram in §8 draws a `postMessage` boundary between the `SimulationShell` client component and the worker, and lists the worker's payload as "metrics/tick" — which is exactly what `TickReport` (`{ tick, scalar, graph }`) encodes over the Comlink wire. The bullet list under "Key architectural commitments" says: (a) the worker-to-UI bridge is a **typed interface**, not ad-hoc postMessage — step 20's `SimulationWorkerApi` TypeScript interface is the single source of truth for that typing, imported into both the worker entry point (via `Comlink.expose<SimulationWorkerApi>`) and the client wrapper (via `Comlink.wrap<SimulationWorkerApi>`); (b) simulation code is pure and deterministic given seed + config, with no wall-clock-dependent logic, no `Math.random()`, and no I/O — step 20 preserves this because the worker module imports the already-deterministic `lib/sim/*` modules verbatim and simply hosts them; (c) rendering is decoupled from simulation, and the worker emits metrics and snapshots at the sampling interval — step 20 implements this by having the main thread call `step(count)` or `run(totalTicks, onProgress)` whenever it wants data, and the worker answers with `TickReport` objects that are `structuredClone`-safe. The §8 commitments are the contract step 20 honors; failing any of them means the architecture is not the one §8 promised.
- `docs/spec.md` **§6 Research Goals → Software Support Matrix** — RQ1 through RQ5 all depend on a functional simulation worker because every downstream observable (communication success rate, assimilation index, segregation index, Louvain modularity, time-to-consensus, etc.) is computed inside the worker and passed to the main thread for rendering/persistence. If step 20 ships a broken or non-deterministic worker, every research question the spec defines is blocked: the dashboards in step 22 cannot read metrics that do not arrive, the run-comparison view in step 29 cannot compare runs that cannot be saved, and the hypothesis presets in step 31 have no engine to run them against. This is why section 10 treats "run(50, …) twice with the same seed matches bit-for-bit" as a first-class acceptance criterion — determinism is the primary epistemic guarantee of the whole research instrument, and the worker boundary is the most likely place for a silent nondeterminism bug to slip in (e.g. via an `await` reordering or a `Comlink.proxy` callback that smuggles wall-clock state back into the RNG).
- `docs/spec.md` **§9 Capability Requirements — "Worker bridge" row.** Comlink is named as the leading candidate "worker bridge" capability ("Comlink — removes postMessage boilerplate; works with Next.js"). Step 19 installed `comlink` as a runtime dependency; step 20 is the first step that exercises it against the real simulation payload rather than the toy hello-world smoke.
- `docs/spec.md` **§11 Open Question 4 (noise/mishearing) and §11 Open Question 5 (weight update mode).** Both open questions are resolved inside the simulation modules (step 13 and step 09), so step 20 inherits whatever the `lib/sim/*` modules decided. This is load-bearing because it means the worker boundary is a _pure transport_ — the worker does not implement any simulation semantics of its own, and any change to noise/weight-update behavior is a step-13 change, not a step-20 change. Section 7 enforces this by forbidding the worker module from containing any simulation logic beyond glue code.
- `docs/spec.md` **§7.1 and §7.3** — the `TickReport` and `RunResult` shapes carry the exact observables listed in these sections. `TickReport` = `{ tick: number; scalar: ScalarMetricsSnapshot; graph: GraphMetricsSnapshot }` unions the per-tick outputs of step 15 and step 16. `RunResult` = `{ summary: RunSummary; metricsTimeSeries: TickReport[] }` pairs the step-17 end-of-run summary with the per-tick history, so a single `run(totalTicks)` call returns everything step 22's dashboard and step 26's persistence layer will later need.

**Local Next.js 16 docs (authoritative per `CLAUDE.md` "Next.js 16 deltas from training data" and `AGENTS.md`):**

**External references (WebFetched):**

5. **Comlink API documentation** — `https://github.com/GoogleChromeLabs/comlink` (WebFetched during research). Load-bearing facts confirmed by the fetch:
   - `Comlink.expose(value, endpoint?)` is called inside the worker with the implementation object. `Comlink.wrap<T>(endpoint)` is called on the main thread and returns a `Remote<T>` proxy whose methods are all promise-returning, regardless of whether the underlying implementation is sync or async.
   - **Callbacks across the wire**: function values are not `structuredClone`-able, so passing `onProgress` directly would throw a `DataCloneError`. The canonical pattern is `Comlink.proxy(callback)` on the **main-thread side** — this wraps the callback in a proxy that Comlink ferries across the wire as a handle, and the worker invokes it via its own Comlink proxy machinery, not via direct cloning. Step 20's `run(totalTicks, onProgress)` API documents this requirement in its JSDoc and in CLAUDE.md's "Worker lifecycle" section so client-component authors do not accidentally pass a bare function.
   - **Cleanup**: `proxy[Comlink.releaseProxy]()` detaches the Remote proxy and allows garbage collection of the channel state. On browsers that support `WeakRef` this can happen automatically, but calling it explicitly is recommended whenever the worker is about to be terminated. Step 20's `terminate()` function calls `api[Comlink.releaseProxy]()` _before_ `worker.terminate()` so the Comlink side of the channel tears down cleanly before the underlying Worker is killed; doing it in the other order can leave dangling resolvers if a `run()` call is in flight.
   - **`Comlink.transfer(value, transferables)`** is available for zero-copy transfer of `ArrayBuffer`, `MessagePort`, etc. Step 20 does **not** use this — the `TickReport` and `RunResult` payloads are plain JSON-serializable objects with `Record<string, number>` metric maps (per step 15-17's `structuredClone`-safety discipline documented in their plan files), and the per-tick payload at N ≤ 500 agents is dominated by a ~50-entry flat metric map that costs well under a millisecond to structured-clone even at 60 ticks per second. The performance budget does not warrant the complexity of transferables in v1; section 4's "paths not taken" records this as an explicit rejection.
   - **Stability**: Comlink is a mature, production library (~13k stars, ~1.1kB min+gz, no runtime dependencies). It ships TypeScript declarations bundled with the main package. Step 19 installed it and verified the bundling; step 20 reuses the already-pinned version.

6. **MDN `Worker()` constructor** — `https://developer.mozilla.org/en-US/docs/Web/API/Worker/Worker` (WebFetched during research). Load-bearing facts:
   - `new Worker(scriptURL, { type: 'module' })` creates a **module worker**, which enables static `import` statements inside the worker file, mandatory strict mode, CORS-aware fetching of the module and its dependencies, and synchronous module resolution before any code executes. Without `{ type: 'module' }`, the worker is a "classic" worker and the `@/lib/sim/*` ESM imports would fail at parse time.
   - The `new URL('./name.worker.ts', import.meta.url)` expression is the canonical bundler-friendly way to name a worker script — the URL is resolved **relative to the current script's location**, not the HTML page URL, which is what allows bundlers (Turbopack, webpack, esbuild, Vite) to statically discover the worker entry point and emit a separate chunk for it. This pattern is the reason step 19 and step 20 can use a single source-of-truth file at `workers/simulation.worker.ts` without any `public/` folder publishing or custom-copy build step.
   - `worker.terminate()` immediately aborts the worker thread without waiting for pending operations to resolve. Any `postMessage` calls in flight will silently never arrive. Step 20's `terminate()` function accepts this semantics: callers who have an in-flight `run()` promise will see it hang indefinitely unless they hold a cancellation token, which the v1 API does not expose (see section 4's "paths not taken"). Client components that call `terminate()` should either await any in-flight `run()` first, or accept that orphaned promises will be garbage-collected along with the component — this is documented in the `createSimulationWorker` JSDoc and in CLAUDE.md's "Worker lifecycle" section so client-component authors are not surprised.
   - `postMessage` uses the **structured clone algorithm** for its arguments. This is compatible with the `TickReport`/`RunResult`/`FullStateSnapshot` shapes because they are composed of JSON primitives, arrays, plain objects, and `null`. They deliberately avoid `Map`, `Set`, class instances, functions, `Symbol`, and `BigInt` — matching step 17's explicit `structuredClone`-safety discipline for `RunSummary`. The one exception is `FullStateSnapshot`, which carries the full agent inventories and may need an explicit serialization shim for the `Map<Language, Map<Referent, Map<TokenLexeme, Weight>>>` nested-Map shape from step 09; section 7 below documents the shim approach (serialize to `[language, referent, lexeme, weight][]` quadruple arrays on the worker side, rehydrate to Maps on the main thread only if a caller actually needs Map access, which v1 does not).

**Paths not taken:**

7. **SharedArrayBuffer for zero-copy metrics streaming.** Considered and rejected. SharedArrayBuffer enables true shared memory between the main thread and a Web Worker, which would let the metrics dashboard in step 22 read the latest tick's scalar metrics without a `postMessage` round-trip. However, SAB requires the page to be served with `Cross-Origin-Opener-Policy: same-origin` and `Cross-Origin-Embedder-Policy: require-corp` headers (the "cross-origin isolation" requirement that browsers enforced post-Spectre). Adding those headers to the Next 16 deployment is non-trivial: they affect every external resource the page loads (images, fonts, third-party scripts, iframes), and Next's `app/(auth)/layout.tsx` currently includes Google Fonts via `next/font/google` which would need to be re-hosted or downgraded. The performance benefit is marginal for v1 scale: a 50-entry metric map clones in well under 1ms, and the 60 Hz rendering budget is 16.6ms per frame, so structured-clone overhead is at most 6% of the frame budget even at worst-case tick rates. The deploy complexity is not justified for the marginal benefit. If v2 research demands sub-millisecond metric streaming (e.g. for real-time multi-worker batch sweeps in step 27), SAB can be added then, as a targeted optimization rather than a v1 requirement. **Recorded here as an explicit rejection** so a future agent does not revisit the design without reading this rationale first. This is the "≥ 1 path not taken" required by the research quality gates.

8. **Webpack worker-loader pattern from the Park.is Next 15 blog post cited in `docs/spec.md` §8/§12.4.** Rejected categorically. `CLAUDE.md` "Next.js 16 deltas from training data" and "Known gotchas" both explicitly name this pattern as obsolete under Turbopack and reject it. A custom `webpack` config fails the v16 build at all. Step 19 already validated the Turbopack-native `new Worker(new URL(...))` path; step 20 reuses it verbatim. No worker-loader package is installed, no `next.config.ts` webpack hook is added. This is called out here (despite being a pre-established rule) because the Park.is post is the top Google result for "Next.js Comlink worker" and a future agent unfamiliar with the project's Next 16 conventions might otherwise reach for it.

9. **Running each simulation module inside its own micro-worker (a worker pool by phase).** Rejected. One worker per simulation phase (bootstrap, tick, metrics) would let each phase run in parallel on different cores, but the Naming Game is inherently sequential across ticks (tick N+1 depends on tick N's state), and the phases within a tick are fast enough that inter-worker `postMessage` overhead would exceed the savings. The single-worker-per-simulation design matches `docs/spec.md` §8's ASCII diagram exactly (one box labeled "Web Worker (ABM engine)"), and step 27 ("batch queue") is the correct place to introduce multi-worker parallelism — at the **run** granularity, not the **phase** granularity. Step 20's API is already compatible with step 27 because `createSimulationWorker()` is a factory: step 27 will call it N times to get N independent workers.

10. **Eagerly constructing the worker at module import time in `lib/sim/worker-client.ts`.** Rejected for an SSR-hostile-on-import, lazy-on-useEffect design. Constructing a `new Worker(...)` at module import time causes the SSR pass to throw (Workers are not defined in Node) — so `lib/sim/worker-client.ts` must expose a _factory function_ rather than a singleton Worker instance, and that factory must be called inside `useEffect` in the consuming client component. `CLAUDE.md` "Worker lifecycle" documents this discipline as "the wrapping client component either (a) constructs the worker inside useEffect to avoid SSR window is undefined, or (b) uses next/dynamic with ssr: false. Pick one and document it in the step's plan file." Step 20 picks (a) — construct inside `useEffect` — because it keeps the module surface simple (no `next/dynamic` import, no loading fallback component) and matches the pattern step 19 validated. The alternative (b) is documented as available for step 21 if the renderer's JSX surface prefers it, but step 20 itself does not exercise (b) because step 20 ships no JSX at all.

11. **Exposing a `subscribe(onTick)` streaming method in addition to `run(totalTicks, onProgress)`.** Considered and rejected for v1. A long-running `subscribe` that pushes every tick to the main thread regardless of caller state would duplicate the `run`-with-`onProgress` callback model without adding expressive power: `onProgress` is already a Comlink-proxied callback and can be invoked every tick (controlled by the caller's `progressIntervalTicks` argument, which defaults to 1 = every tick). Keeping the API to five methods (`init`, `step`, `run`, `getMetrics`, `getSnapshot`, `reset`) matches the user's step-specific context verbatim and avoids the proliferation-of-shapes anti-pattern. If step 21's renderer turns out to need a push-based stream instead of the pull-based `step()` or the callback-based `run(..., onProgress)`, it can be added as a sixth method in that step without breaking the five existing ones.

12. **Making the worker's API inherently cancelable via `AbortSignal`.** Considered and deferred. An `AbortSignal` on `run(totalTicks, onProgress, signal)` would let the caller cancel a long run mid-sweep. However, `AbortSignal` is not `structuredClone`-safe out of the box — Comlink would need to wrap it via `Comlink.proxy(signal)` and the worker would need to poll the proxy's `aborted` flag on every tick, which adds a per-tick Comlink round-trip (several milliseconds of latency at high tick rates). Simpler v1 alternative: `terminate()` is the cancellation primitive — the caller calls `terminate()` and the worker is killed. Step 27 (batch queue) will need finer-grained cancellation because it runs multiple workers concurrently and wants to stop one without stopping the others; that is where `AbortSignal`-like semantics can be added, on top of the v1 primitives rather than as a retrofit to them.

Total research items: **4 local Next docs** (turbopack.md, version-16.md, proxy.md, data-security.md) + **2 external WebFetched references** (Comlink README, MDN Worker constructor) + **6 paths not taken** (SharedArrayBuffer, webpack worker-loader, per-phase workers, eager worker construction, `subscribe` streaming API, `AbortSignal` cancellation) = **12 research items**, comfortably clearing the quality gates (≥ 3 local Next docs, ≥ 2 external URLs WebFetched, ≥ 1 path not taken, total ≥ 5).

## 5. Files to create

- `workers/simulation.worker.ts` — the **Web Worker entry point**. Starts with a file-header comment documenting that this file runs inside a browser Worker context (no `window`, no `document`, no `next/*` imports, no DOM APIs) and is the canonical integration point for the pure simulation core. Imports: `* as Comlink from 'comlink'` for the RPC facade; `bootstrapExperiment` and `World` from `@/lib/sim/bootstrap` (step 11); `tick` and the engine types `SimulationState`, `TickResult`, `InteractionEvent` from `@/lib/sim/engine` (step 13); `computeScalarMetrics` and `ScalarMetricsSnapshot` from `@/lib/sim/metrics/scalar` (step 15); `computeGraphMetrics`, `createInteractionGraph`, `updateInteractionGraph`, and `GraphMetricsSnapshot` from `@/lib/sim/metrics/graph` and `@/lib/sim/metrics/interaction-graph` (step 16); `computeRunSummary` and `RunSummary` from `@/lib/sim/metrics/summary` (step 17); the `ExperimentConfig` type from `@/lib/schema/experiment` (step 01); and the `RNG` type from `@/lib/sim/rng` (step 09). The file declares (and **re-exports** for main-thread TypeScript import) the public interface `SimulationWorkerApi` and the payload types `TickReport`, `RunResult`, `FullStateSnapshot`, `ExperimentConfigInput`. It then defines a module-level mutable state holder `let state: { rng: RNG; world1: World; world2: World; interactionGraph: UndirectedGraph; config: ExperimentConfig; tick: number; timeSeries: TickReport[] } | null = null` that is set by `init()` and cleared by `reset()`. It implements each of the six API methods as an async arrow function closing over `state`. Finally, it calls `Comlink.expose(api)` at the bottom of the module — with no guard, because the module only runs inside a Worker and `self` is always defined there. No other side effects happen at top level. File size: ~200 lines including doc comments.

- `lib/sim/worker-client.ts` — the **main-thread client wrapper**. Starts with a file-header comment: _"Client-safe module. Constructs a simulation Web Worker and wraps it with Comlink. Import from a client component only; SSR execution throws by design (see the SSR guard below). The worker must be constructed inside `useEffect` per `CLAUDE.md` 'Worker lifecycle'."_ Imports: `* as Comlink from 'comlink'` and `type { Remote } from 'comlink'`; `type { SimulationWorkerApi, TickReport, RunResult, FullStateSnapshot, ExperimentConfigInput } from '@/workers/simulation.worker'` (type-only imports, so the worker module itself is not pulled into the main-thread bundle — TypeScript's `import type` is a compile-time construct that Turbopack's tree-shaker elides from runtime). Exports: (a) a re-export of every type from the worker module (`type TickReport`, `type RunResult`, `type FullStateSnapshot`, `type SimulationWorkerApi`, `type ExperimentConfigInput`) so consumers can import all types from `@/lib/sim/worker-client` without reaching into the worker module directly — this is the one-stop-shop for the main-thread API surface; (b) a function `createSimulationWorker(): { api: Remote<SimulationWorkerApi>; terminate: () => void }`. The body of `createSimulationWorker`: (1) SSR guard — `if (typeof window === 'undefined') throw new Error('createSimulationWorker must be called in the browser (inside useEffect or similar).')`; (2) construct the worker — `const worker = new Worker(new URL('../../workers/simulation.worker.ts', import.meta.url), { type: 'module' })`; (3) wrap with Comlink — `const api = Comlink.wrap<SimulationWorkerApi>(worker)`; (4) define `terminate` — `const terminate = () => { api[Comlink.releaseProxy](); worker.terminate(); }`; (5) return `{ api, terminate }`. The JSDoc on `createSimulationWorker` documents: "Call inside `useEffect`, not at module scope. Terminate on component unmount via the effect cleanup. For callbacks passed to `api.run()`, wrap with `Comlink.proxy(callback)` — otherwise Comlink throws DataCloneError on the function value. See CLAUDE.md 'Worker lifecycle'." File size: ~80 lines including doc comments.

See section 9 for the testing strategy decision. **Recommendation: skip this file in v1 and rely on (a) the existing step-18 node-side smoke test that already exercises the simulation modules bit-for-bit-deterministically, and (b) step 21's MCP verification that will be the first real UI consumer of the worker.** If the implementing claude chooses to ship it anyway, the file contains a single `describe('simulation worker determinism')` block with one test: "`run(50, ...)` twice with the same seed produces bit-identical metric time series" — but it invokes the simulation modules **directly** (`bootstrapExperiment → tick → computeScalarMetrics → computeGraphMetrics → computeRunSummary`), not through a mocked Comlink `Worker` shim, because mocking `Worker` under `vitest --environment=node` is fragile (see section 9 for the full rationale). If shipped, this file re-validates the step-18 smoke test's determinism invariant from a slightly different angle — proving that the exact tuple `(RunSummary, TickReport[])` the worker's `run()` method will emit is itself deterministic, independently of any Comlink transport layer. Two tests total: (a) the bit-for-bit determinism assertion; (b) a shape-conformance assertion that the returned tuple matches the `RunResult` interface as declared in the worker module (verified structurally via TypeScript compile check plus a runtime `expect(result.summary).toHaveProperty('classification')` sanity check).

## 7. Implementation approach

The work is ordered so that the worker entry point lands first (it is the foundational contract), then the main-thread client wrapper (which imports the worker's type exports), then the step-19 teardown (which has no code dependencies and can be done last), and finally the CLAUDE.md append and verification. This ordering minimizes the window in which the repository is in an inconsistent state — e.g., if the agent writes the client wrapper before the worker module exists, the type-only imports fail at typecheck time and the next slice cannot proceed. The determinism invariant is verified once at the end as a sanity check against the full step-18 simulation surface, not as a Vitest suite colocated with the worker module (see section 9 for that decision).

**Slice one — define the API types up front inside `workers/simulation.worker.ts`.** Before writing any function bodies, sketch the public TypeScript surface at the top of the file. This is the contract every later slice reads:

```typescript
// (inside workers/simulation.worker.ts, near the top after imports)
export interface TickReport {
  tick: number;
  scalar: ScalarMetricsSnapshot;
  graph: GraphMetricsSnapshot;
}

export interface RunResult {
  summary: RunSummary;
  metricsTimeSeries: TickReport[];
}

export interface FullStateSnapshot {
  tick: number;
  config: ExperimentConfig;
  // Agent inventories serialized to plain-array quadruples for structured-clone compatibility
  world1: {
    agentId: string;
    class: string;
    position: number;
    inventory: Array<[string, string, string, number]>;
  }[];
  world2: {
    agentId: string;
    class: string;
    position: number;
    inventory: Array<[string, string, string, number]>;
  }[];
  // Cumulative interaction graph as an edge list
  interactionGraphEdges: Array<[string, string, number]>; // [speakerId, hearerId, weight]
}

export type ExperimentConfigInput = ExperimentConfig; // raw input; validated inside init()

export interface SimulationWorkerApi {
  init(config: ExperimentConfigInput, seed: number): Promise<void>;
  step(count?: number): Promise<TickReport>;
  run(totalTicks: number, onProgress?: (report: TickReport) => void): Promise<RunResult>;
  getMetrics(): Promise<ScalarMetricsSnapshot & GraphMetricsSnapshot>;
  getSnapshot(): Promise<FullStateSnapshot>;
  reset(): Promise<void>;
}
```

Declaring these types at the top of the file serves two purposes: (a) they are the single source of truth the main-thread client imports via `import type { ... } from '@/workers/simulation.worker'`, and (b) they force the function-body slices that follow to conform to an explicit signature rather than drift. The `ExperimentConfigInput` alias is a cosmetic layer that lets step 25 (experiment config UI) later distinguish "raw config form data validated at the worker boundary" from "fully parsed `ExperimentConfig` object" if the forms evolve to ship unparsed objects; in v1 the two are the same type. The `FullStateSnapshot.inventory` is a `[language, referent, lexeme, weight][]` quadruple array rather than a nested-Map shape because the nested-Map shape from `@/lib/sim/types.ts` is **not** `structuredClone`-safe across the `postMessage` boundary (structured clone supports `Map` in principle, but the brand-string keys serialize as plain strings on the other side, losing the TypeScript brand discriminant — the quadruple-array form keeps the brand information out of the wire format entirely and forces the main thread to explicitly re-brand if it ever wants to materialize the nested Map shape, which v1 does not).

**Slice two — implement `init(config, seed)`.** The body: (1) validate the config via Zod — `const parsed = ExperimentConfig.parse(config)` (or whatever the step-01 schema's canonical parser entry point is named). This catches out-of-range knobs before any simulation state is built. (2) Call `bootstrapExperiment(parsed, seed)` from step 11 to build the initial `world1`, `world2`, and `rng`. (3) Construct a fresh cumulative interaction graph via `createInteractionGraph()` from step 16. (4) Set the module-level `state` holder to `{ rng, world1, world2, interactionGraph, config: parsed, tick: 0, timeSeries: [] }`. The `timeSeries` array accumulates `TickReport` objects on every `step()` or `run()` call; it is the backing store for `getMetrics()` (returns the tail) and for `run()` (returns the whole array inside `RunResult`). (5) Return `Promise<void>` (implicit — the async arrow function just completes). The function is idempotent in the sense that calling `init` twice with the same seed produces identical state; the second call overwrites the first's state. Document this in a JSDoc comment. No `TickReport` is produced by `init` because the semantics are "the experiment has been configured and seeded but no ticks have been run yet"; the first `TickReport` arrives from the first `step()` or the first iteration of `run()`.

**Slice three — implement `step(count = 1)`.** The body: (1) Null-guard `state` — if `state === null`, throw `new Error('simulation worker: step() called before init(). Call init() first.')`. This is a programmer error in the consumer code, not a runtime failure; the error message is explicit so debugging is trivial. (2) For `i` in `0 .. count - 1`: (a) Call `tick(state, state.rng)` from step 13 to advance one tick. This returns `TickResult = { state: SimulationState, interactions: InteractionEvent[], tickNumber: number }`. The returned `state` is the same object reference mutated in place per step 13's "mutate AgentState in place, Inventory is immutable" discipline; assign it back to `state.world1/world2` fields as a no-op to satisfy the linter. (b) Call `updateInteractionGraph(state.interactionGraph, tickResult.interactions)` from step 16 to merge this tick's successful interactions into the cumulative graph. (c) Compute the per-tick metrics: `const scalar = computeScalarMetrics(state.world1, state.world2, tickResult.interactions, state.tick)` and `const graph = computeGraphMetrics(state.world1, state.world2, state.interactionGraph, tickResult.interactions)`. (d) Build the report: `const report: TickReport = { tick: state.tick, scalar, graph }` and push it onto `state.timeSeries`. (e) Increment `state.tick`. (3) After the loop, return the **last** `TickReport` produced. This matches the semantics "step returns the report for the most recent tick" regardless of how many ticks were executed in the batch. A JSDoc comment documents the batching semantics: "`step(10)` runs 10 ticks and returns the 10th tick's report; intermediate reports are stored in the internal time series and can be retrieved via `run` or a future `getTickHistory` method."

**Slice four — implement `run(totalTicks, onProgress?)`.** The body: (1) Null-guard `state` (same as slice three). (2) Record `const startTick = state.tick`. (3) For `i` in `0 .. totalTicks - 1`: (a) Run one tick via the same sub-steps as slice three (a-e). (b) If `onProgress` was provided, call `await onProgress(report)` — the `await` is important because `onProgress` is a `Comlink.proxy`-wrapped main-thread function, so invoking it returns a promise that resolves when the main thread has processed the callback. Without the `await`, the worker would fire and forget, and a slow main-thread consumer could be overwhelmed by a high tick rate. (4) After the loop, compute the final `RunSummary` via `computeRunSummary(state.timeSeries.slice(startTick).map(r => r.scalar), state.timeSeries.slice(startTick).map(r => r.graph), state.config)` from step 17. Note: `summary` covers only the ticks produced by _this `run()` call_, not the full time series back to `init()`. This lets callers invoke `run(100)` then `run(50)` then `run(50)` and get three independent summaries; the full concatenated time series is still available via the internal `state.timeSeries` array (and will be retrievable via a future `getTickHistory` method added in step 22 if needed — not in step 20). (5) Return `{ summary, metricsTimeSeries: state.timeSeries.slice(startTick) }`. JSDoc documents the "summary covers only this call's ticks" semantics and the `Comlink.proxy` requirement for `onProgress`.

**Slice five — implement `getMetrics()`, `getSnapshot()`, and `reset()`.**

- `getMetrics()` — null-guard, then return the last entry of `state.timeSeries` flattened: `{ ...last.scalar, ...last.graph }`. This is the "current latest metrics" accessor for UI polling. If `timeSeries` is empty (no ticks have run yet), throw or return an empty snapshot — pick **throw** with the message "simulation worker: getMetrics() called before any ticks ran. Call step() or run() first." Consumer code should arrange to call `step()` or `run()` before `getMetrics()`; the error makes the expectation loud.

- `getSnapshot()` — null-guard, then build the `FullStateSnapshot` by walking `state.world1.agents` and `state.world2.agents` and serializing each agent's `inventory` nested Map into a `[language, referent, lexeme, weight][]` quadruple array. The serialization loop is: `for (const [lang, refMap] of inventory) for (const [ref, lexMap] of refMap) for (const [lex, w] of lexMap) push([lang, ref, lex, w])`. This produces a compact, `structuredClone`-safe array that flattens the three nested `Map` levels. The interaction graph edges are serialized via `Array.from(state.interactionGraph.edges()).map(edgeKey => [state.interactionGraph.source(edgeKey), state.interactionGraph.target(edgeKey), state.interactionGraph.getEdgeAttribute(edgeKey, 'weight')])`. Return the composed snapshot.

- `reset()` — set `state = null`. A subsequent `init` call is required to resume; this mirrors the "the worker exposes no state-mutating calls beyond init, step, run, reset" invariant in the step-specific context. The JSDoc documents: "after reset, you must call init() before step()/run(); calling step() after reset() without an intervening init() throws."

**Slice six — expose the implementation via Comlink.** At the bottom of `workers/simulation.worker.ts`, define the implementation object `const api: SimulationWorkerApi = { init, step, run, getMetrics, getSnapshot, reset }` (arrow functions close over the module-level `state` holder, so no `this` binding needed). Then call `Comlink.expose(api)`. No other code runs at module scope. This is the whole worker.

**Slice seven — write `lib/sim/worker-client.ts`.** This is the main-thread wrapper. Start the file with the header comment described in section 5. Import `* as Comlink from 'comlink'` and `type { Remote } from 'comlink'`. Use a **type-only import** for the worker's type surface: `import type { SimulationWorkerApi, TickReport, RunResult, FullStateSnapshot, ExperimentConfigInput } from '@/workers/simulation.worker'`. Type-only imports are erased at build time by the TypeScript transform, so the worker's _code_ is not pulled into the main-thread bundle — only its type declarations influence the compiler. This is load-bearing: if the import were a value import, Turbopack would bundle the worker module into the main-thread graph and the `Comlink.expose` call would execute on the main thread, creating a cross-realm aliasing disaster. Re-export the five types via `export type { ... }` so consumers import everything from `@/lib/sim/worker-client` without touching the worker module directly. Implement `createSimulationWorker` as described in section 5: SSR guard first, `new Worker(new URL('../../workers/simulation.worker.ts', import.meta.url), { type: 'module' })` second, `Comlink.wrap<SimulationWorkerApi>(worker)` third, `terminate` closure fourth, return object fifth. The relative path in `new URL(...)` is `../../workers/simulation.worker.ts` because `lib/sim/worker-client.ts` is two directories deep relative to the repository root where `workers/` lives (`lib/sim/` → `lib/` → repo root → `workers/`). If the project layout differs (e.g. if step 19 put workers under `src/workers/`), adjust the relative path to match whatever `workers/simulation.worker.ts` resolves to. Verify by glancing at `tsconfig.json`'s `paths` or just by constructing the URL in one's head against the filesystem layout.

**Slice eight — grep-first teardown of step 19 temporary files.** Run these commands (or their equivalents via the Read and Edit tools): (1) `git show --stat <step-19-commit-sha>` to enumerate the files step 19 actually shipped. The commit subject is `step 19: worker bootstrap smoke`. The shipped files should be `workers/hello.worker.ts`, `app/(public)/_dev/worker-smoke/page.tsx`, possibly `lib/sim/worker-client.ts` (a precursor stub), and the `proxy.ts` edit. (2) Delete each shipped file that is not the precursor `worker-client.ts` (if step 19 shipped `worker-client.ts` in stub form, step 20's `worker-client.ts` **replaces** that stub rather than creates from scratch — treat the step 20 version as the final form). (3) Remove the `_dev` allowlist entries from `proxy.ts` per the grep-first protocol in section 6. (4) Verify the working tree: `grep -rn "_dev" proxy.ts` returns nothing; `ls workers/hello.worker.ts` returns "No such file"; `ls app/\(public\)/_dev/` returns empty or "No such file" (the parens need shell-escaping). The directory tree under `app/(public)/_dev/` should be gone entirely.

## 9. Unit tests

**Decision: skip dedicated worker-level tests in v1. Rely on (a) step 18's existing node-side smoke test, which already exercises the simulation modules deterministically; and (b) step 21's MCP verification, which will be the first real UI consumer of the worker and will catch any end-to-end integration failure.** This is option (b) from the step-specific context, and it is the recommendation.

**Justification for the decision.** Worker testing under Vitest has three viable paths, and each has material drawbacks at the v1 scale of this project:

1. **Happy-dom + a stubbed `Worker` constructor.** Vitest's `happy-dom` environment provides a browser-like global scope, but `happy-dom` does not implement the `Worker` constructor natively — it throws `ReferenceError: Worker is not defined`. To make this work, the test would need to `vi.stubGlobal('Worker', class MockWorker { ... })` with a hand-rolled in-process implementation that routes `postMessage` calls through the real worker module's exported functions. The hand-rolled Worker shim would have to faithfully reproduce Comlink's message-passing semantics (request IDs, promise resolution, `Comlink.proxy` callback marshalling) or the test would be testing the shim rather than the real worker. Maintaining a shim that tracks Comlink's internals across upgrades is fragile and does not exercise the property that actually matters (Turbopack bundling).

2. **Jsdom + the same stubbed `Worker`.** Same problem as (1); jsdom also does not ship a real `Worker` implementation. Switching to a different DOM harness does not move the load-bearing part of the equation.

3. **Real browser harness via Playwright or Puppeteer inside a Vitest plugin.** Would work but introduces a heavyweight browser dependency just for step 20's single test. The project already has chrome-devtools MCP for UI verification (see `CLAUDE.md` "UI verification harness"), and step 21's MCP script will naturally exercise the worker because step 21 is the first UI consumer — adding a parallel Playwright harness at step 20 duplicates that coverage at non-trivial maintenance cost.

**What step 20 relies on instead:**

- **Step 18's node-side smoke test** already proves the simulation modules (`bootstrapExperiment` → `tick` → `computeScalarMetrics` → `computeGraphMetrics` → `computeRunSummary`) produce bit-for-bit-deterministic output for a fixed seed. That test runs under Vitest's default `node` environment, does not touch any Web Worker machinery, and remains green after step 20 lands because step 20 does not modify any `lib/sim/*` module. Step 18 is the determinism baseline.

- **The worker module itself is glue.** It imports from `@/lib/sim/*`, wires the modules together in the order step 18's smoke test already validates, and exposes the result via Comlink. There is no new simulation logic inside `workers/simulation.worker.ts` — no branches, no numeric computations, no RNG draws. The only logic that is truly new is the `TickReport` composition (three `lib/sim` function calls merged into one object), the `timeSeries` array bookkeeping, and the `FullStateSnapshot` inventory flattening. Each of these is a small, branch-free transformation whose correctness is observable in the first MCP-driven tick at step 21 — if any of them is wrong, step 21's lattice renderer will display garbage on its first render and the MCP screenshot will fail the step. The failure mode is loud, the feedback loop is short.

- **Step 21's MCP verification is the first UI-level end-to-end test.** `scripts/run-plan.ts` spins up `next build && next start`, logs in, navigates to the playground, and takes a screenshot. If the worker fails to construct (bundling bug), fails to `init` (type mismatch between the worker's `ExperimentConfig.parse` and the step-01 schema), or fails to produce deterministic output (RNG leak across the wire), step 21 will fail loudly. This is the "push the first UI consumer to the next step" strategy the step-specific context explicitly endorses.

**If the implementing claude disagrees and wants a local test**, ship `lib/sim/worker-client.test.ts` with a single describe block containing:

1. **Determinism invariant (direct, not via Comlink).** Import the simulation modules directly (`bootstrapExperiment`, `tick`, `computeScalarMetrics`, `computeGraphMetrics`, `computeRunSummary`, `createInteractionGraph`, `updateInteractionGraph`). Build two independent runs with `seed = 42`, `totalTicks = 50`, and a small fixture config (`agentCount: 20` per world, lattice 5x5, default policy). For each run, loop 50 ticks, collecting `TickReport` objects at each tick by calling the same composition the worker uses (`tick` → `updateInteractionGraph` → `computeScalarMetrics` → `computeGraphMetrics` → combine). Assert `JSON.stringify(run1.timeSeries) === JSON.stringify(run2.timeSeries)` and `JSON.stringify(run1.summary) === JSON.stringify(run2.summary)`. **This test does not use Comlink or the Worker constructor at all** — it tests the simulation pipeline the worker hosts, not the transport layer. The transport layer is exercised by step 21's MCP. The determinism assertion is the load-bearing guarantee the whole research instrument depends on; this test re-validates it from the exact composition angle the worker will use, catching any drift between step 18's composition order and step 20's composition order.

2. **Shape conformance of the `RunResult` composition.** After the 50-tick run, assert the result object has the shape declared in `workers/simulation.worker.ts`: `expect(result.summary).toHaveProperty('classification')` (from step 17), `expect(result.metricsTimeSeries).toHaveLength(50)`, `expect(result.metricsTimeSeries[0]).toHaveProperty('tick', 0)`, `expect(result.metricsTimeSeries[49]).toHaveProperty('tick', 49)`, `expect(result.metricsTimeSeries[0].scalar).toBeDefined()`, `expect(result.metricsTimeSeries[0].graph).toBeDefined()`. This is a smoke-level structural test; it is not exhaustive but catches the most common "I renamed a field and broke the API" regression.

Neither test imports from `@/lib/sim/worker-client` or `@/workers/simulation.worker` — the former cannot run under `node` environment without a `Worker` stub, and the latter would execute `Comlink.expose` at import time which throws when `self` is undefined. Importing the underlying simulation modules directly sidesteps both problems. The test file's name `worker-client.test.ts` is a slight misnomer in that it tests the _simulation composition the worker hosts_ rather than the worker client itself; a more honest name would be `lib/sim/simulation-pipeline.test.ts`, but colocating it next to `worker-client.ts` makes the step-20 diff easier to review. The implementing claude may choose either name; document the choice in the file header comment.

**Recommendation: do not ship the test file in v1.** The step-18 smoke test already covers the determinism invariant, the composition in step 20 is branch-free glue code whose bugs will surface loudly in step 21's MCP, and shipping a test file that doesn't exercise the Comlink transport layer risks giving a false sense of coverage. If a regression slips through anyway, it is cheaper to add the test in the follow-up commit than to pre-emptively carry maintenance burden for a test whose coverage is subsumed by step 18.

- The simulation worker lives at `workers/simulation.worker.ts`. Main-thread code imports `createSimulationWorker` from `@/lib/sim/worker-client` inside a `useEffect`; the returned `{ api, terminate }` tuple is the only handle to the worker. `api` is a `Remote<SimulationWorkerApi>` whose methods all return promises.
- **Determinism invariant**: `run(N, ...)` twice with the same `(config, seed)` produces bit-identical `TickReport[]` and `RunSummary`. The seeded RNG lives inside the worker and never crosses the wire — message ordering cannot influence RNG state. The worker exposes **no** state-mutating calls beyond `init`, `step`, `run`, `reset`.
- **Callback marshalling**: `onProgress` passed to `api.run(...)` must be wrapped with `Comlink.proxy(callback)` on the main thread. Plain function values throw `DataCloneError` because functions are not structured-cloneable.
- **Cleanup order**: `terminate()` must call `api[Comlink.releaseProxy]()` **before** `worker.terminate()`. Reversing the order can leave dangling Comlink resolvers if a `run()` promise is in flight.
- **Structured-clone safety**: `TickReport` and `RunResult` payloads are plain objects with `Record<string, number>` metric maps — no `Map`, no class instances, no functions. `FullStateSnapshot` serializes agent inventories as `[language, referent, lexeme, weight][]` quadruples rather than the nested-Map shape from `lib/sim/types.ts`, because brand-string keys would lose their TypeScript discriminant across the wire.
